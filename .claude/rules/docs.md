## Общие правила по написанию документации

- Минимальный диф: трогаем только релевантные разделы, не переписываем целиком.
- Если из изменения не очевидно, как обновить документ — спросить у разработчика, не выдумывать.

## README как источник правды

`README.md` описывает **текущее** состояние проекта верхнего уровня: возможности, стек, архитектуру, команды, структуру. Он должен совпадать с кодом, а не отставать от него.

`openspec/` — про **планы и спецификации изменений**. Содержимое в README не дублируем.

### Когда обязательно править README

Если изменение попадает под один из триггеров — апдейт README идёт в том же коммите/PR, что и сам код:

- **Новая пользовательская возможность** (новая команда бота в `src/bot/handlers/`, новый use-case в `src/services/`, новая модель в `src/domain/`) → раздел «Ключевые возможности».
- **Новая или удалённая зависимость в `pyproject.toml`**, влияющая на стек (aiogram, ORM, тест-фреймворк, и т.п.) → раздел «Технологический стек».
- **Изменение слоёв архитектуры** (новый слой, переименование, новые правила зависимостей между слоями) → раздел «Архитектура».
- **Новая команда в `Makefile`** или изменение поведения существующей → раздел «Команды разработки».
- **Новая переменная окружения** или новый формат `.env` → раздел «Быстрый старт».
- **Изменение требований к запуску** (версия Python, замена `uv` на что-то ещё, смена БД по умолчанию) → раздел «Быстрый старт» и/или «Технологический стек».
- **Появление новой папки верхнего уровня** или удаление существующей → раздел «Структура проекта».

### Когда НЕ нужно править README

- Багфиксы без изменения поведения для пользователя.
- Внутренний рефакторинг в пределах одного слоя без новых публичных интерфейсов.
- Изменения в `tests/`, `scripts/`, `openspec/` сами по себе.
- Косметика текстов в `src/bot/messages.toml` без изменения сценариев.

### Как править

- Не дублируем CLAUDE.md и правила из `.claude/rules/` — README про *что* и *как пользоваться*, CLAUDE.md и rules про *как разрабатывать*.

## Каталог `docs/` — техническая документация

`docs/` — расширенное описание проекта для разработчиков и LLM-агентов: фичи, архитектура, схема БД, ключевые технические решения. Документы должны совпадать с кодом.

Состав каталога:
- `docs/README.md` — индекс.
- `docs/features.md` — пользовательские возможности и сценарии.
- `docs/architecture.md` — слои Clean Architecture, поток данных, конфиг.
- `docs/database.md` — таблицы, индексы, миграции.
- `docs/bot.md` — структура aiogram-приложения: dispatcher, router'ы, хендлеры, каталог текстов.
- `docs/decisions/` — директория с ключевыми техническими решениями и их обоснованием. Каждый файл должен в названии иметь дату решения и краткий slug, например 2026-05-20_add_social_auth.md

### Обязательные триггеры для апдейта `docs/`

Изменения ниже идут в том же PR, что и код. Это важнее README, потому что `docs/` — это «контракт» для будущих агентов и контрибьюторов.

| Изменение в коде                                                                                     | Куда писать                                                  |
| ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| Новая ORM-модель, FK, индекс, миграция, изменение типа колонки                                       | `docs/database.md` (таблица + диаграмма + раздел миграций)   |
| Новая команда бота, изменение поведения хендлера, новый текст в `messages.toml`                      | `docs/bot.md`                                                |
| Новая user-facing фича или существенное изменение поведения существующей                             | `docs/features.md`                                           |
| Новый слой, новая папка в `src/`, новый паттерн DI/зависимостей, смена правил между слоями           | `docs/architecture.md`                                       |
| Сознательное архитектурное решение (выбор библиотеки, отказ от async, новая стратегия поиска и т.п.) | `docs/decisions/` — добавить новый файл с **что/зачем/цена** |
| Новая обязательная переменная окружения                                                              | `docs/architecture.md` (таблица конфигурации) и `README.md`  |

### Когда `docs/` править НЕ нужно

- Опечатки и переформулировки в текстах бота без изменения сценариев.
- Рефакторинг без изменения публичного контракта (имена внутренних функций, разбиение модулей внутри одного слоя).
- Правки тестов, скриптов в `scripts/`, спецификаций в `openspec/`.
- Багфиксы, не меняющие поведение, описанное в документации.

### Как править `docs/`

- Минимальный диф: правим только релевантные разделы, не переписываем файлы целиком.
- `docs/decisions/` — append-only по смыслу: старые решения не удаляем, помечаем устаревшими и создаем новые с обоснованием смены.
- Не дублировать содержимое между документами: ссылка лучше копии.
- Перед коммитом — пробежаться по затронутому файлу глазами: не появилось ли расхождений с соседними разделами.
