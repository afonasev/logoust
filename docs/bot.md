# Telegram-бот

Бот построен на **aiogram 3** в режиме long-polling. Все пользовательские тексты — в `src/bot/messages.toml`, в коде формулировок нет.

## Сборка приложения

- `src/bot/dispatcher.py::build_dispatcher(messages, session_factory) -> Dispatcher` — собирает `Dispatcher`, подключает роутеры. **Порядок важен:** фичевые роутеры (`clients`, `schedule`, `settings`, `windows`) включаются раньше `start`-роутера — fallback `start` ловит любой текст (онбординг по вставленному коду), поэтому иначе он перехватывал бы reply-кнопки и ввод в визардах. Нераспознанный текст всё равно проваливается в `start`.
- `src/bot/handlers/start.py::build_router(messages, session_factory) -> Router` — регистрирует обработчик `/start` и fallback на вставленный код/ссылку. Различает онбординг специалиста (токен без префикса) и привязку клиента (токен с префиксом `cli_`, см. `src/bot/deeplink.py`): `cli_`-payload идёт в `link_client_by_token`, остальное — в специалистский `consume_invite`.
- `src/bot/handlers/clients.py::build_router(messages, session_factory) -> Router` — роутер управления клиентами: reply-кнопка, inline-меню, визард добавления (FSM), карточка, редактирование полей, архив/возврат, списки. Из карточки клиента ведут переходы в записи (`sched:*`). Состояние FSM — на дефолтном `MemoryStorage` (теряется при рестарте процесса).
- `src/bot/handlers/schedule.py::build_router(messages, session_factory) -> Router` — роутер записей: инлайн-календарь и сетка слотов с пометкой занятости (`build_calendar`, `build_slots_keyboard`), FSM свободного времени и комментария, карточка записи (перенос/удаление с подтверждением). Раздел «📅 Расписание» открывается на **сегодня** с навигацией по дням (◀/▶) и кнопкой «📆 Неделя»; история специалиста — текстом, по календарным неделям. Будущие записи клиента показываются кнопками в его карточке, история клиента — текстом по отдельной кнопке. Колбэки по схеме `sched:<action>[:<args>]`. Тот же роутер содержит **регулярные записи** (`RecurringHandlers`, `_register_recurring`): карточку серии, редактирование и действия пропуска/переноса/остановки даты, колбэки по схеме `recur:<action>[:<args>]` (создание идёт через обычный поток записи, см. ниже). Будущие повторы серий подмешиваются в день/неделю/занятость/карточку клиента как виртуальные `Appointment` (`id=None`, `series_id`/`origin_date` заполнены). Пометку 🔁 несёт транзиентный флаг `recurring_mark` — он стоит только у «чистой» будущей occurrence; перенесённая отдельная дата индивидуализируется и значка не получает; тап ведёт на карточку серии, а не записи.
- `src/bot/navigation.py::Navigator` — нейтральный реестр «`back`-префикс → корутина `(specialist_id, back) -> (text, keyboard)`», собираемый в `schedule.build_router` (`_build_navigator`) из «чистых» билдеров роутеров `schedule`/`recurring`/`clients`. Позволяет действию в одном роутере переоткрыть меню, которым владеет другой (например карточку клиента из роутера записей), без ре-диспатча синтетического колбэка и без перекрёстного импорта хендлеров (`clients` никогда не импортирует `schedule`). См. «Навигация после необратимых действий».
- `src/bot/handlers/settings.py::build_router(messages, session_factory) -> Router` — роутер настроек расписания: меню, выбор таймзоны из списка российских зон, FSM-ввод начала/конца дня и длины слота с валидацией, мульти-тоггл рабочих дней недели (немедленное сохранение), тумблер авто-напоминаний клиентам и FSM-ввод времени напоминаний. Колбэки по схеме `settings:<action>[:<value>]`.
- `src/bot/handlers/subscriptions.py::build_router(messages, session_factory) -> Router` — роутер абонементов: reply-кнопка «🎫 Абонементы» (раздел со списком активных абонементов всех клиентов и историей закрытых, постранично), карточка абонемента, FSM ввода числа встреч при создании/продлении (с кнопкой принятия дефолта специалиста), действия списания/закрытия. Колбэки по схеме `subs:<action>[:<id|page>]`. Переиспользует `SpecialistMiddleware`. Reply-кнопка зарегистрирована до FSM-хендлеров (escape-hatch из визарда, как «Клиенты»). Создание инициируется кнопкой на карточке клиента; «Назад» с карточки абонемента ведёт на карточку клиента (`clients:card:<id>`). Карточка **закрытого** абонемента — read-only (только возврат к клиенту).
- `src/bot/handlers/reminders.py::build_router(messages, session_factory) -> Router` — **клиентский** роутер ответа на напоминание (колбэк `appt:cfm:<reminder_id>:<y|n>`). Вне `SpecialistMiddleware`: актор — клиент. Записывает ответ (`apply_reminder_response`), отвечает клиенту тостом; при переходе в `declined` шлёт специалисту сообщение с кнопкой «→ Открыть запись». Включён в dispatcher до `start`-роутера.
- `src/bot/handlers/windows.py::build_router(messages, session_factory) -> Router` — роутер раздела «🪟 Окна»: по reply-кнопке рендерит read-only список свободных окон на ближайшие 5 рабочих дней (`list_free_windows`), без inline-колбэков и пагинации.
- `src/bot/handlers/start.py::make_start_handler(messages, session_factory)` — фабрика хендлера `/start` (вынесена для удобства тестов).
- `src/bot/handlers/start.py::make_token_handler(messages, session_factory)` — фабрика fallback-хендлера: ловит обычный текст и через `extract_token()` достаёт токен из голого кода или deep-link.
- `src/__main__.py` — точка входа: грузит `settings`, инициализирует logging, создаёт `Bot`, `session_factory`, `messages`, `Dispatcher`, запускает `asyncio.gather(dp.start_polling(bot), _scheduler_loop(...))` — polling рядом с минутным планировщиком напоминаний (см. `docs/architecture.md`). Тестируемый проход — `src/bot/scheduler.py::run_reminder_pass`.

## Хендлеры

| Команда | Файл | Что делает |
| --- | --- | --- |
| `/start [<token>]` | `src/bot/handlers/start.py` | Токен с префиксом `cli_` → привязка клиента (`link_client_by_token`, idempotent rebind `chat_id`/`linked_at`; `from_user.username` автозаполняет пустой `contact_telegram`; ответы `clients.linked` / `clients.link_unknown`). Иначе — онбординг специалиста: ищет по токену, идемпотентно проставляет `chat_id`/`username`/`welcomed_at`, четыре ветки (welcomed, already_welcomed, unknown_token, no_token). |
| Текст с кодом/ссылкой | `src/bot/handlers/start.py` (`make_token_handler`) | Fallback: вставленный голый токен или deep-link целиком. Сначала `extract_client_token` (ловит `cli_`-payload → привязка клиента), затем `extract_token` (специалистский онбординг). На сообщения, не похожие на токен, молчит. Зарегистрирован после `/start`. |
| Кнопка «👶 Клиенты» | `src/bot/handlers/clients.py` (`show_menu`/`show_active`) | Reply-кнопка (показывается после онбординга) сразу открывает постраничный список активных клиентов (сортировка по `child_name`). Каждая строка (`_client_row`) — два столбца: клиент (→ `clients:card:<id>`) и его **ближайшая будущая запись** (`nearest_future_by_client`; компактно, без комментария; → карточка записи для переноса) либо «➕ Записать» (`sched:new:<id>`), если записей нет. Действия «➕ Добавить» и «🗄 Архив». Нажатие также сбрасывает любой активный визард. Навигация: `clients:active:<page>`, размер страницы — `_ACTIVE_PAGE_SIZE`. |
| Управление клиентами | `src/bot/handlers/clients.py` (`ClientsHandlers`) | Добавление (FSM: имя ребёнка → имя контакта → телефон → telegram), карточка с общей информацией; **будущие записи клиента — кнопками** (`_future_button` → `sched:card:<id>`, тап открывает запись для переноса/отмены), затем «➕ Записать», строка «✏️ Изменить / 🗂 История / 📦 В архив» и «⬅️ Назад». Ближайшая occurrence активной серии клиента показывается среди будущих записей с пометкой 🔁 (`_appt_callback` → `recur:card:<series>:<origin_date>~<back>`). У активной карточки есть кнопка абонемента (создать или перейти с остатком, см. строку «Абонементы») и кнопка приглашения в бота (`clients:invite:<id>`, `send_invite`): лениво выдаёт `cli_`-токен (переиспользуя существующий), отдельным **пересылаемым** сообщением шлёт специалисту ссылку `t.me/<bot>?start=cli_<token>` (`build_client_start_link`). Подпись отражает состояние: «Пригласить в бота» / «✅ Привязан» при заполненном `telegram_chat_id`. Архивация — **с подтверждением** (`clients:archiveask:<id>` → `clients:archive:<id>`); возврат из архива — сразу. История записей клиента — **только текстом**, с пагинацией, без кнопок на записи. Кнопка «⬅️ Назад» на карточке клиента **контекстная** (origin в `clients:card:<id>~<back>`; по умолчанию — список активных/архив по статусу). Колбэки по схеме `clients:<action>:<id>`. |
| Архив клиентов | `src/bot/handlers/clients.py` (`show_archive`) | Постраничный архив, сортировка по `archived_at` убыванию (свежие сверху), дата архивации в строке. Размер страницы — `_ARCHIVE_PAGE_SIZE`. Навигация: `clients:list:archived` (стр. 0) и `clients:arch:<page>`; «⬅️ Меню» (`clients:menu`) возвращает к активному списку. Наличие следующей страницы определяется выборкой `page_size + 1` строки (без COUNT). |
| Колбэк `appt:cfm:` | `src/bot/handlers/reminders.py` (`ReminderHandlers`) | **Клиентский** ответ на напоминание. `appt:cfm:<reminder_id>:y` → `confirmed`, `:n` → `declined` (`apply_reminder_response`, изоляция по `chat_id` ответившего). Клиенту — тост; при переходе в `declined` специалисту шлётся `reminder.specialist_declined` с кнопкой «→ Открыть запись» (разовая → `sched:card:<id>` через `find_by_occurrence`; виртуальный повтор → `recur:card:<series>:<origin_date>`; перенесённая/исчезнувшая разовая → `sched:day_view:<iso>`). Ответ можно менять; повторный тот же ответ специалиста не дёргает. |
| Кнопка «📅 Расписание» | `src/bot/handlers/schedule.py` (`show_feed`/`open_day`/`show_week`) | Reply-кнопка открывает расписание на **релевантный день**: сегодня, если сегодня рабочий день или на сегодня есть записи, иначе ближайший вперёд **показываемый** день (`schedule_landing_day`). Заголовок «дата, день недели» (текст), записи дня — каждая отдельной кнопкой `время · ребёнок` → карточка. Запись несёт префикс статуса ответа клиента (`statuses_for_appointments` → `ReminderMessages.status_mark`): ✅ при `confirmed`, ❌ при `declined`, ничего при `pending`/без напоминания; на карточке записи/серии при `confirmed` — строка-пометка `reminder.card_confirmed`. Те же ✅/❌ показываются у кнопки ближайшей записи в списке клиентов и у будущих записей на карточке клиента. Кнопки ◀/▶ (`sched:day_view:<iso>`) ведут на ближайший показываемый день в направлении (`adjacent_shown_day`, series-aware — `SeriesContext` прокидывается в оба вызова): пустые нерабочие дни пропускаются, а нерабочий день с записью **или** с приземлением будущего повтора активной серии (с учётом пропусков/переносов) остаётся доступным; назад серии не учитываются (прошлое — только реальные строки); если показываемого дня в сторону нет — стрелка не рисуется. «📆 Неделя» (`sched:week`) — read-only обзор 7 дней (сгруппированный текст; пустые дни не выводятся), «🗂 История» — прошедшие записи **только текстом**, сгруппированы по дням, навигация **по календарным неделям** (пн–вс, `sched:hist:<week>`, ◀ старше / ▶ новее), без кнопок на отдельные записи. |
| Запись на приём | `src/bot/handlers/schedule.py` (`ScheduleHandlers`) | Из карточки клиента: календарь (прошлое заблокировано) → слот из сетки или «Другое время» (`ЧЧ:ММ`) → опциональный комментарий → создание и показ карточки записи. Слоты помечаются 🟢 (свободен) / 🔴 (на это время уже есть запись специалиста; `taken_slot_times`), без запрета бронирования. Перенос переиспользует тот же пикер, меняет только `starts_at` и исключает свой текущий слот из «занятых»; удаление — с подтверждением `[Да, удалить] [Отмена]`. На карточке записи кнопка «⬅️ Назад» **контекстная** — возвращает туда, откуда открыли (origin кодируется в `sched:card:<id>~<back-callback>`; по умолчанию — день записи). **Последним шагом** результата создания/переноса/отмены, **только если у клиента заполнен `telegram_chat_id`**, бот отдельным сообщением задаёт вопрос «Уведомить клиента?» (`notify_ask`) с **предпросмотром текста** уведомления и кнопками `[✅ Да, отправить] [Нет]` (`_ask_notify`/`_notify_ask_keyboard`). «Да» несёт `sched:ntf:<event>:<client_id>:<stamp>` (`event ∈ c/r/x` = created/rescheduled/cancelled, `stamp` — `starts_at` в UTC как `YYYYMMDDHHMM`; для отмены данные берутся **до** удаления, поэтому текст собирается и после исчезновения записи), «Нет» — `sched:ntfno`. Тап «Да» (`notify`) повторно проверяет привязку через `get_for_specialist`: при отсутствии `telegram_chat_id` вопрос заменяется на `notify_not_linked` без отправки; иначе шлёт клиенту **простой текст** (`notify_created`/`notify_rescheduled`/`notify_cancelled`, настенное время в tz специалиста) и заменяет вопрос на `notify_sent` (клавиатура снимается — повторный тап невозможен). Сбой доставки (`TelegramForbiddenError`/`TelegramBadRequest`) → вопрос заменяется на `notify_failed`, запись/операция не трогается. «Нет» (`notify_skip`) заменяет вопрос на `notify_skipped`. Непривязанному клиенту шаг не показывается; обычный просмотр карточки (`show_card`) уведомление не предлагает. Тот же шаг-вопрос показывается и для **регулярных записей** (см. строку «Регулярная запись»): для серии целиком — отдельный колбэк `sched:ntfs:<event>:<client_id>:<weekday>:<hhmm>` (`event ∈ c/m/x` = created/modified/cancelled; текст про правило «каждый четверг в 14:00», `notify_series_created`/`notify_series_changed`/`notify_series_cancelled`), обрабатывается `notify_series` (tz не нужен — время в правиле уже настенное); для отдельной даты серии (перенос/отмена следующей) — обычный одиночный `sched:ntf:` с конкретными датой/временем. |
| Регулярная запись | `src/bot/handlers/schedule.py` (`ScheduleHandlers`, `RecurringHandlers`) | Создаётся **в обычном потоке записи**: после выбора времени бот спрашивает «сделать регулярной?» (`sched:reg:1`/`sched:reg:0`, `choose_regular`) перед комментарием. При «да» → `create_series` с `start_date` = выбранная дата (первая встреча) и `weekday` этой даты, затем показывается карточка серии; при «нет» → обычная разовая запись. Тап по 🔁-повтору в дне/неделе/карточке клиента открывает **карточку серии** (`recur:card:<series>:<origin_date>[~<back>]`), повторяющую раскладку карточки записи. Карточка показывает **дату, день недели и время следующей встречи** (`_occurrence_display` — учитывает перенос этой даты, поэтому может отличаться от правила серии). Верхний ряд «📅 Перенести следующую» / «🗑 Отменить следующую» действует на эту дату: перенос (`start_move`) идёт **инлайн-календарём → сеткой слотов** (выбор новой даты, затем времени → `move_date`), отмена — `skip_date` с подтверждением. Второй ряд «⚙️ Настроить» / «🗑 Отменить всё» — на серию (`edit_series`; смена дня недели пересчитывает `start_date`/`materialized_through` — `stop_series` с подтверждением). Календарь переноса использует тот же `build_calendar` с префиксом `recur` (`recur:cal:`/`recur:day:`). Кнопки «⬅️ Назад» и «Отмена» **контекстные** — возвращают на карточку серии / её origin (callback хранится в `~<back>` и в FSM-состоянии). Перенесённая отдельная дата индивидуализируется: значка 🔁 у неё нет. Все действия ограничены своими сериями (`get_for_specialist`). После операции — **только для привязанного клиента** — предлагается шаг-вопрос «Уведомить клиента?» (см. строку «Запись на приём»): создание серии и «Настроить» → текст про правило (`sched:ntfs:` event=c/m, `_ask_series_notify`); «Отменить всё» → текст про правило (event=x); «Перенести следующую» / «Отменить следующую» → текст про конкретную дату (`sched:ntf:` event=r/x, `_ask_notify`). |
| Кнопка «⚙️ Настройки» | `src/bot/handlers/settings.py` (`SettingsHandlers`) | Reply-кнопка открывает настройки расписания: таймзона (из списка рос. зон), начало/конец рабочего дня и длина слота (FSM-ввод с валидацией), рабочие дни недели, авто-напоминания клиентам. Новое значение сразу применяется к генерации сетки, трактовке времени и расчёту окон. Рабочие дни — экран мульти-тоггла (7 кнопок ✅/⬜ Пн–Вс + «⬅️ Назад»): тап инвертирует день (`settings:wd:<idx>`), сразу сохраняет канонизированный набор (`toggle_working_day`) и перерисовывает клавиатуру, без отдельного подтверждения. Напоминания: кнопка-тумблер `settings:reminder` (вкл/выкл, `toggle_reminder`) и `settings:reminder_time` (FSM-ввод `ЧЧ:ММ` по образцу начала дня); текущее состояние и время видны в `title`. Дефолтное число встреч в абонементе — `settings:subscription_default` (FSM-ввод положительного числа по образцу длины слота); подставляется в подсказку создания/продления абонемента. |
| Кнопка «🎫 Абонементы» | `src/bot/handlers/subscriptions.py` (`SubscriptionsHandlers`) | Reply-кнопка (второй ряд меню) открывает постраничный список **активных** абонементов всех клиентов (строка `имя · осталось N` → карточка, `subs:active:<page>`) с кнопкой «🗄 Закрытые». История закрытых (`subs:closed:<page>`) — строка `имя · закрыт {дата}`, кнопка «⬅️ К активным». Имена тянутся через `client_name_map`. Карточка **закрытого** абонемента — read-only. |
| Абонементы (карточка/действия) | `src/bot/handlers/subscriptions.py` (`SubscriptionsHandlers`) | На активной карточке клиента — кнопка абонемента: «➕ Создать абонемент» (`subs:create:<client_id>`), если активного нет, либо «🎫 Абонемент · {остаток}» (`subs:card:<sid>`), если есть; на архивной карточке кнопка скрыта. Создание/продление спрашивают число встреч (FSM) с кнопкой принятия дефолта специалиста; невалидный ввод переспрашивается. Карточка абонемента: дата создания, куплено, остаток и действия «➖ Вычесть встречу» (`subs:dec:`, −1, alert при остатке 0), «➕ Продлить» (`subs:extend:`, оба счётчика +N), «✖ Закрыть» (`subs:closeask:` → `subs:close:`, с подтверждением, запись не удаляется). Не более одного активного абонемента на клиента; все действия ограничены своими клиентами. |
| Кнопка «🪟 Окна» | `src/bot/handlers/windows.py` (`WindowsHandlers`) | Reply-кнопка (верхний ряд меню) присылает одним сообщением свободные окна на ближайшие 5 рабочих дней, сгруппированные по дням. Окно = слот сетки без записи на это время; для сегодня прошедшие окна скрыты. День без свободных окон всё равно показывается со строкой «свободных окон нет» (предсказуемый счёт пяти дней). Если рабочие дни не заданы (`working_days` пуст) — подсказка открыть «Настройки». Экран read-only: inline-кнопок и пагинации нет. |

Доступ ограничивает `SpecialistMiddleware` (inner-middleware, переиспользуется всеми фичевыми роутерами): резолвит специалиста по `chat_id` через `SpecialistsRepo.find_by_chat_id`, кладёт `specialist_id` в данные хендлера, а апдейты от неонбординнутых пользователей роняет. На каждое действие специалиста middleware вызывает `settle` — материализует прошедшие occurrence активных серий в строки `appointments` (планировщика нет; идемпотентно, с дневным guard'ом через `materialized_through`).

Других команд бота (включая `/help`) сознательно нет — приветствие не должно врать о возможностях.

## Навигация после необратимых действий

После необратимого действия с записью (удаление разовой, перенос/остановка/пропуск даты серии, отмена флоу) бот **не** показывает тупиковый экран с единственной кнопкой «Назад». Единый хелпер `Navigator.open_after_action` (через тонкий шим `schedule._post_action`):

1. оставляет **отдельное сообщение-результат** в истории чата («Запись удалена/перенесена/…»). В колбэк-флоу (`edit=True`) исходная карточка с уже неактуальными кнопками редактируется в текст результата без клавиатуры — мёртвых кнопок не остаётся; во флоу, завершённом введённым текстом (`edit=False`, своё сообщение пользователя редактировать нельзя), результат шлётся новым сообщением;
2. сразу открывает **исходное меню** (`Navigator.render` по префиксу `back`) новым сообщением — самый свежий экран. Цель определяется тем же `back`-колбэком, что нёс бывший «Назад»: `sched:day_view:`/`sched:feed` → день, `recur:card:` → карточка серии, `clients:card:`/`clients:active:`/`clients:arch:` → картотека, `sched:chist:` → история клиента. Неизвестная/пустая цель → сегодняшний день расписания (fallback).

Если у клиента заполнен `telegram_chat_id`, следом идёт шаг-вопрос «Уведомить клиента?» (см. таблицу хендлеров). Промежуточные экраны-**подтверждения** (`[Да, удалить] [Отмена]` и т.п.) правилом не затронуты — у них есть осмысленный выбор. Уже-навигирующие действия (перенос разовой записи показывает карточку записи; архив/восстановление клиента переоткрывают карточку на месте) сохраняют прежнее видимое поведение.

## Каталог текстов

Файл: `src/bot/messages.toml`. Загрузка: `src/bot/messages.py::load_messages(path) -> BotMessages` (frozen dataclass).

Обязательные ключи (отсутствие → `RuntimeError` на старте, до приёма апдейтов):

- `[start].welcome` — приветствие при первом валидном `/start <token>`.

## Логи

- `specialist.invite_created` — успешный `create_invite`.
- `specialist.welcomed` — первый успешный `/start <token>`.
- `specialist.invite_replayed` — повтор `/start` для уже привязанного токена.
- `specialist.invite_unknown` — `/start` с неизвестным токеном.
- `client.created` / `client.field_updated` / `client.archived` / `client.restored` — бизнес-события картотеки (в `extra`: `specialist_id`, `client_id`).
- `client.invite_created` / `client.linked` — выдача приглашения и привязка Telegram клиента (в `extra`: `specialist_id`, `client_id`). `client.link_unknown` — переход по неизвестному `cli_`-токену (в `extra`: `token_prefix`).
- `appointment.created` / `appointment.rescheduled` / `appointment.deleted` — бизнес-события записей (в `extra`: `specialist_id`, `appointment_id`, для создания ещё `client_id`).
- `specialist.setting_updated` — изменение настройки расписания (в `extra`: `specialist_id`, `field`; для тоггла рабочих дней `field="working_days"`, для напоминаний `field="reminder_enabled"`/`"reminder_time"`, для дефолта абонемента `field="subscription_default"`).
- `subscription.created` / `subscription.decremented` / `subscription.extended` / `subscription.closed` — бизнес-события абонементов (в `extra`: `specialist_id`, `client_id`, `subscription_id`). Журналятся в сервисе `services/subscriptions.py`.
- `appointment.reminder_sent` / `appointment.reminder_failed` — отправка напоминания клиенту прошла / упала на `TelegramForbiddenError`/`TelegramBadRequest` (в `extra`: `specialist_id`, `client_id`). Журналятся в `bot/scheduler.py`.
- `appointment.reminder_confirmed` / `appointment.reminder_declined` — ответ клиента на напоминание (в `extra`: `specialist_id`, `client_id`). Журналятся в сервисе `apply_reminder_response`.
- `appointment.notified` / `appointment.notify_failed` — ручное уведомление клиента о записи (создание/перенос/отмена) отправлено / упало на `TelegramForbiddenError`/`TelegramBadRequest` (в `extra`: `specialist_id`, `client_id`). Журналятся в хендлере `notify` (`bot/handlers/schedule.py`).

В `extra` всегда передаются `specialist_id` (если известен) и `token_prefix` — первые 6 символов токена, чтобы не светить полный токен в журнале.
