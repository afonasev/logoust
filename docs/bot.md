# Telegram-бот

Бот построен на **aiogram 3** в режиме long-polling. Все пользовательские тексты — в `src/bot/messages.toml`, в коде формулировок нет.

## Сборка приложения

- `src/bot/dispatcher.py::build_dispatcher(messages, session_factory) -> Dispatcher` — собирает `Dispatcher`, подключает роутеры. **Порядок важен:** фичевые роутеры (`clients`, `schedule`, `settings`, `windows`) включаются раньше `start`-роутера — fallback `start` ловит любой текст (онбординг по вставленному коду), поэтому иначе он перехватывал бы reply-кнопки и ввод в визардах. Нераспознанный текст всё равно проваливается в `start`.
- `src/bot/handlers/start.py::build_router(messages, session_factory) -> Router` — регистрирует обработчик `/start` и fallback на вставленный код/ссылку. Различает онбординг специалиста (токен без префикса) и привязку клиента (токен с префиксом `cli_`, см. `src/bot/deeplink.py`): `cli_`-payload идёт в `link_client_by_token`, остальное — в специалистский `consume_invite`.
- `src/bot/handlers/clients.py::build_router(messages, session_factory) -> Router` — роутер управления клиентами: reply-кнопка, inline-меню, визард добавления (FSM), карточка, редактирование полей, архив/возврат, списки. Из карточки клиента ведут переходы в записи (`sched:*`). Состояние FSM — на дефолтном `MemoryStorage` (теряется при рестарте процесса).
- `src/bot/handlers/schedule.py::build_router(messages, session_factory) -> Router` — роутер записей: инлайн-календарь и сетка слотов с пометкой занятости (`build_calendar`, `build_slots_keyboard`), FSM свободного времени и комментария, карточка записи (перенос/удаление с подтверждением). Раздел «📅 Расписание» открывается на **сегодня** с навигацией по дням (◀/▶) и кнопкой «📆 Неделя»; история специалиста — текстом, по календарным неделям. Будущие записи клиента показываются кнопками в его карточке, история клиента — текстом по отдельной кнопке. Колбэки по схеме `sched:<action>[:<args>]`. Тот же роутер содержит **регулярные записи** (`RecurringHandlers`, `_register_recurring`): карточку серии, редактирование и действия пропуска/переноса/остановки даты, колбэки по схеме `recur:<action>[:<args>]` (создание идёт через обычный поток записи, см. ниже). Будущие повторы серий подмешиваются в день/неделю/занятость/карточку клиента как виртуальные `Appointment` (`id=None`, `series_id`/`origin_date` заполнены). Пометку 🔁 несёт транзиентный флаг `recurring_mark` — он стоит только у «чистой» будущей occurrence; перенесённая отдельная дата индивидуализируется и значка не получает; тап ведёт на карточку серии, а не записи.
- `src/bot/handlers/settings.py::build_router(messages, session_factory) -> Router` — роутер настроек расписания: меню, выбор таймзоны из списка российских зон, FSM-ввод начала/конца дня и длины слота с валидацией, мульти-тоггл рабочих дней недели (немедленное сохранение). Колбэки по схеме `settings:<action>[:<value>]`.
- `src/bot/handlers/windows.py::build_router(messages, session_factory) -> Router` — роутер раздела «🪟 Окна»: по reply-кнопке рендерит read-only список свободных окон на ближайшие 5 рабочих дней (`list_free_windows`), без inline-колбэков и пагинации.
- `src/bot/handlers/start.py::make_start_handler(messages, session_factory)` — фабрика хендлера `/start` (вынесена для удобства тестов).
- `src/bot/handlers/start.py::make_token_handler(messages, session_factory)` — фабрика fallback-хендлера: ловит обычный текст и через `extract_token()` достаёт токен из голого кода или deep-link.
- `src/__main__.py` — точка входа: грузит `settings`, инициализирует logging, создаёт `Bot`, `session_factory`, `messages`, `Dispatcher`, запускает `dp.start_polling(bot)`.

## Хендлеры

| Команда | Файл | Что делает |
| --- | --- | --- |
| `/start [<token>]` | `src/bot/handlers/start.py` | Токен с префиксом `cli_` → привязка клиента (`link_client_by_token`, idempotent rebind `chat_id`/`linked_at`; `from_user.username` автозаполняет пустой `contact_telegram`; ответы `clients.linked` / `clients.link_unknown`). Иначе — онбординг специалиста: ищет по токену, идемпотентно проставляет `chat_id`/`username`/`welcomed_at`, четыре ветки (welcomed, already_welcomed, unknown_token, no_token). |
| Текст с кодом/ссылкой | `src/bot/handlers/start.py` (`make_token_handler`) | Fallback: вставленный голый токен или deep-link целиком. Сначала `extract_client_token` (ловит `cli_`-payload → привязка клиента), затем `extract_token` (специалистский онбординг). На сообщения, не похожие на токен, молчит. Зарегистрирован после `/start`. |
| Кнопка «👶 Клиенты» | `src/bot/handlers/clients.py` (`show_menu`/`show_active`) | Reply-кнопка (показывается после онбординга) сразу открывает постраничный список активных клиентов (сортировка по `child_name`). Каждая строка (`_client_row`) — два столбца: клиент (→ `clients:card:<id>`) и его **ближайшая будущая запись** (`nearest_future_by_client`; компактно, без комментария; → карточка записи для переноса) либо «➕ Записать» (`sched:new:<id>`), если записей нет. Действия «➕ Добавить» и «🗄 Архив». Нажатие также сбрасывает любой активный визард. Навигация: `clients:active:<page>`, размер страницы — `_ACTIVE_PAGE_SIZE`. |
| Управление клиентами | `src/bot/handlers/clients.py` (`ClientsHandlers`) | Добавление (FSM: имя ребёнка → имя контакта → телефон → telegram), карточка с общей информацией; **будущие записи клиента — кнопками** (`_future_button` → `sched:card:<id>`, тап открывает запись для переноса/отмены), затем «➕ Записать», строка «✏️ Изменить / 🗂 История / 📦 В архив» и «⬅️ Назад». Ближайшая occurrence активной серии клиента показывается среди будущих записей с пометкой 🔁 (`_appt_callback` → `recur:card:<series>:<origin_date>~<back>`). У активной карточки есть кнопка приглашения в бота (`clients:invite:<id>`, `send_invite`): лениво выдаёт `cli_`-токен (переиспользуя существующий), отдельным **пересылаемым** сообщением шлёт специалисту ссылку `t.me/<bot>?start=cli_<token>` (`build_client_start_link`). Подпись отражает состояние: «Пригласить в бота» / «✅ Привязан» при заполненном `telegram_chat_id`. Архивация — **с подтверждением** (`clients:archiveask:<id>` → `clients:archive:<id>`); возврат из архива — сразу. История записей клиента — **только текстом**, с пагинацией, без кнопок на записи. Кнопка «⬅️ Назад» на карточке клиента **контекстная** (origin в `clients:card:<id>~<back>`; по умолчанию — список активных/архив по статусу). Колбэки по схеме `clients:<action>:<id>`. |
| Архив клиентов | `src/bot/handlers/clients.py` (`show_archive`) | Постраничный архив, сортировка по `archived_at` убыванию (свежие сверху), дата архивации в строке. Размер страницы — `_ARCHIVE_PAGE_SIZE`. Навигация: `clients:list:archived` (стр. 0) и `clients:arch:<page>`; «⬅️ Меню» (`clients:menu`) возвращает к активному списку. Наличие следующей страницы определяется выборкой `page_size + 1` строки (без COUNT). |
| Кнопка «📅 Расписание» | `src/bot/handlers/schedule.py` (`show_feed`/`open_day`/`show_week`) | Reply-кнопка открывает расписание на **релевантный день**: сегодня, если сегодня рабочий день или на сегодня есть записи, иначе ближайший вперёд **показываемый** день (`schedule_landing_day`). Заголовок «дата, день недели» (текст), записи дня — каждая отдельной кнопкой `время · ребёнок` → карточка. Кнопки ◀/▶ (`sched:day_view:<iso>`) ведут на ближайший показываемый день в направлении (`adjacent_shown_day`): пустые нерабочие дни пропускаются, нерабочий день с записями остаётся доступным; если показываемого дня в сторону нет — стрелка не рисуется. «📆 Неделя» (`sched:week`) — read-only обзор 7 дней (сгруппированный текст; пустые дни не выводятся), «🗂 История» — прошедшие записи **только текстом**, сгруппированы по дням, навигация **по календарным неделям** (пн–вс, `sched:hist:<week>`, ◀ старше / ▶ новее), без кнопок на отдельные записи. |
| Запись на приём | `src/bot/handlers/schedule.py` (`ScheduleHandlers`) | Из карточки клиента: календарь (прошлое заблокировано) → слот из сетки или «Другое время» (`ЧЧ:ММ`) → опциональный комментарий → создание и показ карточки записи. Слоты помечаются 🟢 (свободен) / 🔴 (на это время уже есть запись специалиста; `taken_slot_times`), без запрета бронирования. Перенос переиспользует тот же пикер, меняет только `starts_at` и исключает свой текущий слот из «занятых»; удаление — с подтверждением `[Да, удалить] [Отмена]`. На карточке записи кнопка «⬅️ Назад» **контекстная** — возвращает туда, откуда открыли (origin кодируется в `sched:card:<id>~<back-callback>`; по умолчанию — день записи). |
| Регулярная запись | `src/bot/handlers/schedule.py` (`ScheduleHandlers`, `RecurringHandlers`) | Создаётся **в обычном потоке записи**: после выбора времени бот спрашивает «сделать регулярной?» (`sched:reg:1`/`sched:reg:0`, `choose_regular`) перед комментарием. При «да» → `create_series` с `start_date` = выбранная дата (первая встреча) и `weekday` этой даты, затем показывается карточка серии; при «нет» → обычная разовая запись. Тап по 🔁-повтору в дне/неделе/карточке клиента открывает **карточку серии** (`recur:card:<series>:<origin_date>[~<back>]`), повторяющую раскладку карточки записи. Карточка показывает **дату, день недели и время следующей встречи** (`_occurrence_display` — учитывает перенос этой даты, поэтому может отличаться от правила серии). Верхний ряд «📅 Перенести следующую» / «🗑 Отменить следующую» действует на эту дату: перенос (`start_move`) идёт **инлайн-календарём → сеткой слотов** (выбор новой даты, затем времени → `move_date`), отмена — `skip_date` с подтверждением. Второй ряд «⚙️ Настроить» / «🗑 Отменить всё» — на серию (`edit_series`; смена дня недели пересчитывает `start_date`/`materialized_through` — `stop_series` с подтверждением). Календарь переноса использует тот же `build_calendar` с префиксом `recur` (`recur:cal:`/`recur:day:`). Кнопки «⬅️ Назад» и «Отмена» **контекстные** — возвращают на карточку серии / её origin (callback хранится в `~<back>` и в FSM-состоянии). Перенесённая отдельная дата индивидуализируется: значка 🔁 у неё нет. Все действия ограничены своими сериями (`get_for_specialist`). |
| Кнопка «⚙️ Настройки» | `src/bot/handlers/settings.py` (`SettingsHandlers`) | Reply-кнопка открывает настройки расписания: таймзона (из списка рос. зон), начало/конец рабочего дня и длина слота (FSM-ввод с валидацией), рабочие дни недели. Новое значение сразу применяется к генерации сетки, трактовке времени и расчёту окон. Рабочие дни — экран мульти-тоггла (7 кнопок ✅/⬜ Пн–Вс + «⬅️ Назад»): тап инвертирует день (`settings:wd:<idx>`), сразу сохраняет канонизированный набор (`toggle_working_day`) и перерисовывает клавиатуру, без отдельного подтверждения. |
| Кнопка «🪟 Окна» | `src/bot/handlers/windows.py` (`WindowsHandlers`) | Reply-кнопка (верхний ряд меню) присылает одним сообщением свободные окна на ближайшие 5 рабочих дней, сгруппированные по дням. Окно = слот сетки без записи на это время; для сегодня прошедшие окна скрыты. День без свободных окон всё равно показывается со строкой «свободных окон нет» (предсказуемый счёт пяти дней). Если рабочие дни не заданы (`working_days` пуст) — подсказка открыть «Настройки». Экран read-only: inline-кнопок и пагинации нет. |

Доступ ограничивает `SpecialistMiddleware` (inner-middleware, переиспользуется всеми фичевыми роутерами): резолвит специалиста по `chat_id` через `SpecialistsRepo.find_by_chat_id`, кладёт `specialist_id` в данные хендлера, а апдейты от неонбординнутых пользователей роняет. На каждое действие специалиста middleware вызывает `settle` — материализует прошедшие occurrence активных серий в строки `appointments` (планировщика нет; идемпотентно, с дневным guard'ом через `materialized_through`).

Других команд бота (включая `/help`) сознательно нет — приветствие не должно врать о возможностях.

## Каталог текстов

Файл: `src/bot/messages.toml`. Загрузка: `src/bot/messages.py::load_messages(path) -> BotMessages` (frozen dataclass).

Обязательные ключи (отсутствие → `RuntimeError` на старте, до приёма апдейтов):

- `[start].welcome` — приветствие при первом валидном `/start <token>`.
- `[start].already_welcomed` — ответ на повторный `/start` уже использованного токена.
- `[start].unknown_token` — ответ, если токен не найден.
- `[start].no_token` — ответ, если `/start` пришёл без payload.
- `[clients].*` — секция текстов управления клиентами: `button`, заголовки меню и списков (`menu_title`, `list_active_title`, `list_archived_title`, `empty_active`, `empty_archived`), шаблон карточки `card` и метки статуса (`status_active`/`status_archived`, `dash`), приглашения визарда (`ask_child_name`, `ask_contact_name`, `ask_phone`, `ask_telegram`) и подтверждения/ошибки (`added`, `archive_confirm`, `archived`, `restored`, `updated`, `cancelled`, `empty_required`, `need_contact_channel`, `edit_prompt`, `not_found`), а также приглашение клиента в бота (`invite_button`, `invite_button_linked`, `invite_forward`, `linked`, `link_unknown`, `tg_linked_badge` — бейдж «в боте» в поле Telegram карточки). Короткие подписи inline-кнопок живут константами в `clients.py` (навигационная «обвязка», не сценарные тексты).
- `[schedule].*` — секция записей: `button`, день/неделя/история (`day_title`, `day_empty`, `week_title`, `week_empty`, `history_title`, `history_empty`), записи клиента (`client_future_empty`, `client_history_title`, `client_history_empty`), инлайн-блок будущих в карточке клиента (`card_future_title`, `card_appt_line`), шаблоны строк (`line`, `line_full`, `day_header`, `comment_suffix`), шаблон карточки `card`, тексты пикера и FSM (`pick_date`, `pick_time`, `ask_custom_time`, `bad_time`, `ask_comment`), подтверждения (`created`, `rescheduled`, `deleted`, `not_found`, `confirm_delete`) и подписи кнопок (`btn_*`). Подписи навигации календаря (`◀`/`▶`, дни недели) и маркеры занятости слотов (🟢/🔴) — константы в `schedule.py`.
- `[recurring].*` — секция регулярных записей: `mark` (🔁-пометка повтора), приглашения и FSM (`pick_weekday`, `pick_move_date`, `pick_time`, `ask_custom_time`, `bad_time`, `ask_comment`), шаблоны `card`/`line`, подписи кнопок карточки серии (`btn_edit`, `btn_move_date`, `btn_skip_date`, `btn_stop`), подтверждения (`confirm_stop`, `btn_confirm_stop`, `stopped`, `skip_confirm`, `btn_confirm_skip`, `skipped`, `moved`, `created`, `cancelled`, `not_found`).
- `[settings].*` — секция настроек: `button`, шаблон `title` (текущие значения, включая `working_days`), подписи кнопок (`btn_timezone`, `btn_day_start`, `btn_day_end`, `btn_slot`, `btn_working_days`, `btn_back`), приглашения и ошибки FSM (`pick_timezone`, `pick_working_days`, `ask_day_start`, `ask_day_end`, `ask_slot`, `no_working_days`, `bad_time`, `bad_slot`, `saved`, `not_found`). Список таймзон — `RUSSIAN_TIMEZONES`, короткие подписи дней — `RU_WEEKDAYS_SHORT` в `src/domain/schedule.py`.
- `[windows].*` — секция раздела «Окна»: `button`, `title`, `day_header` (шаблон с `{date}`), `empty_day` (строка для дня без свободных окон), `no_working_days` (подсказка, когда рабочие дни не заданы).

Тексты редактируются вручную и не требуют деплоя кода; перечитываются при перезапуске процесса.

## Логи

- `specialist.invite_created` — успешный `create_invite`.
- `specialist.welcomed` — первый успешный `/start <token>`.
- `specialist.invite_replayed` — повтор `/start` для уже привязанного токена.
- `specialist.invite_unknown` — `/start` с неизвестным токеном.
- `client.created` / `client.field_updated` / `client.archived` / `client.restored` — бизнес-события картотеки (в `extra`: `specialist_id`, `client_id`).
- `client.invite_created` / `client.linked` — выдача приглашения и привязка Telegram клиента (в `extra`: `specialist_id`, `client_id`). `client.link_unknown` — переход по неизвестному `cli_`-токену (в `extra`: `token_prefix`).
- `appointment.created` / `appointment.rescheduled` / `appointment.deleted` — бизнес-события записей (в `extra`: `specialist_id`, `appointment_id`, для создания ещё `client_id`).
- `specialist.setting_updated` — изменение настройки расписания (в `extra`: `specialist_id`, `field`; для тоггла рабочих дней `field="working_days"`).

В `extra` всегда передаются `specialist_id` (если известен) и `token_prefix` — первые 6 символов токена, чтобы не светить полный токен в журнале.
