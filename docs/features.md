# Возможности

## Управление списком клиентов

Картотека клиентов специалиста прямо в Telegram. Reply-кнопка «👶 Клиенты» (появляется после онбординга) сразу открывает постраничный список активных клиентов с кнопками «➕ Добавить» и «🗄 Архив». Клиент принадлежит специалисту — каждый видит и редактирует только своих.

### Что умеет

- **Добавить клиента** — пошаговый визард (FSM): имя ребёнка → имя контакта-родителя → телефон (можно пропустить) → Telegram (можно пропустить). Обязательный минимум: имя ребёнка, имя контакта и хотя бы один способ связи (телефон или Telegram); иначе бот вернёт на шаг телефона.
- **Нормализация телефона** — ввод с `8`, `+7`, `7` или без кода приводится к `+7XXXXXXXXXX`; нераспознанный номер (иностранный и т.п.) сохраняется как введён, без блокировки. Telegram хранится без ведущего `@`.
- **Редактировать любое поле** карточки (имя ребёнка, имя/телефон/Telegram контакта, доп. контакты, заметка) — «выбрать поле → прислать значение». Обязательные поля нельзя очистить.
- **Пригласить в бота и привязать Telegram** — кнопка в активной карточке выдаёт пересылаемое сообщение со ссылкой `t.me/<bot>?start=cli_<token>`. Клиент переходит по ней и в один тап («Запустить», без ввода команд) привязывает свой Telegram к карточке — это канал для будущих уведомлений. При привязке, если у клиента есть публичный @username, он автоматически подставляется в поле Telegram карточки (ручной ввод специалиста не затирается). В тексте карточки поле Telegram показывает `@username` (тап открывает переписку) и бейдж «✅ в боте», когда клиент привязан. Повторный переход перезаписывает привязку (idempotent): специалист может протестировать рассылку под своим же аккаунтом без второй учётки, а реальный клиент потом перепривяжется.
- **Архивировать и вернуть** — `active ⇄ archived` без потери данных. Архивация — с подтверждением (`[Да, в архив] [Отмена]`); возврат из архива — сразу. Удаления нет.
- **Списки**: активные — постранично, отсортированы по имени ребёнка (домашний экран раздела). Каждая строка — два столбца кнопок: клиент (→ карточка) и его **ближайшая запись** (компактно, без комментария; → сразу в карточку записи для переноса); если записей нет — вторая кнопка «➕ Записать» (создать запись). Архив — постранично, отсортирован по дате архивации (свежие сверху), с датой архивации в каждой строке и навигацией ◀/▶.

### Что **не** входит в этот сценарий

- Проверка дубликатов, жёсткая валидация телефона/Telegram, пагинация списков — сознательно отложено (YAGNI).
- Уведомления клиенту (о записи, напоминания) — канал привязки готов, сами уведомления идут отдельной фичей.
- Одноразовость/протухание клиентского токена — пока ссылка бессрочна и переиспользуема (YAGNI).

## Записи клиентов на приём

Учёт записей клиентов прямо в боте: создание из карточки клиента, общая лента расписания и история. Записи принадлежат специалисту — каждый видит только свои.

### Что умеет

- **Создать запись** из карточки клиента (кнопка «➕ Записать»): инлайн-календарь (прошлые дни заблокированы) → время из сетки слотов расписания или произвольный ввод `ЧЧ:ММ` → опциональный комментарий («Пропустить»/«Отмена») → карточка записи. В сетке слотов каждый слот помечен эмодзи: 🟢 — свободен, 🔴 — на это же время у специалиста уже есть запись (подсказка не блокирует — записать поверх можно). При переносе собственный текущий слот записи не считается занятым.
- **Расписание по дням** (reply-кнопка «📅 Расписание»): открывается на релевантный день — сегодня, если сегодня рабочий день или есть записи; иначе ближайший вперёд рабочий день (или день с записями). Заголовок «дата, день недели», под ним каждая запись отдельной **кнопкой** (`время · имя ребёнка`), по тапу — карточка записи. Кнопки ◀/▶ переключают день, **пропуская пустые нерабочие дни** (выходные без записей) и останавливаясь на ближайшем рабочем дне, дне с записью или дне, на который приземляется будущий повтор активной серии (в т.ч. перенесённый туда); нерабочий день с записью или повтором серии не пропускается. «📆 Неделя» показывает ближайшие 7 дней как сгруппированный по дням обзор (только дни с записями).
- **История** записей специалиста — прошедшие, только текстом, сгруппированы по дням; навигация **по календарным неделям** (пн–вс; ◀ старше / ▶ новее), без кнопок на отдельные записи.
- **В карточке клиента** будущие записи показываются **кнопками** под общей информацией — тап открывает запись, чтобы перенести или отменить её. Прошедшие — по кнопке «🗂 История записей» (только текстом, с пагинацией).
- **Перенос** (по тапу на запись) — новые дата и время тем же пикером; меняется только время, комментарий сохраняется.
- **Удаление** — с подтверждением `[Да, удалить] [Отмена]`, **жёсткое, без архива**.
- **Время** хранится в UTC; ввод и показ конвертируются через таймзону специалиста. Граница «будущее/история» и группировка по дням — по календарному дню в таймзоне специалиста, а не по времени сервера (см. [решение](decisions/2026-06-04_appointment_time_in_utc_per_specialist_tz.md)).

### Что **не** входит в этот сценарий

- Проверка пересечений записей — вне рамок (записать поверх занятого слота можно). Обзор свободных окон вынесен в отдельный раздел «Окна» (ниже).
- Напоминания клиентам/специалисту о записях — отдельная будущая фича.
- Общий календарь нескольких специалистов.

## Регулярные записи

Еженедельно повторяющаяся запись клиента («серия»): задал день недели и время один раз — встреча повторяется каждую неделю, пока её не остановят, и ведёт себя как обычная запись во всех экранах занятости и расписания.

### Что умеет

- **Создать серию** прямо в обычном потоке записи: после выбора даты и времени бот спрашивает «сделать запись регулярной — повторять каждую неделю?». При «да» создаётся серия, начиная с выбранной даты (она и есть первая встреча); при «нет» — обычная разовая запись. Отдельного пункта меню для регулярной записи нет.
- **Пометка в списках.** Регулярные будущие записи помечены значком 🔁 в карточке клиента, списке клиентов, расписании по дням и на неделю — сразу видно, где разовые, а где регулярные. Если у регулярной записи перенести отдельную дату на другое время, она индивидуализируется и значок 🔁 на ней больше не показывается.
- **Занятость везде.** Будущие повторы помечают слот занятым (🔴) в инлайн-календаре при создании/переносе записей, исключают слот из «Окон», показываются в расписании по дням и на неделю и как ближайшая встреча в карточке клиента.
- **История «по факту».** Когда плановая дата проходит, встреча становится обычной записью в истории (материализуется при заходе специалиста в бота). Прошедшие дни расписания и история не отличают её от разовой записи.
- **Карточка серии** (тап по 🔁-повтору): клиент, **дата, день недели и время следующей встречи** (если её перенесли — показываются новые), комментарий и действия ниже.
- **Настроить серию** — день недели, время, комментарий. Правка затрагивает только будущие повторы; прошлые встречи в истории не меняются.
- **Отменить всё** — с подтверждением: будущие повторы исчезают, прошлые записи остаются.
- **Отменить следующую** — конкретная встреча отменяется (ребёнок заболел), остальные повторы не затрагиваются.
- **Перенести следующую** — выбираешь новую дату и время; исходный слот освобождается, встреча показывается на новой дате, остальные повторы не меняются.
- **Доступ** — серию видит и меняет только её владелец-специалист.
- **Время** трактуется как настенное в таймзоне специалиста и конвертируется в UTC отдельно на каждую дату (корректно при переходах летнего времени). Граница «материализованное прошлое / вычисляемое будущее» — сегодняшний день (см. [решение](decisions/2026-06-05_recurring_materialized_past_virtual_future.md)).

### Что **не** входит в этот сценарий

- Любые периоды, кроме «каждую неделю» (никаких «через неделю», «по чётным», месячных).
- Учёт фактической явки/неявки ребёнка.
- Напоминания по регулярным записям.

## Абонементы

Учёт абонементов клиента на несколько встреч прямо в боте: сколько куплено, сколько осталось, когда создан — с простыми действиями списания, продления и закрытия. Раньше специалист вёл остаток вручную; теперь это в карточке.

### Что умеет

- **Кнопка на карточке клиента.** На активной карточке: если активного абонемента нет — «➕ Создать абонемент»; если есть — «🎫 Абонемент · {остаток}» с переходом на карточку абонемента. На архивной карточке кнопка не показывается.
- **Раздел «🎫 Абонементы» в главном меню.** Постраничный список активных абонементов всех клиентов (имя ребёнка + остаток встреч), переход на карточку. Оттуда — история закрытых абонементов (имя + дата закрытия); карточка закрытого открывается только для просмотра, без действий.
- **Создать абонемент** — спрашивает число встреч; в подсказке подставлен дефолт специалиста (можно принять кнопкой или ввести своё число). Создаётся запись со статусом «активен», `куплено = осталось = N`. **Не более одного активного абонемента на клиента.**
- **Карточка абонемента** — дата создания, сколько куплено, сколько осталось и действия.
- **Вычесть встречу** — остаток −1 (вручную, не связано со встречами/расписанием). На нуле списание не уходит в минус — бот сообщает, что списывать нечего.
- **Продлить** — куплено и остаток растут на введённое число встреч (кумулятивно); число подтверждается или вводится, с дефолтом в подсказке.
- **Закрыть** — с подтверждением, даже если остаток больше нуля. Закрытый абонемент сохраняется в БД (не удаляется); после закрытия у клиента снова можно создать новый.
- **Дефолтное число встреч** настраивается в «⚙️ Настройки» (стартовое значение 8).
- **Доступ** — абонемент видит и меняет только его владелец-специалист (через клиента).

### Что **не** входит в этот сценарий

- Связь со встречами/расписанием — авто-списание при проведении встречи (остаток меняется только вручную).
- Уведомления клиенту об остатке/исчерпании.
- Авто-закрытие абонемента при архивации клиента.

## Напоминания клиентам о записи

Бот сам накануне напоминает привязанному клиенту о встрече и собирает подтверждение, а специалист видит итог одним взглядом и узнаёт об отказе сразу.

### Что умеет

- **Авто-напоминание о записи на завтра.** Раз в день (по умолчанию **12:00** настенного времени специалиста) бот шлёт каждому привязанному клиенту (`telegram_chat_id` задан), у которого есть запись на завтра, сообщение с датой/временем и кнопками «✅ Подтверждаю» / «❌ Не смогу прийти». В срез завтрашнего дня входят и виртуальные повторы серий — так же, как их показывает расписание. Непривязанным клиентам ничего не шлётся.
- **Подтверждение и отказ.** Ответ клиента сохраняется как статус напоминания (`pending`/`confirmed`/`declined`); ответ можно изменить в любой момент.
- **Видно специалисту.** В списках записей — дневном экране расписания, списке клиентов (кнопка ближайшей записи) и в будущих записях на карточке клиента — подтверждённая запись помечается ✅, а отклонённая («не смогу прийти») — ❌. На карточке самой записи (и серии для повтора) подтверждение показывается строкой-пометкой.
- **Отказ → сразу сигнал.** При каждом переходе ответа в «отказ» специалисту приходит сообщение с кнопкой «→ Открыть запись» (ведёт на карточку записи/серии). Авто-отмены нет — специалист сам переносит/отменяет. Повторный тот же ответ нового уведомления не порождает.
- **Настройка.** Напоминания можно выключить, время отправки — настраиваемое (по умолчанию включены, 12:00) — в «⚙️ Настройки».
- **Надёжность.** Проход выполняется ровно один раз в день (антидубль через журнал; устойчив к минутному тику и рестарту), с догоном после простоя. Сбой доставки одному клиенту (заблокировал бота) не прерывает остальных.

### Что **не** входит в этот сценарий

- Напоминание о записи, созданной **после** дневного прохода (только батч; ручной канал — отдельная фича `appointment-notify`).
- Авто-отмена/перенос при отказе, напоминание за N минут до встречи, многократные напоминания и эскалации.
- Запрет менять ответ — ответ перезаписывается всегда.

## Настройки расписания

Reply-кнопка «⚙️ Настройки» открывает настройки расписания специалиста:

- **Таймзона** — выбором из курированного списка российских зон (дефолт `Asia/Yekaterinburg`).
- **Начало** и **конец** рабочего дня (`ЧЧ:ММ`) и **длина слота** в минутах — FSM-ввод с валидацией.
- **Рабочие дни недели** — мульти-тоггл (7 кнопок ✅/⬜ Пн–Вс): тап инвертирует день и сразу сохраняется, без подтверждения. Дефолт — Пн–Пт. Применяются при расчёте свободных окон («ближайшие рабочие дни»).
- **Напоминания клиентам** — тумблер вкл/выкл и время отправки (`ЧЧ:ММ`, дефолт 12:00). См. раздел «Напоминания клиентам о записи».
- **Встреч в абонементе** — дефолтное число встреч, подставляемое в подсказку при создании/продлении абонемента (положительное число, дефолт 8). См. раздел «Абонементы».
- Слоты генерируются от начала дня шагом в длину слота, пока старт слота меньше конца дня (например `14:00, 14:50, … , 19:00` при `14:00`/`19:40`/`50`). Новые значения сразу применяются к следующей сетке и трактовке времени.
- Существующим специалистам настройки проставляются дефолтами при миграции.

## Свободные окна

Reply-кнопка «🪟 Окна» (верхний ряд главного меню) — быстрый read-only обзор свободного времени, чтобы оперативно ответить клиенту «когда можно прийти».

- Одно сообщение со свободными окнами на **ближайшие 5 рабочих дней** (рабочие дни — из настройки), сгруппированное по дням. Пагинации нет.
- **Свободное окно** = слот из сетки настроек (`generate_slots`), на который у специалиста нет записи **и** нет повтора активной регулярной серии (с учётом пропусков/переносов этой даты). Окна каждого дня — по возрастанию времени.
- Нерабочие дни недели пропускаются и не входят в счёт пяти дней.
- Для **сегодня** окна, чьё время уже прошло (в таймзоне специалиста), не показываются.
- День, где все слоты заняты или прошли, всё равно показывается со строкой «свободных окон нет» — ради предсказуемого счёта пяти дней.
- Если рабочие дни не заданы (пустой набор) — экран сообщает об этом и предлагает открыть «Настройки».
- Экран строго read-only: тап по окну ничего не делает; запись по-прежнему идёт из карточки клиента.

## Онбординг специалиста через invite-ссылку

Сценарий первого подключения специалиста к ассистенту. Цель — получить рабочий канал коммуникации в Telegram, к которому далее будут пристёгиваться остальные фичи.

### Поток

1. Администратор запускает `make create_invite`. CLI создаёт в таблице `specialists` запись с уникальным криптостойким `invite_token`, без `telegram_chat_id`, без `welcomed_at`.
2. В stdout печатается одна строка — Telegram deep-link:
   `https://t.me/<TELEGRAM_BOT_USERNAME>?start=<token>`.
3. Администратор передаёт ссылку специалисту любым удобным способом.
4. Специалист подключается одним из двух способов:
   - **по ссылке** — переходит по deep-link, Telegram отправляет боту `/start <token>`;
   - **вставкой** — отправляет боту обычным сообщением голый код-токен или ссылку целиком; бот извлекает токен и обрабатывает так же, как `/start <token>`.
5. Бот сопоставляет токен с записью, фиксирует `telegram_chat_id`, `telegram_username`, `welcomed_at = now()` и отправляет приветствие на русском.

### Ветки обработки токена

| Ситуация | Что делает бот |
| --- | --- |
| Валидный, ещё не использованный токен | Пишет в запись `chat_id`, `username`, `welcomed_at`, отправляет `[start].welcome`. |
| Тот же токен ещё раз (`welcomed_at` уже стоит) | Ничего не меняет в БД, отправляет `[start].already_welcomed`. |
| Неизвестный токен | Не меняет БД, отправляет `[start].unknown_token`. |
| `/start` без payload | Не меняет БД, отправляет `[start].no_token`. |
| Текст, не похожий на токен/ссылку | Не меняет БД, не отвечает (молча игнорирует). |

### Ограничения и решения

- Язык интерфейса — только русский, без i18n-инфраструктуры.
- Связь по `username` или ручной ввод `chat_id` не используются: выбран deep-link с одноразовым токеном.
- Webhook-режим не используется, бот работает в long-polling.
- Имя специалиста в БД не хранится — по требованию заказчика.
- Тексты, видимые специалисту, живут в `src/bot/messages.toml` (не в коде).
- Один Telegram-аккаунт может быть привязан только к одной записи специалиста (`UNIQUE`-индекс на `telegram_chat_id`).

### Что **не** входит в этот сценарий

- Команды бота, кроме `/start` (даже `/help`).
- Веб-интерфейс, REST/HTTP API.
- Отзыв/деактивация токенов, список специалистов — отдельные изменения позже.
