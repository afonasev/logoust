## Context

Кабинет переходит к основной цели — расписанию и напоминаниям, для которых нужна
картотека клиентов. Сейчас в проекте есть только capability `specialists` (онбординг по
инвайтам): один domain-entity `Specialist`, `SpecialistsRepo`-протокол, реализация на
SQLAlchemy 2.0 (`Mapped`/`mapped_column`, async-сессии), aiogram-router `start`.

Эта capability `clients` — первый CRUD-функционал кабинета. Архитектурно он повторяет
сложившийся паттерн (`domain` → `services` → `infrastructure` → `bot`), но впервые вводит:
сущность с владельцем (FK на `specialists`), редактируемые поля, статусную модель
(active/archived) и доменную нормализацию ввода (телефон). Развилки модели контактов,
владения и обязательных полей уже решены в обсуждении — этот документ их фиксирует.

## Goals / Non-Goals

**Goals:**

- Картотека клиентов специалиста: добавление, редактирование полей, архивация/возврат,
  списки активных и архивных.
- Один владелец у клиента (`specialist_id`); специалист видит и меняет только своих.
- Нормализация телефона контакта к канону `+7XXXXXXXXXX` с безопасным фолбэком.
- Сохранить совместимость с правилами проекта: Clean Architecture, async-сессии,
  100% покрытие, тексты в `messages.toml`.

**Non-Goals:**

- Расписание, слоты, напоминания — отдельные будущие capability.
- Реальное связывание Telegram-аккаунта родителя с чатом бота (хранится только как текст).
- Жёсткая валидация телефона/Telegram, проверка дубликатов, пагинация списков.
- Удаление клиентов (только архивация) и общий для кабинета доступ к клиентам.
- Telegram Mini App: UI этой capability — только нативные сообщения/кнопки бота.

## Decisions

### 1. Модель: одна таблица `clients`, контакт «плоскими» полями

Клиент = карточка ребёнка с одним структурированным контактом родителя и свободными
полями. Без отдельной таблицы `contacts`.

```
Client
  id              int PK
  specialist_id   FK → specialists  (владелец)
  child_name      str    — обязательно, непустое
  contact_name    str    — обязательно, непустое
  contact_phone   str?   — нормализуется; phone|telegram: хотя бы одно непустое
  contact_telegram str?  — хранится без ведущего @
  extra_contacts  str?   — свободный текст, НЕ нормализуется
  note            str?   — свободный текст
  status          enum   — "active" | "archived"
  archived_at     dt?    — выставляется при архивации, NULL у активного
  created_at      dt
  updated_at      dt     — обновляется при любом изменении полей/статуса
```

**Почему так, а не таблица `Contact` (1→N):** YAGNI. Реально у ребёнка один основной
контакт; второй родитель/бабушка идут в `extra_contacts` свободным текстом. Отдельная
таблица потребовала бы вложенного под-CRUD по контактам на кнопках — дорогой UX ради
редкого случая. Выбор зафиксирован пользователем (вариант «один контакт + доп. текст»).
Цена: для будущих напоминаний адресуем только основной контакт — это приемлемо.

**Альтернатива (отклонено):** плоские текстовые поля для всех контактов — теряем
структуру основного контакта, которая нужна для напоминаний.

### 2. Статус — enum `active`/`archived`, без удаления

Архивация — это смена `status` + установка `archived_at`; возврат — `status=active`,
`archived_at=NULL`. Жёсткого удаления нет: данные клиента не теряем никогда.

**Почему enum, а не `is_archived: bool`:** статус — доменное понятие, и enum читаемее в
коде/логах; список фильтруется `WHERE status = ...`. `archived_at` хранится отдельно как
факт времени архивации (полезно для будущей аналитики/сортировки архива).

### 3. Владелец через FK, фильтрация на уровне запросов

`clients.specialist_id` — обязательный FK. Все выборки/мутации репозитория принимают
`specialist_id` и фильтруют по нему. Это и модель данных, и граница доступа: специалист
не может прочитать/изменить чужого клиента, потому что запрос всегда сужен его id.

**Риск-зеркало:** при будущем «общем кабинете» (отклонённый вариант) фильтр придётся
снимать — но это осознанный откладываемый сценарий, а не блокер.

### 4. Нормализация телефона — чистая доменная функция с фолбэком

Отдельная функция (например `normalize_phone(raw: str) -> str`) в `domain` (без
зависимостей), переиспользуется при создании и редактировании:

```
strip всё кроме цифр →
  11 цифр, ведущая 8  → отрезать 8 → 10 цифр
  11 цифр, ведущая 7  → отрезать 7 → 10 цифр
  10 цифр             → как есть   → 10 цифр
  иначе               → НЕ сводится
сводится     → "+7" + 10 цифр
не сводится  → вернуть исходный ввод как есть (trim)   ← фолбэк, выбор пользователя (Б)
```

**Почему фолбэк, а не отказ:** пользователь выбрал не блокировать специалиста (иностранные
/ нестандартные номера всё равно сохраняем). Цена: в `contact_phone` теоретически возможен
не-канон — допустимо, т.к. это поле для людей, а не для автодозвона.

Пустой/отсутствующий телефон не нормализуется (остаётся пустым) — нормализация только для
непустого ввода.

### 5. Telegram-username контакта — нормализуем срез `@`

Непустой `contact_telegram` храним без ведущего `@` (`@masha` → `masha`). Формат не
валидируем (YAGNI). Тоже маленькая чистая функция в `domain` рядом с телефоном.

### 6. Слои и распределение ответственности (по conventions проекта)

- `domain/client.py`: `@dataclass(slots=True) Client`, `ClientStatus` enum,
  `ClientsRepo` Protocol, чистые `normalize_phone` / `normalize_telegram`,
  доменные исключения по необходимости.
- `services/clients.py`: use-case функции (`add_client`, `edit_client_field`,
  `archive_client`, `restore_client`, `list_clients`) — валидация обязательного минимума
  здесь, перед обращением к repo; логируют бизнес-события (`client.created`,
  `client.archived`, …) с `extra={"specialist_id", "client_id"}`.
- `infrastructure/clients_repo.py`: `ClientORM(Base)` + `to_domain` + `SqlAlchemyClientsRepo`
  (повторяет паттерн `specialists_repo.py`: `flush` → `commit`, `session.get` для
  единичной записи, `select(...).where(specialist_id == ...)` для списков).
- `bot/handlers/clients.py`: aiogram-router, FSM добавления/редактирования, навигация
  inline-кнопками по спискам и карточке. Все тексты — в `src/bot/messages.toml`.

Валидацию обязательных полей держим в `services` (а не в хендлере и не в ORM), чтобы она
не зависела от способа ввода (бот сейчас, потенциально Mini App позже).

### 7. Миграция

Новая Alembic-миграция: таблица `clients`, FK `specialist_id → specialists.id`, индексы по
`specialist_id` и по `(specialist_id, status)` (списки всегда фильтруются обоими).
`status` хранится как строковый enum. `make run` применяет миграции до старта бота
(существующее поведение capability `specialists`).

## Risks / Trade-offs

- **Не-канон в `contact_phone` из-за фолбэка** → допустимо для capability картотеки; когда
  появятся автонапоминания, добавим явный флаг «номер не распознан» или проверку на этапе
  отправки. Не блокируем сейчас.
- **Один контакт вместо N** → второй родитель в свободном `extra_contacts` не адресуем
  программно. Принято осознанно; при реальной потребности заводим таблицу `contacts`
  отдельным change без слома данных (поля переезжают в первую строку контакта).
- **FSM редактирования полей** → много состояний (по полю на правку). Митигируем единым
  обобщённым флоу «выбрать поле → прислать новое значение», а не отдельным состоянием на
  каждое поле.
- **Рост списка без пагинации** → у одного специалиста десятки клиентов, не тысячи; пока
  безопасно. Если список перестанет влезать в сообщение — вводим пагинацию отдельным change.

## Migration Plan

1. Добавить ORM-модель `ClientORM` и сгенерировать Alembic-миграцию (`clients` + индексы + FK).
2. Реализовать `domain` → `services` → `infrastructure` → `bot` слой за слоем, с тестами
   на success/failure пути (включая таблицу примеров нормализации телефона).
3. Зарегистрировать router в dispatcher; добавить тексты в `messages.toml`.
4. Обновить `docs/` (database, architecture, bot, features) и `README.md`,
   синхронизировать `openspec/specs/clients/`.

Откат: миграция аддитивна (новая таблица, существующие не трогаются) — `alembic downgrade`
удаляет таблицу `clients`, остальной код кабинета не затрагивается.
