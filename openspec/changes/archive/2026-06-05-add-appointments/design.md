## Context

Картотека клиентов (`clients`) и онбординг специалистов (`specialists`) уже работают.
Записи на приём — следующий слой: сущность, принадлежащая одновременно специалисту и
клиенту. Проект следует Clean Architecture (`domain` → `services` → `infrastructure`/`bot`),
хранит все `datetime` как aware UTC (`datetime.now(UTC)`), использует SQLAlchemy 2.0 +
async, Alembic-миграции и aiogram с FSM-визардами и инлайн-клавиатурами. Сторонних
библиотек для календаря/таймзон в проекте нет и заводить не хотим.

Развилки решены в режиме исследования: храним время в UTC с таймзоной на специалисте,
ввод даты/времени — собственный инлайн-пикер, граница «будущее/история» по календарному
дню, удаление с подтверждением, параметры сетки времени и таймзона — настройки специалиста.

## Goals / Non-Goals

**Goals:**
- Специалист создаёт, переносит и удаляет записи клиента, не выходя из бота.
- Корректная работа во времени независимо от таймзоны сервера деплоя.
- Будущие записи и история доступны на уровне специалиста (вся лента) и клиента (его записи).
- Настройки расписания (таймзона, рабочий день, слот) редактируются специалистом.
- Никаких новых зависимостей.

**Non-Goals:**
- Проверка пересечений записей, занятость слотов, «свободные окна» — явно вне рамок.
- Напоминания клиентам/специалисту о записях (отдельная будущая фича).
- Повторяющиеся/серийные записи.
- Несколько специалистов на один кабинет, общий календарь — каждый видит только свои записи.

## Decisions

### Решение 1: время в aware UTC, таймзона — на специалисте

Поле `appointments.starts_at` хранится как aware UTC, консистентно со всей базой.
У специалиста появляется `timezone` (IANA-строка, дефолт `Asia/Yekaterinburg`). Ввод из
пикера трактуется как настенное время в таймзоне специалиста и конвертируется в UTC через
`zoneinfo.ZoneInfo`; показ — обратная конверсия. «Сегодня», граница будущее/история,
блокировка прошлого и группировка по дням считаются по календарному дню в таймзоне
специалиста: `datetime.now(UTC).astimezone(tz).date()`.

- **Почему не naive local**: деплой может идти в произвольной таймзоне, опираться на
  локальное время сервера нельзя (требование разработчика). UTC + явная TZ специалиста
  снимает зависимость от окружения.
- **Почему IANA, а не целый оффсет**: `zoneinfo` в stdlib, корректно навсегда; выбор в
  настройках ограничен курированным списком российских зон, так что UX от полноты IANA не
  страдает. Россия без DST, но IANA не мешает.

### Решение 2: собственный инлайн-пикер даты и времени (без зависимостей)

Дата — инлайн-календарь на месяц: шапка `[◀] Месяц Год [▶]`, строка дней недели, сетка
дней. Навигация по месяцам через callback. Дни раньше «сегодня» (в TZ специалиста)
отображаются неактивной заглушкой и не порождают выбор. После выбора дня — сетка слотов.

Слоты генерируются от `day_start` шагом `slot_minutes`, пока старт слота `< day_end`
(пример: 14:00, 14:50, 15:40, … при day_end 19:40 последний слот 19:00). Плюс кнопка
`🕑 Другое время` → FSM-шаг с текстовым вводом `ЧЧ:ММ` (валидация формата и диапазона
00:00–23:59).

- **Почему не сторонний календарь (aiogram-calendar)**: YAGNI и отсутствие зависимостей в
  проекте; месячный календарь и сетка слотов — простые клавиатуры.
- **Состояние пикера** передаётся через `callback_data` (год-месяц при навигации, выбранная
  дата при выборе слота), без раздувания FSM. Свободный ввод времени и шаг комментария —
  через FSM, как существующие визарды клиентов.

### Решение 3: модель данных

```
appointments
  id            PK
  specialist_id FK -> specialists.id   NOT NULL
  client_id     FK -> clients.id       NOT NULL
  starts_at     DateTime (aware UTC)   NOT NULL
  comment       Text                   NULL
  created_at    DateTime (UTC)         NOT NULL
  updated_at    DateTime (UTC)         NOT NULL
  Index ix_appointments_specialist_starts (specialist_id, starts_at)
  Index ix_appointments_client_starts     (client_id, starts_at)
```

`specialists` получает: `timezone` String, `day_start` / `day_end` (хранятся как `String(5)`
`HH:MM` либо `Time` — выбрать в реализации; время суток без даты), `slot_minutes` Integer.

- **Почему отдельная capability `appointments`, а не расширение `clients`**: запись —
  самостоятельная сущность с собственным жизненным циклом и принадлежностью двум сторонам;
  лента расписания специалиста не привязана к карточке клиента.
- **Доступ только к своим**: все запросы фильтруются по `specialist_id` (как у клиентов).

### Решение 4: будущее/история по календарному дню

Запись «будущая», если её дата (в TZ специалиста) `>= сегодня`; иначе — «история».
Сегодняшние записи висят в будущих весь день, даже если время уже прошло. Будущие
сортируются по возрастанию `starts_at` и группируются по дню; история — по убыванию, с
пагинацией (паттерн «limit+1» из `list_archived`).

### Решение 5: жёсткое удаление с подтверждением

Удаление физически удаляет строку (без архива — в отличие от клиентов). Перед удалением —
шаг подтверждения `[Да, удалить] [Отмена]`, т.к. операция необратима. Перенос меняет только
`starts_at` (дата+время через тот же пикер), `comment` не трогает.

### Решение 6: форматирование дат на русском вручную

Месяц (в родительном: «мая») и день недели маппятся вручную таблицами, без зависимости от
системной локали (её может не быть в окружении деплоя). Формат строки записи:
`{child} · {dd} {month}, {weekday} · {HH:MM}` + комментарий при наличии.

## Risks / Trade-offs

- **Неверная таймзона специалиста → смещённые записи** → дефолт `Asia/Yekaterinburg`,
  явный выбор из списка в настройках, показ выбранной зоны в карточке настроек.
- **`zoneinfo` без базы tzdata в минимальном окружении** (часто на Windows/slim-контейнерах)
  → задокументировать зависимость от системного tzdata; при необходимости полагаемся на то,
  что Linux-деплой содержит её. (Пакет `tzdata` не добавляем, пока не подтвердится проблема.)
- **Свободный ввод времени даёт «ломаные» минуты** (14:37) → допускаем осознанно как
  fallback; валидируем только формат и диапазон, не привязываем к сетке.
- **Много будущих записей в одном дне раздувают сообщение/клавиатуру** → будущая лента
  тоже может потребовать ограничения; на старте показываем как есть, при росте добавим
  пагинацию (вне текущих рамок, отметить).
- **Миграция существующих специалистов** → колонки настроек добавляются с server-default
  (`Asia/Yekaterinburg`, `09:00`, `20:00`, `60`), чтобы у уже онбординнутых специалистов
  сразу была рабочая сетка.

## Migration Plan

1. Alembic-ревизия: `CREATE TABLE appointments` + два индекса; `ALTER TABLE specialists`
   добавляет `timezone`, `day_start`, `day_end`, `slot_minutes` с server-default.
2. Деплой через `make run` (применяет миграции до старта polling).
3. Откат: down-ревизия удаляет таблицу и колонки. Данные записей при откате теряются
   (фича новая, обратной совместимости не требуется).
