## Context

Сетка слотов уже генерируется из настроек специалиста (`generate_slots(day_start, day_end, slot_minutes)` в `src/domain/schedule.py`), а занятость дня вычисляется в `taken_slot_times()` (`src/services/appointments.py`). Свободные окна — это разность этих двух множеств. Недостающие части:

1. В настройках нет понятия «рабочие дни недели» — диапазон времени одинаков для всех дней. Чтобы «ближайшие 5 рабочих дней» имели смысл, нужно новое поле `working_days`.
2. Нет обзорного экрана: занятость видна только внутри одного дня при создании записи.

Настройки персистятся немедленно при выборе (паттерн `set_timezone`: сохранил → перерисовал меню), без отдельной кнопки «Сохранить» и без FSM для выбора из набора.

## Goals / Non-Goals

**Goals:**
- Специалист задаёт рабочие дни недели мульти-тогглом в настройках; значение сразу применяется.
- Read-only экран «Окна»: свободные окна на ближайшие 5 рабочих дней одним сообщением, сгруппированные по дням, без пагинации.
- Для сегодняшнего дня скрываем окна, чьё время уже прошло.
- Переиспользуем `generate_slots` и `taken_slot_times`, без новых тяжёлых запросов.

**Non-Goals:**
- Запись по нажатию на окно — экран строго read-only (тап ничего не делает).
- Индивидуальный диапазон времени/слота на каждый день недели — диапазон остаётся общим.
- Праздники, отпуска, единичные нерабочие даты — только дни недели.
- Постраничная навигация по окнам.

## Decisions

### 1. Хранение `working_days` — строка индексов дней недели

`working_days` хранится как канонически отсортированная строка индексов `date.weekday()` (Пн=0 … Вс=6), например `"0,1,2,3,4"`. Дефолт — Пн–Пт.

- **Почему так:** в духе текущих настроек (`day_start`/`day_end` — простые строки, парсятся хелперами из `domain/schedule.py`). Server-default в миграции тривиален; значение читаемо в БД.
- **Альтернатива — битмаска int (0–127):** компактнее, но непрозрачно в БД и требует битовой арифметики при отладке. Выгоды на наших объёмах нет.
- Домен предоставляет хелперы `parse_working_days(raw) -> list[int]` (отсортированный, без дублей) и `format_working_days(days) -> str` (канонизация перед записью).

### 2. «Ближайшие N рабочих дней» — чистый хелпер в `domain/schedule.py`

`next_working_days(today: date, working_days: set[int], count: int) -> list[date]` идёт от `today` вперёд, отбирая дни, чей `weekday()` входит в `working_days`, пока не наберёт `count`.

- Сегодня включается в набор, если это рабочий день (даже если все окна уже прошли — день показывается со строкой «свободных окон нет», чтобы счёт из 5 был предсказуем).
- **Guard:** пустой `working_days` → возвращаем `[]` (иначе бесконечный цикл). Экран в этом случае показывает «рабочие дни не заданы».

### 3. Расчёт свободных окон — в сервисе записей

Новая функция в `src/services/appointments.py`: для каждого дня из `next_working_days` берёт `generate_slots(settings)`, вычитает `taken_slot_times(repo, day)`; для сегодняшнего дня дополнительно отбрасывает слоты со временем `<= now` (в таймзоне специалиста). Возвращает структуру «день → список свободных `HH:MM`».

- Один запрос занятости на день (как в существующем дневном экране) — N≤5, дёшево.
- Сравнение «слот в прошлом» делаем в настенном времени таймзоны специалиста, консистентно с остальной логикой (см. `utc_to_wall`, `today_in_tz`).

### 4. Выбор рабочих дней — немедленный тоггл, не FSM

В меню настроек — ряд из 7 кнопок-тогглов (✅/⬜ + Пн…Вс). Тап инвертирует день, сразу сохраняет полный новый набор через `update_settings` и перерисовывает экран. Это повторяет flow таймзоны и не требует текстового ввода/валидации диапазона.

- В `services/specialists.py` добавляем операцию тоггла одного дня (читаем текущий набор, инвертируем, пишем канонизированную строку).

### 5. «Окна» — отдельная кнопка и отдельный хендлер

Третья кнопка в верхнем ряду главного reply-меню и отдельный модуль-хендлер (`src/bot/handlers/`), а не вкладка внутри «Расписания».

- **Почему так:** «Окна» — самостоятельный read-only сценарий с одной точкой входа; смешивать с FSM создания/переноса записи в `schedule.py` усложнит и без того большой модуль.
- **Альтернатива — кнопка внутри экрана «Расписание»:** меньше кнопок в главном меню, но прячет обзорную функцию на второй уровень; пользователь явно просил пункт в главном меню.

## Risks / Trade-offs

- **[Пустой `working_days` → бесконечный цикл в `next_working_days`]** → guard на пустой набор возвращает `[]`; экран показывает подсказку открыть настройки. Покрыто тестом.
- **[Несогласованность индексации дней]** `date.weekday()` (Пн=0) должен совпадать с подписями кнопок и порядком хранения → единый источник нумерации в `domain/schedule.py`, тест на round-trip parse/format.
- **[Длинное сообщение]** 5 дней × ~11 окон ≈ 0.5 КБ — в лимите Telegram 4096 с запасом; отдельной обрезки не делаем.
- **[День без свободных окон]** (всё занято или прошло) всё равно входит в 5 и показывается со строкой «свободных окон нет» — намеренно, ради предсказуемого счёта дней.

## Migration Plan

- Новая Alembic-миграция: добавляет колонку `working_days` в `specialists`, тип `String`, `nullable=False`, `server_default="0,1,2,3,4"` — существующие специалисты получают Пн–Пт. Откат — `drop_column`.
