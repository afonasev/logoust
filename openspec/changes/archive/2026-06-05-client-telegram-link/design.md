## Context

Картотека клиентов (`clients`) хранит `contact_telegram` как username-строку. По username
Telegram-боту писать нельзя — нужен `telegram_chat_id`, который появляется только когда
пользователь сам инициирует диалог с ботом. Поэтому, чтобы в будущем слать клиенту
уведомления (следующий change `appointment-notify`), нужно сперва завести канал связи:
пригласить клиента в бота и захватить его `chat_id`.

Механика приглашения уже отработана для специалистов (`specialists`): deep-link
`t.me/<bot>?start=<token>`, обработка в `src/bot/handlers/start.py`
(`CommandStart` + fallback на вставленный текст/ссылку), сервис `consume_invite`.
Этот change повторяет паттерн для клиентов, но с тремя принципиальными отличиями
(см. Decisions).

## Goals / Non-Goals

**Goals:**

- Специалист генерирует приглашение клиента и получает пересылаемое сообщение со ссылкой.
- Клиент в один тап (без ввода команд) привязывает свой `chat_id` к карточке.
- Повторный переход по ссылке безопасен и перезаписывает `chat_id` (idempotent rebind).
- Один Telegram-аккаунт может быть одновременно специалистом и клиентом (тест без второй учётки).
- Карточка клиента отражает состояние привязки.

**Non-Goals:**

- Отправка уведомлений о записях — следующий change `appointment-notify`.
- Кнопки/интерактив в сообщениях клиенту (пока только текст).
- Входящий диалог с клиентом: клиент только получает сообщения, ничего не «ведёт» в боте.
- Изменение онбординга специалистов.

## Decisions

### 1. Префикс `cli_` для дисамбигуации токенов

Deep-link клиента — `t.me/<bot>?start=cli_<token>`, где `<token>` — 22-символьный
URL-safe токен (как у специалистов). Префикс `cli_` позволяет `/start`-хендлеру и
fallback-обработчику вставленного текста однозначно различить «токен специалиста» и
«токен клиента» без обращения к обеим таблицам.

- **Альтернатива А — пробовать обе таблицы по очереди.** Отвергнута: неявно, порядок
  важен, два запроса, хрупко при добавлении третьего типа токена.
- **Альтернатива Б — `encode=True`/`decode_payload` (base64url).** Отвергнута: непрозрачный
  payload, по нему труднее различать тип во fallback-обработчике вставленного текста.

Specialist-токены остаются без префикса — обратная совместимость существующих ссылок.

### 2. Идемпотентный rebind (в отличие от одноразового онбординга специалиста)

`link_telegram` **перезаписывает** `telegram_chat_id` и `linked_at` при каждом валидном
переходе. У специалиста онбординг одноразовый (`welcomed_at`-guard), у клиента — нет:
это нужно для тестов (специалист сперва привязывается сам, потом реальный клиент
перепривязывается) и для смены устройства/аккаунта клиентом.

### 3. `clients.telegram_chat_id` НЕ уникален (в отличие от `specialists`)

У специалистов `telegram_chat_id` уникален (один аккаунт = один оператор). У клиентов —
**нет** уникальности: при тестировании один аккаунт специалиста может быть привязан к
нескольким карточкам клиентов одновременно. Маршрутизация оператора идёт исключительно
через `specialists.find_by_chat_id`, а карточки клиентов по `chat_id` не ищутся — значит
коллизий нет. `invite_token` при этом уникален (нужен для поиска при онбординге).

### 4. Сосуществование ролей: специалист всегда «главнее»

Один `chat_id` может присутствовать и в `specialists`, и в `clients`. Бот никогда не
получает входящих от клиента (кроме одного тапа по `cli_`-deep-link), поэтому обычные
сообщения такого аккаунта всегда трактуются как действия оператора-специалиста.
Клиентский онбординг срабатывает только на явный `cli_`-префикс.

### 5. Лениво генерируемый `invite_token`

Токен создаётся в момент, когда специалист нажимает «Пригласить в бота», и сохраняется в
карточке (`set_invite_token`). Повторное нажатие переиспользует существующий токен (ссылка
стабильна). Карточка показывает «Пригласить в бота», если `telegram_chat_id IS NULL`, и
«✅ Привязан (переотправить приглашение)», если привязка уже есть.

### 6. Deep-link UX: один тап, без ввода команд

Клиент не печатает `/start`. Telegram при переходе по ссылке показывает системную кнопку
«Запустить» (для нового бота) либо сразу отправляет `/start cli_<token>` (если бот уже
открывался). Полностью «ноль тапов» Telegram не даёт намеренно. В пересылаемом сообщении
даём короткую подсказку клиенту нажать «Запустить». Сборку ссылки делаем
`aiogram.utils.deep_linking.create_start_link(bot, payload="cli_<token>")` — username бота
берётся из `bot`, не хардкодится.

## Risks / Trade-offs

- **Специалист тестирует под своим аккаунтом и «засоряет» chat_id нескольких карточек** →
  Допустимо by design (решение 3); это и есть тестовый сценарий. Реальный клиент позже
  перепривязывается (решение 2).
- **Утечка/расшаривание клиентской ссылки** → токен привязывает к конкретной карточке;
  rebind перезапишет `chat_id` на того, кто перешёл последним. Для кабинета логопеда риск
  низкий; при необходимости в будущем можно добавить одноразовость или протухание токена.
- **Fallback-обработчик вставленного текста** теперь должен отличать `cli_`-токен от
  specialist-токена → строгий разбор по префиксу; обычный текст по-прежнему игнорируется,
  чтобы бот не отвечал на каждое сообщение.

## Migration Plan

Одна Alembic-миграция, добавляющая в `clients`: `telegram_chat_id BIGINT NULL`,
`invite_token TEXT NULL` (UNIQUE-индекс, частичный/обычный), `linked_at TIMESTAMP NULL`.
Существующие строки получают NULL во всех трёх колонках — это валидное состояние
«клиент ещё не приглашён». Откат — обратная миграция (drop колонок). Данные не теряются.
