## Context

Канал «бот → клиент» только появляется (`appointment-notify`, ещё не начат). История
исходящих и значимых действий нигде не хранится: есть лишь Python-логи в stdout/JSON
(`logger.info("event.name", extra=...)`), недоступные специалисту. Архитектура — Clean
Architecture: `domain` (чистые сущности + протоколы репозиториев), `services` (зависят
только от `domain`), `infrastructure` (SQLAlchemy 2.0, async-сессии), `bot` (aiogram).
Отправка в Telegram по архитектуре проекта — побочный эффект в слое `bot/`; сервисы
чистые. Главное меню — reply-клавиатура в `build_main_keyboard`
(`src/bot/handlers/clients.py`); пагинация уже отработана в картотеке клиентов.

## Goals / Non-Goals

**Goals:**

- Одна таблица `audit_log`, хранящая две природы строк: `message` (отправка клиенту с
  полным текстом и статусом доставки) и `action` (ключевое действие специалиста).
- Сквозной слой записи: сервисы пишут `action` через инъектируемый `AuditRepo` (остаются
  чистыми), `bot/` пишет `message` после `send_message` с фактическим статусом.
- Просмотр из бота: кнопка `📜 Аудит` рядом с `⚙️ Настройки`, лента новейшее-сверху с
  пагинацией. Специалист видит только свои события.

**Non-Goals:**

- Не логируем каждое входящее сообщение/нажатие — только узкий список значимых событий.
- Нет фильтра по клиенту, поиска, экспорта (YAGNI; лента хронологическая).
- Нет ретеншна/прунинга — объём мал (один специалист), храним всё.
- Не реализуем здесь саму отправку уведомлений — это `appointment-notify`. Здесь только
  фундамент, в который он будет писать.
- Идемпотентность записи не требуется: каждое событие = отдельная строка.

## Decisions

### 1. Единая таблица `audit_log` с дискриминатором `kind`

Колонки: `id`, `specialist_id` (FK), `created_at` (UTC, `lambda: datetime.now(UTC)`),
`kind` (`message` | `action`), `event` (str, dot-free slug вида `notify_created`,
`appt_deleted`), `client_id` (nullable FK), `text` (nullable, полный текст для `message`),
`status` (nullable: `sent` | `failed`), `error` (nullable, причина сбоя). Индекс
`(specialist_id, created_at)` под ленту. `message`-строки несут `text`+`status`,
`action`-строки — только `event` (+ `client_id`, где есть).

- **Альтернатива — две таблицы (messages / actions).** Отвергнута: лента всё равно общая
  и хронологическая, два запроса с merge усложняют пагинацию; YAGNI.
- **Альтернатива — JSON-колонка payload.** Отвергнута: нужные поля известны и
  немногочисленны; явные колонки проще для индексации и чтения.

### 2. Запись `action` — через `AuditRepo`, инъектируемый в сервисы

Протокол `AuditRepo` живёт в `domain/audit.py`, реализация — в `infrastructure`. Сервисы
(`clients.py`, `appointments.py`) после успешной операции вызывают
`audit_repo.record(...)` — тем же сеансом/в той же транзакции, что и сама операция, чтобы
строка аудита и эффект были согласованы. Сервис остаётся чистым (зависит от протокола
домена, как от прочих репо).

- **Альтернатива — писать `action` в слое `bot/`.** Отвергнута: действие — это бизнес-
  событие; ему место рядом с операцией (как и существующим `logger.info`-событиям в
  сервисах). `bot/` пишет только `message`, т.к. факт доставки известен лишь после
  `send_message`.

### 3. Запись `message` — в слое `bot/` после `send_message`

Текст и статус доставки известны только в `bot/` (успех vs `TelegramForbiddenError` /
`TelegramBadRequest`). Хендлер отправки пишет `message`-строку (`text`, `status=sent`)
при успехе и (`status=failed`, `error`) при сбое. Сбой доставки не откатывает аудит-запись
— наоборот, факт «не доставлено» и есть ценная информация.

### 4. Кто пишет какие события (узкий список)

- `message`: `notify_created` / `notify_rescheduled` / `notify_cancelled` (из
  `appointment-notify`), `welcome` (привязка клиента), задел под `reminder`.
- `action`: `client_created` / `client_archived` / `client_restored`,
  `appt_created` / `appt_rescheduled` / `appt_deleted`.

Список фиксированный (enum). Не плодим события «отредактировал поле», «открыл меню» —
это шум и дубль Python-логов.

### 5. Просмотр: лента с пагинацией, переиспользуем паттерн картотеки

`AuditHandlers` (`src/bot/handlers/audit.py`): кнопка `📜 Аудит` (reply) открывает первую
страницу; inline-кнопки `◀ позже / раньше ▶` листают. Чтение —
`list_audit(repo, specialist_id, page)` с лимитом на страницу, сортировка
`created_at DESC`. Рендер строки зависит от `kind`: для `message` — иконка статуса
(`✅`/`⚠️`) + событие + имя клиента + текст; для `action` — иконка действия + событие +
имя клиента + детали. Имя клиента подтягиваем join'ом/догрузкой по `client_id`.

### 6. Связь с `appointment-notify` (порядок работ)

Аудит — фундамент. `appointment-notify` ещё не начат (0/22). После мерджа аудита его
хендлер отправки обязан писать `message`-строку, а из его Non-Goals убирается «Хранение
истории отправок». В `tasks.md` этого change фиксируем явную заметку о зависимости, но
правки самого `appointment-notify` — в его change, не здесь.

## Risks / Trade-offs

- **`action`-строки частично дублируют Python-логи и неявные данные таблиц.** →
  Митигация: узкий фиксированный enum событий; не логируем мелочь.
- **Хранение полного текста = персональные данные (имя ребёнка, время) во второй
  таблице.** → Принято осознанно: это данные самого специалиста, доступ только ему;
  ретеншна нет, но объём мал.
- **Рост таблицы со временем.** → Объём низкий (один специалист, десятки событий в день);
  индекс `(specialist_id, created_at)` держит ленту быстрой. Прунинг — позже, если
  понадобится (YAGNI).
- **Запись аудита в той же транзакции, что операция.** → Если запись аудита упадёт,
  упадёт и операция. Это приемлемо: аудит-строка для `action` дёшева; согласованность
  важнее. Для `message` запись идёт после факта доставки и операцию не трогает.
- **Рассинхрон, если новое место отправки забудет писать в аудит.** → Митигация: единая
  точка записи `message` в хелпере отправки, через который проходят все исходящие.

## Migration Plan

- Alembic-миграция `0007_audit_log` (0006 занят recurring): создать таблицу `audit_log` с
  колонками из решения 1 и индексом `(specialist_id, created_at)`. Откат — `drop table`.
- Миграция только создаёт структуру; бэкфилла прошлых событий нет (истории не было).
- Развёртывание — обычный `make run` (применяет миграции и стартует бота).
