# audit

## Purpose

Журнал значимых событий бота в БД и его просмотр специалистом из Telegram. Хранит две
природы строк: исходящие клиентам (с полным текстом и статусом доставки) и ключевые
действия специалиста. Журнал принадлежит специалисту — каждый видит только свои события.
Цель — уверенность («кому и что отправляли, дошло ли») и постфактум-разбор без доступа к
серверным логам.

## ADDED Requirements

### Requirement: Запись исходящих сообщений клиентам

Система **SHALL** при отправке клиенту сообщения от бота создавать строку журнала с
`kind = message`, фиксируя `specialist_id`, `client_id`, `event` (тип события), полный
`text` отправленного сообщения, `status` доставки и `created_at` в UTC.

При успешной доставке `status` **MUST** быть `sent`. При сбое доставки (клиент заблокировал
бота, чат недоступен) `status` **MUST** быть `failed`, а `error` **SHALL** содержать
причину сбоя; сбой доставки **MUST NOT** препятствовать созданию строки журнала — факт «не
доставлено» сам по себе ценен.

#### Scenario: Успешная отправка уведомления

- **WHEN** бот успешно отправляет клиенту уведомление о записи
- **THEN** в `audit_log` появляется строка с `kind = message`, `status = sent`, заполненными `client_id`, `event`, `text` и `created_at`
- **AND** `error IS NULL`

#### Scenario: Сбой доставки

- **WHEN** бот пытается отправить клиенту сообщение, но клиент заблокировал бота
- **THEN** в `audit_log` появляется строка с `kind = message`, `status = failed` и непустым `error`
- **AND** строка содержит полный `text`, который пытались отправить

### Requirement: Запись ключевых действий специалиста

Система **SHALL** при выполнении ключевого действия специалиста создавать строку журнала с
`kind = action`, фиксируя `specialist_id`, `event`, `created_at` в UTC и, где применимо,
`client_id`. Для `action`-строк поля `text`, `status`, `error` **MUST** быть пустыми.

Фиксируемые действия ограничены закрытым списком: создание, архивирование и восстановление
клиента; создание, перенос и удаление записи. Прочие действия (редактирование отдельных
полей, открытие меню, изменение настроек) журналироваться **MUST NOT**, чтобы не дублировать
серверные логи и не создавать шум.

#### Scenario: Создание клиента

- **WHEN** специалист создаёт нового клиента
- **THEN** в `audit_log` появляется строка с `kind = action`, `event` создания клиента, `client_id` нового клиента и `created_at`
- **AND** `text`, `status`, `error` пусты

#### Scenario: Удаление записи

- **WHEN** специалист удаляет запись клиента
- **THEN** в `audit_log` появляется строка с `kind = action`, `event` удаления записи и `client_id` затронутого клиента

#### Scenario: Нелогируемое действие

- **WHEN** специалист меняет настройку расписания (например, таймзону)
- **THEN** строка в `audit_log` не создаётся

### Requirement: Принадлежность журнала специалисту

Каждая строка журнала **SHALL** принадлежать одному специалисту (`specialist_id`).
Специалист **MUST** видеть только свои строки журнала; строки других специалистов
**MUST NOT** попадать в его ленту.

#### Scenario: Изоляция между специалистами

- **WHEN** специалист открывает журнал
- **THEN** в ленте присутствуют только строки с его `specialist_id`
- **AND** события других специалистов не показываются

### Requirement: Просмотр журнала лентой из бота

Бот **SHALL** предоставлять специалисту, прошедшему онбординг, кнопку `📜 Аудит` в главном
меню рядом с `⚙️ Настройки`. По нажатию бот **SHALL** показывать ленту событий специалиста,
отсортированную по `created_at` по убыванию (новейшие сверху), постранично.

Лента **SHALL** поддерживать перелистывание на более поздние и более ранние страницы.
Если событий нет, бот **SHALL** сообщить, что журнал пуст.

#### Scenario: Открытие журнала с событиями

- **WHEN** специалист с непустым журналом нажимает `📜 Аудит`
- **THEN** бот показывает первую страницу событий, новейшие сверху
- **AND** при наличии более ранних событий доступна кнопка перехода на следующую страницу

#### Scenario: Пустой журнал

- **WHEN** специалист без единого события открывает журнал
- **THEN** бот сообщает, что журнал пуст

#### Scenario: Перелистывание

- **WHEN** специалист на первой странице нажимает «раньше»
- **THEN** бот показывает следующую страницу более ранних событий

### Requirement: Различимое отображение строк журнала

Бот **SHALL** отображать строку журнала в зависимости от её природы. Для `kind = message`
строка **SHALL** показывать индикатор статуса доставки (доставлено / не доставлено), тип
события, имя клиента и текст сообщения. Для `kind = action` строка **SHALL** показывать тип
действия и, где есть, имя клиента, с временем события.

#### Scenario: Отображение доставленного сообщения

- **WHEN** в ленте присутствует `message`-строка со `status = sent`
- **THEN** строка показывает индикатор успешной доставки, имя клиента и текст сообщения

#### Scenario: Отображение недоставленного сообщения

- **WHEN** в ленте присутствует `message`-строка со `status = failed`
- **THEN** строка показывает индикатор сбоя доставки

#### Scenario: Отображение действия

- **WHEN** в ленте присутствует `action`-строка
- **THEN** строка показывает тип действия и имя затронутого клиента (если применимо), без текста и статуса доставки
