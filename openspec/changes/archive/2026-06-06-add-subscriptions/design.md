## Context

Проект — Clean Architecture (`domain` / `services` / `infrastructure` / `bot`). Абонементы
ложатся как новая capability рядом с `clients` и `appointments`: доменная сущность +
протокол репозитория, ORM-реализация, use-case-сервис, роутер aiogram. Карточка клиента уже
агрегирует данные из соседней capability (ближайшие встречи через `appointments`), так что
зависимость `clients`-хендлера от сервиса абонементов — это существующий паттерн, а не новый.

Дефолтное число встреч — настройка специалиста, по аналогии со `slot_minutes`: поле в
`Specialist`, `SettingField`, пункт меню в `settings.py`, server-default в миграции.

## Goals / Non-Goals

**Goals:**
- Учёт абонементов: создание, карточка, ручное списание, продление, закрытие.
- Не более одного активного абонемента на клиента; закрытые сохраняются в БД.
- Раздел «Абонементы» в главном меню: список активных всех клиентов + история закрытых.
- Минимальный диф, повторяющий существующие паттерны слоёв и тестов.

**Non-Goals:**
- Связь абонемента со встречами/расписанием (авто-списание при проведении встречи).
- Уведомления клиенту об остатке/исчерпании.
- Авто-закрытие абонемента при архивации клиента.

## Decisions

### Решение 1: Отдельная capability `subscriptions`, а не часть `clients`

Абонемент — самостоятельная сущность с собственным жизненным циклом (active → closed),
действиями и карточкой. Складывать в `clients` раздуло бы и без того крупный
`handlers/clients.py` (уже `# noqa: PLR0904`). Делаем `domain/subscription.py`,
`infrastructure/subscriptions_repo.py`, `services/subscriptions.py`,
`bot/handlers/subscriptions.py`. `clients.py` правим точечно — только кнопку на карточке.

_Альтернатива:_ встроить в clients — отвергнуто из-за размера и связности.

### Решение 2: Денормализуем `specialist_id` в таблице `subscriptions`

Абонемент логически принадлежит специалисту через клиента, но храним `specialist_id` прямо в
строке — как в `appointments`. Это даёт дешёвую проверку владельца при открытии карточки и
действиях без джойна к `clients`, и единообразие с соседней таблицей.

_Альтернатива:_ выводить владельца через `client_id → clients.specialist_id` — отвергнуто:
лишний джойн на каждое действие, расходится с приёмом в `appointments`.

### Решение 3: Инвариант «один активный» — на уровне запроса, индекс по `(client_id)` с фильтром

Активный абонемент ищем запросом `client_id = ? AND status = 'active'`. Создание сперва
проверяет отсутствие активного. Для скорости — индекс на `(client_id, status)`. Частичный
unique-индекс на SQLite поддерживается, но текущая БД — SQLite, а инвариант проще и надёжнее
держать в сервисе (создание идёт строго из одной точки UI). Жёсткий партиал-uniqueness
оставляем на будущее, если появится конкуренция записей.

_Альтернатива:_ partial unique index `WHERE status='active'` — избыточно для однопользова­
тельного по специалисту потока; держим инвариант в сервисе.

### Решение 4: `purchased` и `remaining` — оба кумулятивные счётчики

`purchased` — всего куплено за жизнь абонемента (растёт при продлении), `remaining` — текущий
остаток. Создание: `remaining = purchased = N`. Продление: оба `+= N`. Списание: `remaining
-= 1` с нижней границей 0. Это прямо отвечает на «сколько куплено / сколько осталось» на
карточке без отдельной таблицы операций.

_Альтернатива:_ журнал операций (таблица движений) — отвергнуто (YAGNI): истории движений
сейчас не показываем.

### Решение 5: Ввод числа встреч — FSM-шаг с дефолтом в подсказке

Создание и продление спрашивают число через FSM-состояние (как ввод `slot_minutes` в
`settings.py` и шаги мастера в `clients.py`). В тексте подсказки подставлен
`subscription_default`. Валидация числа — общий хелпер в `services/subscriptions.py`:
положительное целое с верхним порогом (по аналогии с `_MAX_SLOT_MINUTES`).

### Решение 6: Карточка абонемента редактируется in-place через `edit_text`

Все действия (списание, продление-после-ввода, закрытие/подтверждение) перерисовывают ту же
карточку через `edit_text`, как карточки клиента и записи. Кнопка возврата ведёт на карточку
клиента (`clients:card:<client_id>`).

### Решение 7: Раздел «Абонементы» в меню — списки активных и закрытых

В дополнение к входу с карточки клиента — reply-кнопка «🎫 Абонементы» во втором ряду главного
меню (рядом с «Настройки»). Открывает постраничный список активных абонементов всех клиентов
(имя ребёнка + остаток, сортировка по `created_at` убыв.), оттуда — история закрытых (сорт. по
`closed_at` убыв.). Пагинация — паттерн «`LIMIT page_size + 1`», как у клиентов. Имена
подтягиваются через `client_name_map` (как карточка клиента подтягивает соседнюю capability).
Карточка закрытого абонемента — read-only (только возврат), чтобы действия не применялись к
закрытому. Reply-кнопка регистрируется до FSM-хендлеров роутера — служит escape-hatch из
визарда, как «Клиенты».

_Альтернатива:_ оставить вход только с карточки клиента — отвергнуто по требованию: нужен общий
обзор и доступ к истории.

## Risks / Trade-offs

- **[Гонка двойного создания]** Два параллельных нажатия «создать» могли бы создать два
  активных абонемента. → Поток ввода идёт от одного специалиста последовательно (FSM), риск
  практически нулевой; при необходимости позже добавим partial unique index.
- **[Рассинхрон остатка с реальностью]** Ручное списание зависит от дисциплины специалиста —
  остаток может разойтись с фактом. → Осознанный Non-Goal этой итерации; авто-списание со
  встреч — отдельная будущая работа.
- **[Архивный клиент с активным абонементом]** Кнопка скрыта на архивной карточке, абонемент
  «подвисает». → Допустимо: при восстановлении клиента кнопка возвращается; авто-закрытие не
  делаем (YAGNI).
- **[Рост clients.py]** Добавление кнопки требует подгрузки активного абонемента в
  `_card_view`. → Один вызов сервиса `get_active`, по образцу `nearest_future_by_client`.

## Migration Plan

- Одна alembic-миграция: создаёт таблицу `subscriptions` (`id`, `client_id` FK, `specialist_id`
  FK, `purchased`, `remaining`, `status`, `created_at`, `closed_at`) с индексом
  `(client_id, status)`; добавляет колонку `subscription_default` в `specialists` со
  server-default `8`.
- Откат — `downgrade` удаляет таблицу и колонку. Данные абонементов при откате теряются
  (ожидаемо для новой фичи).
