## Context

Записи создаются/переносятся/удаляются в `src/bot/handlers/schedule.py`
(`ScheduleHandlers`). Привязка клиента к боту даёт `clients.telegram_chat_id` (может быть
NULL). Сервисы (`services/appointments.py`) — чистые, без aiogram. Отправка сообщения в
Telegram — побочный эффект, который по архитектуре проекта живёт в слое `bot/`.

Текущие результаты операций:
- create → `_finish_create` создаёт запись и показывает карточку (`_send_card`);
- reschedule → `_do_reschedule` переносит и показывает карточку;
- delete → `do_delete` удаляет и показывает сообщение «удалено» (`_deleted_keyboard`).

## Goals / Non-Goals

**Goals:**

- Уведомить клиента в один тап на результате create/reschedule/delete — только для
  привязанного клиента.
- Простой текст уведомления, корректное настенное время в tz специалиста.
- Сбой доставки не ломает операцию с записью.

**Non-Goals:**

- Кнопки/интерактив внутри сообщения клиенту (только текст).
- Автоматическая рассылка без действия специалиста, напоминания по расписанию.
- Уведомление с обычной карточки просмотра записи (там событие неоднозначно).
- Хранение истории отправок/идемпотентность отправки (повторный тап = повторная отправка).

## Decisions

### 1. Явный финальный шаг «Уведомить клиента?» с предпросмотром текста

На результате операции, **только если клиент привязан**, отдельным сообщением задаётся
вопрос «Уведомить клиента? Ему будет отправлено сообщение: «<текст>»» с кнопками
`[✅ Да, отправить] [Нет]`. Специалист видит точный текст до отправки. «Да» → отправить;
«Нет» → заменить вопрос на «Уведомление не отправлено» (кнопки убираются). Непривязанному
клиенту шаг не показывается (отправлять некуда).

- **Альтернатива — опциональная кнопка «Уведомить» на карточке записи.** Отвергнута по
  просьбе: специалист хочет явный последний шаг с предпросмотром текста, а не отдельную
  кнопку на карточке (её легко не заметить и неясно, что именно уйдёт клиенту).

### 2. Полезная нагрузка события — в callback_data, без перезагрузки записи

Кнопка несёт `sched:ntf:<event>:<client_id>:<stamp>`, где `event ∈ {c,r,x}` (created /
rescheduled / cancelled), `stamp` — время записи в UTC как `YYYYMMDDHHMM`. Так уведомление
об **отмене** работает даже после физического удаления записи (данные взяты до удаления),
а для create/reschedule не нужен лишний запрос. `chat_id` берётся по `client_id` в момент
отправки (свежее значение; владелец проверяется через `get_for_specialist`).

- **Альтернатива — `sched:ntf:<event>:<appt_id>` с перезагрузкой.** Отвергнута: для
  отмены записи уже нет (`appt_id` мёртв), пришлось бы городить отдельную ветку.

`callback_data` укладывается в лимит 64 байта: `sched:ntf:r:123456:202606101200` ≈ 32.

Для **серий** отдельный колбэк `sched:ntfs:<event>:<client_id>:<weekday>:<hhmm>` (`event ∈
{c,m,x}` = created / modified / cancelled). Текст описывает **правило** (день недели + время
настенное), а не конкретную дату, поэтому tz при отправке не нужен. Перенос/отмена **одной
даты** серии используют обычный одиночный `sched:ntf:` с конкретным `stamp`. Префикс
`sched:ntfs:` не пересекается с `sched:ntf:` (после `ntf` идёт `s`, а не `:`).

### 3. Отправка в слое `bot/`, обработка ошибок Telegram

Сервисы остаются чистыми. Хендлер `notify` вызывает `callback.bot.send_message(chat_id,
text)`. Сбой `TelegramForbiddenError` (клиент заблокировал бота) и `TelegramBadRequest`
(чат недоступен) ловятся → специалисту «не доставлено», запись/операция не трогается.
Успех → «отправлено» (тост). Лог: `appointment.notified` / `appointment.notify_failed`.

### 4. Гейтинг по привязке + повторная проверка при отправке

Кнопка показывается только при `client.telegram_chat_id IS NOT NULL`. На самой отправке
повторно загружаем клиента: если к моменту тапа привязки нет (редкий рейс) — «клиент не
привязан», без отправки.

### 5. Текст уведомления — настенное время в tz специалиста

`stamp` (UTC) → `utc_to_wall(dt, specialist.timezone)` → `format_ru_date` + `%H:%M`.
Три шаблона в `messages.toml`: записан / перенесено / отменено.

## Risks / Trade-offs

- **Повторный тап → повторная отправка.** Идемпотентности нет (YAGNI). Специалист видит
  «отправлено»; дубль маловероятен и не критичен.
- **Кнопка живёт в сообщении специалиста сколько угодно.** Тап позже отправит уведомление
  с тем временем, что зашито в `stamp` (для cancelled — временем уже удалённой записи). Это
  ожидаемо: stamp фиксирует момент операции.
- **Рейс отвязки клиента между показом кнопки и тапом** → закрыт повторной проверкой
  (решение 4).

## Migration Plan

Миграций нет — change использует `clients.telegram_chat_id` из `client-telegram-link`.
Только код + тексты. Откат — обратный revert кода.
