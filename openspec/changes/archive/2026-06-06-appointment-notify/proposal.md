## Why

Канал связи с клиентом готов (`client-telegram-link`: у привязанного клиента есть
`telegram_chat_id`). Теперь нужно им воспользоваться по прямому сценарию: при операциях с
записью специалист хочет одним действием уведомить клиента в Telegram («Вы записаны…»,
«Запись перенесена…», «Запись отменена…»), не переключаясь в ручную переписку.

## What Changes

- После операций с записью — **создание**, **перенос**, **отмена** — бот последним шагом
  задаёт явный вопрос «Уведомить клиента?» с **предпросмотром текста** уведомления и
  кнопками «Да» / «Нет». Шаг показывается **только** если у клиента заполнен
  `telegram_chat_id` (клиент привязан к боту); иначе не показывается. «Нет» → ничего не
  отправляется (вопрос заменяется на «не отправлено»).
- По нажатию «Да» бот отправляет клиенту простое **текстовое** сообщение, соответствующее
  событию (записан / перенесено / отменено), с датой и временем в таймзоне специалиста.
- Тот же шаг есть и для **регулярных записей** (серий): при создании/изменении/отмене всей
  серии текст описывает правило («каждый четверг в 14:00»), а при переносе/отмене одной даты
  серии — конкретную дату и время (как у разовой записи).
- Сбой доставки (клиент заблокировал бота и т.п.) **не откатывает** саму операцию с
  записью: специалисту показывается «не доставлено», запись остаётся как есть.
- Для отмены данные записи (`client_id`, время) фиксируются **до** удаления, чтобы текст
  уведомления можно было собрать после удаления.

## Capabilities

### New Capabilities

(нет — поведение расширяет существующую capability записей)

### Modified Capabilities

- `appointments`: добавляются требования об уведомлении клиента в Telegram при создании,
  переносе и отмене записи (доступно только для привязанного клиента; простой текст;
  устойчивость к сбою доставки).

## Impact

- **bot**: `src/bot/handlers/schedule.py` — шаг-вопрос «Уведомить клиента?» с
  предпросмотром и кнопками Да/Нет на результатах create/reschedule/delete, колбэк-хендлеры
  отправки и отказа, обработка ошибок Telegram
  (`TelegramForbiddenError`/`TelegramBadRequest`). Отправка идёт в слое `bot/`
  (`bot.send_message`), сервисы остаются чистыми.
- **Тексты**: новые ключи в `[schedule]` (`messages.toml`): подпись кнопки, тексты для
  клиента по трём событиям, статусы «отправлено / не доставлено / не привязан».
- **Логи**: события `appointment.notified` / `appointment.notify_failed`.
- **Документация**: `docs/bot.md`, `docs/features.md`, `openspec/specs/appointments/spec.md`.
- Не меняет схему БД (использует `clients.telegram_chat_id` из прошлого change) и не
  трогает онбординг/картотеку.
