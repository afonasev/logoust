## Context

Канал связи с привязанным клиентом готов (`clients.telegram_chat_id`). Срез записей на день
собирает `list_specialist_day(repo, specialist_id, day, tz, series)` — реальные записи плюс
**виртуальные** повторы активных серий. По решению
`2026-06-05_recurring_materialized_past_virtual_future.md` будущее серий **не
материализуется**: завтрашний повтор — это `Appointment` с `id=None`, без строки в БД.

Бот — long-polling без фонового планировщика. Соседний change `daily-digest` (ещё не
реализован) проектирует минутный asyncio-таск рядом с polling. Этой фиче нужен такой же
проход в заданное время, поэтому планировщик-петля общий.

Конвертация времени и форматирование — в `src/domain/schedule.py` (`utc_to_wall`,
`today_in_tz`, `format_ru_date`, `parse_hhmm`). Настройки специалиста живут на `specialists`
и редактируются в `src/bot/handlers/settings.py`. Отправка в Telegram — побочный эффект,
который по архитектуре проекта живёт в слое `bot/`; сервисы остаются без aiogram.

## Goals / Non-Goals

**Goals:**

- Раз в день (дефолт 12:00 в tz специалиста) автоматически напомнить привязанному клиенту о
  записи на завтра с двумя кнопками подтверждения.
- Хранить статус подтверждения так, чтобы он работал и для виртуальных повторов серий.
- Показать подтверждение специалисту (✅ в дневном экране и на карточке) и сразу уведомить об
  отказе с переходом на карточку.
- Время/вкл-выкл настраиваются; логика «кому пора» — чистая функция, тестируется без таймера.
- Падение фоновой задачи не роняет polling.

**Non-Goals:**

- Напоминание о записях, созданных **после** дневного прохода (только батч; ручной канал —
  `appointment-notify`).
- Автоматическая отмена/перенос при отказе — решает специалист вручную.
- Идемпотентность ответа клиента/запрет менять ответ (ответ перезаписывается всегда).
- Напоминание за N минут до встречи, многократные напоминания, эскалации.
- Материализация завтрашних повторов серий ради хранения статуса.

## Decisions

### 1. Журнал `appointment_reminders` вместо колонки на записи

Статус подтверждения нельзя положить колонкой `appointments.confirmed`: у виртуального повтора
серии **нет строки**. Заводим отдельную таблицу-журнал, где occurrence идентифицируется
натуральным ключом `(specialist_id, client_id, starts_at)`:

| колонка        | тип        | назначение                                            |
| -------------- | ---------- | ----------------------------------------------------- |
| `id`           | int PK     | для callback_data кнопок                              |
| `specialist_id`| int        | владелец (изоляция)                                   |
| `client_id`    | int        | получатель                                            |
| `starts_at`    | datetime   | UTC; вместе с client_id идентифицирует occurrence     |
| `series_id`    | int NULL   | NULL для разовой записи                                |
| `origin_date`  | date NULL  | NULL для разовой записи                                |
| `status`       | str        | `pending` / `confirmed` / `declined`                  |
| `sent_at`      | datetime   | момент отправки                                       |
| `responded_at` | datetime NULL | момент последнего ответа                           |

`UNIQUE(specialist_id, client_id, starts_at)` + `INSERT … ON CONFLICT DO NOTHING` дают
идемпотентность прохода: повторный тик и рестарт не плодят дубли. Дневной экран и карточка
читают статус `LEFT JOIN`'ом журнала по `(client_id, starts_at)` — работает одинаково для
разовых и виртуальных occurrence.

- **Альтернатива — колонка `confirmed` на `appointments`.** Отвергнута: нет строки у
  виртуального повтора серии.
- **Альтернатива — материализовать завтрашний день перед отправкой.** Отвергнута: ломает
  инвариант «материализованное прошлое + виртуальное будущее», воскрешает отвергнутый риск
  «горизонт молча закончится».

### 2. Общий планировщик-петля, две due-джобы

Минутный asyncio-таск рядом с polling в `src/__main__.py`. В него регистрируются due-джобы:
утренняя сводка (`daily-digest`, 10:00) и напоминание (12:00). Петлю физически вводит тот
change, что мёрджится первым; второй подключает свою джобу. Весь проход обёрнут в
`try/except Exception` с логом — исключение в джобе не роняет polling.

- **Альтернатива — отдельный loop на джобу.** Отвергнута: дублирует паттерн сна/тика; один
  процесс — один планировщик.
- **Альтернатива — APScheduler/внешний cron.** Отвергнута (как в `daily-digest`): новая
  зависимость/инфра ради per-specialist настенного времени и одного кабинета.

### 3. «Кому пора» — чистая функция, идемпотентность через журнал

`is_reminder_due(specialist, now)`: `reminder_enabled` И `utc_to_wall(now, tz) >=
reminder_time` И `reminder_last_run_on != today_in_tz(now, tz)`. `reminder_last_run_on`
служит дешёвым guard'ом, чтобы не сканировать завтрашний день каждую минуту после 12:00;
**настоящая** гарантия отсутствия дублей отправки — `UNIQUE` журнала. Проход: на первом
подходящем тике собрать записи на завтра, для каждого привязанного клиента вставить журнал и
отправить, затем `reminder_last_run_on = today` в любом случае (догон после простоя через
порог `>=`).

### 4. Записи на завтра, включая серии — общий путь дневного экрана

Срез завтрашнего дня собирается тем же `list_specialist_day(..., series=ctx)`, что и экран
расписания, → напоминания согласованы с тем, что видит специалист. Гейт по привязке:
шлём только тем occurrence, у кого `client.telegram_chat_id IS NOT NULL`.

### 5. Кнопки и ответ клиента

Сообщение клиенту: настенные дата/время записи (`utc_to_wall` + `format_ru_date` + `%H:%M`) +
inline-кнопки `[✅ Подтверждаю]` `[❌ Не смогу]`. `callback_data = appt:cfm:<reminder_id>:y|n`
(reminder_id — реальный PK, ≈ 20 байт, в лимите 64). Хендлер `appt:cfm:` грузит журнал,
проверяет владельца через `specialist_id`, перезаписывает `status` и `responded_at`.

Ответ можно менять всегда. На каждом переходе `* → declined` (а не на повторный тап того же
❌) специалисту шлётся сообщение об отказе с кнопкой `[→ Открыть запись]`. Переход в
`confirmed` специалисту не шлётся — он увидит ✅ в плане/на карточке.

### 6. Отображение статуса специалисту

- **Дневной экран расписания**: ✅ префикс у строки записи, для которой `status=confirmed`.
- **Карточка записи**: строка-пометка «подтверждена» при `confirmed`. (Для отказа отдельной
  визуальной пометки на карточке не вводим — специалист пришёл по кнопке из уведомления.)

Источник — журнал, ключ `(client_id, starts_at)`. Кнопка `[→ Открыть запись]` в уведомлении
об отказе ведёт на существующую карточку записи; для разовой записи — по `appointment_id`,
для виртуального повтора серии карточка открывается по `(series_id, origin_date)` тем же
путём, что и из дневного экрана.

### 7. Настройки специалиста

Новые колонки на `specialists`: `reminder_enabled` (bool, server-default true),
`reminder_time` (str, server-default `'12:00'`), `reminder_last_run_on` (date NULL). Дефолты
и в миграции (существующие — opt-out включён), и в домене. В меню настроек — тумблер вкл/выкл
и ввод времени через `parse_hhmm` (FSM по образцу `day_start`), симметрично `daily-digest`.

## Risks / Trade-offs

- **Долгий проход блокирует тик** → один кабинет, последовательный обход; при росте — батч.
  Сейчас YAGNI.
- **Сбой `send_message` (клиент заблокировал бота)** → ловим
  `TelegramForbiddenError`/`TelegramBadRequest`, лог `appointment.reminder_failed`; журнал
  всё равно создан (не зацикливаемся), статус остаётся `pending`. День помечается обработанным.
- **«Качели» ответа** (`✅→❌→✅`) → специалист получает уведомление на каждый переход в
  `declined`. Приемлемо: отказ — событие, требующее реакции; повторный тап того же ответа не
  шлёт.
- **Запись изменилась/удалена после отправки напоминания** → журнал ключуется по `starts_at`
  момента отправки. После переноса записи occurrence по новому времени не совпадёт со старым
  журналом → старое подтверждение «отвяжется» (запись покажется неподтверждённой). Приемлемо:
  перенос меняет договорённость, повторное подтверждение логично. Флагнуто как ограничение.
- **Падение фоновой задачи** → весь проход в `try/except` с логом; `asyncio.gather` не
  отменяет polling.
- **Тестируемость 100%** → due-логика и рендер — чистые функции/сервис; цикл со `sleep`
  тонкий, тестируется на один проход с замоканными зависимостями.

## Migration Plan

1. Alembic-миграция: таблица `appointment_reminders` (+ `UNIQUE(specialist_id, client_id,
   starts_at)`) и три колонки на `specialists` с server-default'ами.
2. Код: домен (сущность напоминания + поля `Specialist` + `is_reminder_due`), репозитории,
   сервисы (проход, обработка ответа, чтение статусов), фоновая джоба в общем loop, колбэк и
   отображение в боте, настройки, тексты.
3. Откат: обратный revert кода + downgrade миграции (drop таблицы и колонок). Данных,
   требующих сохранения, миграция не создаёт.
