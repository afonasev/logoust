## Context

Бот — aiogram, роутеры собираются в `build_dispatcher` (`src/bot/dispatcher.py`).
Необратимые действия с записью редактируют текущее сообщение в экран-результат с
клавиатурой `_deleted_keyboard` (`src/bot/handlers/schedule.py:505`) — одна кнопка «Назад»
(в текущем `appointment-notify` — иногда + «Уведомить»). Кнопка «Назад» несёт колбэк цели
(`clients:card:*`, `sched:day_view:*`, `recur:card:*`, `sched:chist:*`, `clients:active:*`,
`clients:arch:*`, `sched:feed`) и работает только потому, что её **жмут** — aiogram сам
диспатчит по префиксу.

Уже сейчас часть действий навигирует сама: перенос разовой записи
(`_do_reschedule`, `schedule.py:846-855`) шлёт отдельное сообщение-результат и затем
карточку; архив/восстановление клиента (`clients.py:776,786`) переоткрывает карточку.
Это и есть эталон желаемого поведения — его надо обобщить.

Зависимость: «Уведомить клиента» переписывается в **отдельный шаг Да/Нет** (см. proposal,
раздел Dependency). Этот change реализуется поверх — пост-действенный экран к моменту
авто-возврата не содержит осмысленных действий.

## Goals / Non-Goals

**Goals:**
- Единый хелпер пост-действия: ① отдельное сообщение-результат (остаётся в истории) → ②
  открыть исходное меню как самый свежий экран.
- Программный рендер кросс-роутерной цели `back` (например `clients:card` из роутера
  `schedule`).
- Свести к этому хелперу все необратимые действия: удаление разовой, перенос регулярной,
  стоп серии, пропуск даты, отмена флоу; а также уже-навигирующие (перенос разовой,
  архив/восстановление) — ради единообразия кода.

**Non-Goals:**
- Не меняем экраны-**подтверждения** (Да/Отмена) — у них есть выбор, это не тупик.
- Не трогаем поведение самих операций (что удаляется/переносится) — только навигацию
  после.
- Не реализуем сам notify-шаг Да/Нет (это change `appointment-notify`).

## Decisions

### 1. Две отдельные посылки вместо edit-in-place тупика

После действия: `message.answer(<результат>)` (новое сообщение, остаётся в истории) →
затем меню как **новое сообщение** внизу (самый свежий экран). Порядок в чате:
[результат] → [меню], что точно соответствует «пишем, что изменилось, и затем открываем
меню». Для колбэк-флоу исходную карточку с уже неактуальными кнопками гасим (edit в текст
результата без клавиатуры), чтобы не осталось «мёртвых» кнопок.

_Альтернатива (отклонена):_ редактировать карточку в меню + результат отдельным
сообщением снизу — тогда меню оказывается **выше** результата и не является самым свежим
сообщением; противоречит запрошенному порядку.

### 2. Навигация по префиксу `back` → (text, keyboard), доставка отдельным сообщением

Вводим маленький navigation-реестр: `back`-префикс → корутина, возвращающая
`(text, InlineKeyboardMarkup)`. Хелпер пост-действия шлёт результат через `message.answer`.
Реестр живёт в нейтральном модуле (`src/bot/navigation.py`), чтобы `schedule` мог
отрендерить цель из `clients` без циклического импорта. Хендлеры stateless
(`messages` + `session_factory`) — реестр строит нужный view через уже существующие
«чистые» билдеры или их тонкие обёртки.

_Альтернатива (отклонена):_ ре-диспатч синтетического `CallbackQuery`
(`model_copy(update={"data": back})`) через диспетчер. Минус: завязка на внутренности
aiogram, повторный прогон outer-middleware (`settle`, поиск специалиста), frozen-модель,
повторный `answer`, плохая тестируемость. DRY-выигрыш не оправдывает рисков.

### 3. Извлечь недостающие «чистые» билдеры

Готовы к переиспользованию: `_day_view` (`schedule.py:1024`), `_active_view`
(`clients.py:571`), вью карточки клиента (`clients.py:685`). Требуют извлечения в чистый
`(text, keyboard)`-билдер (сейчас только edit-рендер): `recur:card`, `sched:chist`,
`clients:arch`. Это основной объём работы. `sched:card` как цель возврата не нужен (после
необратимого действия записи уже нет / показываем меню-источник).

### 4. Fallback

Неизвестный/пустой `back` → сегодняшний день расписания (`sched:feed`) — сохраняем текущее
поведение `_deleted_keyboard` (ветка без `back`).

## Risks / Trade-offs

- **Рефактор многих рендереров под чистые билдеры** → ограничиваем строго префиксами, что
  реально встречаются как `back` после необратимых действий; остальные не трогаем.
- **Два сообщения вместо одного на действие** (рост истории чата) → принимаем осознанно:
  пользователь явно просил, чтобы факт изменения оставался в истории; перенос разовой уже
  так работает.
- **Циклические импорты `schedule` ↔ `clients`** → реестр в нейтральном
  `src/bot/navigation.py`, билдеры импортируются туда, а не крест-накрест между хендлерами.
- **Рассинхрон с notify-рерайтом** → реализуем строго поверх него; в tasks первой строкой
  фиксируем проверку, что notify уже стал шагом Да/Нет.
- **`recur:card` требует `series_id`+`origin_date`** из `back` → парсер префикса должен
  поддержать составной ключ; покрыть тестом.

## Migration Plan

Изменение только в слое `bot/`, без миграций БД. Откат — ревертом коммита; внешнего
состояния не создаёт. Выкатывается после слияния переписанного `appointment-notify`.

## Open Questions

- Полный ли список целей возврата: `sched:day_view`, `sched:feed`, `sched:chist`,
  `clients:card`, `clients:active`, `clients:arch`, `recur:card`? (выведен из кода —
  подтвердить при реализации).
- Для колбэк-флоу гасить исходную карточку через edit-в-текст-результата или просто снять
  клавиатуру (`edit_reply_markup(None)`) и слать результат отдельно — уточнить на ревью UX.
