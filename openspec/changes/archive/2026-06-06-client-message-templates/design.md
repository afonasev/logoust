## Context

Клиентские тексты вводятся тремя change'ами, и до их мёрджа большинства настраиваемых
шаблонов в коде ещё нет:

- `appointment-reminder` — «Завтра в {time} запись…» (+ кнопки подтверждения).
- `appointment-notify` — записан / перенесено / отменено.
- `subscription-payment-reminder` — просьба продлить абонемент.

Из замёрженного клиенту уходят только `invite_forward` и `linked` (архивный
`client-telegram-link`). Каждый notify-change рендерит свой текст через `messages.*.format(...)`
прямо в хендлере слоя `bot/`. Эта change вводит единую точку получения текста
(`resolve_template`) и переводит на неё все места отправки клиенту.

Настройки специалиста сейчас — плоские колонки на `specialists` + `update_settings(fields)`.
Шаблоны так не лягут: это набор переменной длины, растущий с каждой клиентской фичей, — поэтому
отдельная таблица, а не колонки.

## Goals / Non-Goals

**Goals:**

- Специалист переопределяет текст любого **клиентского** сообщения и сбрасывает к дефолту.
- Дефолты остаются в `messages.toml`; в БД — только переопределённое.
- Строгая защита инвариантов: обязательные обезличенные плейсхолдеры на месте, незнакомых нет.
- Имена детей/контактов в клиентские шаблоны не попадают (минимизация ПДн).
- Логика валидации и разрешения — чистые функции/каталог, тестируются без БД.

**Non-Goals:**

- Настройка текстов, которые видит сам специалист (меню/кнопки/подсказки) — YAGNI.
- Плейсхолдеры с именами (`{child}`, имя контакта) — запрещены намеренно.
- Богатый шаблонизатор (условия, циклы, форматтеры) — только плоская подстановка `{name}`.
- Превью с подставленными примерными значениями — можно добавить позже, не в этой change.
- Версионирование/история правок шаблонов, мультиязычность.

## Decisions

### 1. Отдельная таблица `message_templates`, дефолт — в `messages.toml`

`message_templates(id, specialist_id FK, template_key, body, UNIQUE(specialist_id,
template_key))`. Строка есть → это override; строки нет → рендерится дефолт из `messages.toml`.

- **Альтернатива — JSON-колонка `specialists.message_overrides`.** Отвергнута: нет FK-дисциплины,
  валидация целиком на аппе, неудобный точечный сброс одного ключа.
- **Цена**: при добавлении клиентского текста в будущем заводится новый ключ в каталоге
  `CLIENT_TEMPLATES` и дефолт в `messages.toml`; миграция данных не нужна (отсутствие строки =
  дефолт).

### 2. Каталог настраиваемых шаблонов — чистые данные в домене

`CLIENT_TEMPLATES: dict[str, TemplateSpec]`, где `TemplateSpec` = `{allowed: frozenset[str],
required: frozenset[str]}`. Только обезличенные плейсхолдеры:

| template_key         | allowed         | required        | дефолт-источник (messages.toml) |
| -------------------- | --------------- | --------------- | ------------------------------- |
| `appt_reminder`      | `{date} {time}` | `{date} {time}` | `appointment-reminder` ключ     |
| `notify_created`     | `{date} {time}` | `{date} {time}` | `[schedule]` notify-ключ        |
| `notify_rescheduled` | `{date} {time}` | `{date} {time}` | `[schedule]` notify-ключ        |
| `notify_cancelled`   | `{date} {time}` | `{date} {time}` | `[schedule]` notify-ключ        |
| `payment_reminder`   | `(нет)`         | `—`             | `subscription-payment-reminder` |
| `invite_forward`     | `{link}`        | `{link}`        | `[clients].invite_forward`      |
| `linked`             | `(нет)`         | `—`             | `[clients].linked`              |

Точный состав ключей и их привязка к ключам `messages.toml` финализируется на реализации, по
факту замёрженных текстов зависимостей. Каталог — единственный источник правды о том, что
вообще настраивается и какие плейсхолдеры допустимы.

### 3. Запрет имён — обоснование

Плейсхолдеры с именем ребёнка/контакта в whitelist клиентских шаблонов отсутствуют. Сообщение
уходит самому родителю на его `telegram_chat_id` (он и так знает имя) — поэтому ценность не в
сокрытии от читателя, а в:

- **радиусе при misdelivery**: `telegram_chat_id` может протухнуть/смениться; без имени утекает
  только факт записи, не имя ребёнка;
- **минимизации ПДн**: имя не размазывается по хранилищу шаблонов, превью и логам.

**Трейд-офф (зафиксирован):** для клиента с несколькими детьми в терапии текст без `{child}`
теряет однозначность («у вас запись завтра» — про кого?). Для одного кабинета Logoust это
редкий кейс; принят осознанно в пользу минимизации ПДн.

### 4. Валидация — чистая функция, строгий режим

`validate_template(key, body) -> list[str]` (домен, без БД):

1. Разобрать плейсхолдеры в `body` (имена в фигурных скобках; экранированные `{{`/`}}` — не
   плейсхолдеры).
2. Любой плейсхолдер вне `allowed[key]` → ошибка «недопустимый плейсхолдер `{x}`».
3. Любой из `required[key]`, отсутствующий в `body` → ошибка «нужен плейсхолдер `{y}`».
4. Пустой `body` → ошибка.

Пустой список ошибок = валиден. `save_template_override` зовёт валидатор и пишет только при
пустом списке; иначе возвращает ошибки в хендлер для показа специалисту. Так клиент не получит
битый текст: переопределение, ломающее инвариант, в БД не попадает.

### 5. Разрешение текста при рендере

`resolve_template(repo, messages, specialist_id, key) -> str`:

- `override = repo.get(specialist_id, key)`; вернуть `override.body` если есть, иначе дефолт из
  `messages` по соответствию `key → messages.toml`.
- Подстановка значений (`.format(date=…, time=…)`) — на стороне вызывающего, как сейчас.
  `resolve_template` отдаёт **шаблон**, не готовую строку, чтобы место отправки само
  подставляло свои данные.
- Защита рендера: при `.format()` пользовательский текст уже прошёл валидацию на сохранении, но
  на всякий случай подстановка идёт по известному набору kwargs; лишних плейсхолдеров в
  сохранённом тексте быть не может (валидатор это гарантирует).

### 6. UX раздела настроек

```
⚙️ Настройки → 📝 Тексты для клиента
   ├ Напоминание о записи        [✏️] [↩️]
   ├ Запись создана              [✏️] [↩️]
   ├ Запись перенесена           [✏️] [↩️]
   ├ Запись отменена             [✏️] [↩️]
   ├ Просьба продлить абонемент   [✏️] [↩️]
   ├ Приглашение в бота           [✏️] [↩️]
   └ Подтверждение привязки       [✏️] [↩️]

✏️ → показать текущий текст + список доступных плейсхолдеров с пояснением
     «Пришлите новый текст» (FSM) → validate → сохранить / показать ошибки
↩️ → удалить override (вернуться к дефолту)
```

- Callback'и: `tpl:edit:<key>`, `tpl:reset:<key>` (укладываются в 64 байта — ключи короткие).
- При правке показываем доступные плейсхолдеры и помечаем обязательные, чтобы специалист не
  гадал. Кнопка «↩️ Сброс» активна всегда; если override нет — она no-op с сообщением «уже
  дефолт».
- FSM по образцу ввода `day_start`/времени в существующих настройках.

## Risks / Trade-offs

- **Специалист сохранит грамматически кривой, но валидный текст** → за смысл текста отвечает
  специалист; мы гарантируем только наличие обязательных данных и отсутствие битых плейсхолдеров.
- **Запрет `{child}` ухудшает однозначность для мультидетных клиентов** → принято осознанно
  (см. решение 3); при появлении реальной потребности — отдельная change с продуманной моделью
  ПДн.
- **Рассинхрон каталога и `messages.toml`** (ключ в каталоге без дефолта или наоборот) → тест,
  проверяющий, что каждый `template_key` каталога имеет дефолт в `messages.toml`, и набор
  клиентских ключей не разъезжается.
- **Тестируемость 100%** → валидация и каталог — чистые функции; репозиторий — обычные CRUD-
  тесты; разрешение — сервис с замоканным repo.

## Migration Plan

1. Alembic-миграция: таблица `message_templates` (`specialist_id` FK → `specialists.id`,
   `template_key`, `body`, `UNIQUE(specialist_id, template_key)`). Данные не наполняются —
   отсутствие строки = дефолт.
2. Код: каталог `CLIENT_TEMPLATES` + `validate_template` (домен), протокол + реализация
   репозитория, сервисы `resolve_template` / `save_template_override` / `reset_template`,
   раздел настроек в боте, перевод мест отправки клиенту на `resolve_template`, тексты раздела.
3. Откат: revert кода + downgrade миграции (drop таблицы). Переопределения теряются, поведение
   возвращается к дефолтам из `messages.toml` — данных, требующих сохранения, нет.
