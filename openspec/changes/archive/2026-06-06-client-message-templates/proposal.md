## Why

Формулировки сообщений, которые бот шлёт клиенту (напоминание о записи, уведомления о
создании/переносе/отмене, просьба продлить абонемент, приглашающий текст), сейчас зашиты в
`messages.toml` и одинаковы для всех. Специалист хочет задавать **свои** формулировки —
тон, обращение, детали кабинета — не трогая код. Нужна возможность переопределить текст
каждого клиентского сообщения в настройках, сохранив при этом инварианты: обязательные данные
(дата/время/ссылка) не теряются, а персональные данные (имена детей и контактов) в шаблоны не
попадают.

## What Changes

- В `/settings` появляется раздел **«📝 Тексты для клиента»** со списком **клиентских**
  шаблонов. Специалист может **переопределить** текст любого из них и **сбросить к дефолту**.
  Тексты, которые видит сам специалист (меню, кнопки, подсказки FSM), **не** настраиваются.
- Дефолт каждого шаблона остаётся в `messages.toml` (источник правды «как из коробки»). В БД
  хранится **только переопределённый** текст; нет строки — рендерится дефолт. Новый клиентский
  текст в будущем = новый ключ в каталоге, без миграции данных.
- **Строгая валидация плейсхолдеров при сохранении** (whitelist + обязательные). У каждого
  шаблона свой набор: разрешён только известный плейсхолдер, обязательные (например `{date}`,
  `{time}`, `{link}`) должны присутствовать. Незнакомый плейсхолдер или пропуск обязательного →
  сохранение отклоняется с подсказкой. Это защищает клиента от битого текста и потери смысла.
- **Имена — под запретом.** Плейсхолдеры с именем ребёнка/контакта (`{child}` и т.п.) в
  whitelist клиентских шаблонов **не входят**. Разрешены только обезличенные `{date}`,
  `{time}`, `{link}`. Причина — минимизация ПДн и сокращение радиуса при ошибочной доставке
  (протухший `telegram_chat_id`): без имени утекает только факт записи, не имя ребёнка.
- Все места отправки клиенту перестают читать `messages.toml` напрямую и ходят через
  `resolve_template(specialist_id, key)` — override либо дефолт.

## Capabilities

### New Capabilities

- `client-message-templates`: переопределение специалистом текстов клиентских сообщений —
  каталог настраиваемых шаблонов с наборами плейсхолдеров (whitelist + обязательные, без имён),
  строгая валидация при сохранении, разрешение «override ?? дефолт» при рендере, раздел
  настроек с правкой и сбросом к дефолту.

### Modified Capabilities

- `appointment-reminder`, `appointment-notify`, `subscription-payment-reminder`: текст,
  отправляемый клиенту, берётся из переопределяемого шаблона специалиста, а не из статического
  `messages.toml`. Дельты к этим спекам оформляются **на этапе реализации** — после слияния
  соответствующих change'ей (см. depends-on), когда их живые спеки появятся в `openspec/specs/`.

## Impact

- **Зависимости (depends-on)**: реализуется **после** мёрджа `appointment-reminder`,
  `appointment-notify`, `subscription-payment-reminder` — именно они вводят клиентские тексты и
  точки их отправки, которые эта change переводит на `resolve_template`. Спека оформляется
  сейчас, реализация — после слияния зависимостей. До мёрджа из реально настраиваемого есть
  только `invite_forward` / `linked` (архивный `client-telegram-link`).
- **Схема БД**: новая таблица `message_templates` (`specialist_id`, `template_key`, `body`,
  `UNIQUE(specialist_id, template_key)`, FK на `specialists`) + alembic-миграция. Колонки на
  `specialists` не добавляются.
- **domain**: каталог `CLIENT_TEMPLATES` (`template_key → {allowed, required}`, чистые данные);
  чистая функция `validate_template(key, body) -> list[ошибок]`, проверяющая whitelist и
  обязательные через разбор плейсхолдеров; протокол репозитория шаблонов.
- **services**: `resolve_template(repo, messages, specialist_id, key) -> str` (override ??
  дефолт); `save_template_override(repo, specialist_id, key, body)` (валидирует перед записью);
  `reset_template(repo, specialist_id, key)` (удаляет override); логи
  `template.overridden` / `template.reset`.
- **bot**: новый раздел в `src/bot/handlers/settings.py` — список шаблонов, показ текущего
  текста + списка доступных плейсхолдеров, FSM-ввод нового текста, кнопка «↩️ Сброс»; места
  отправки клиенту в notify-хендлерах переводятся с `messages.*.format(...)` на
  `resolve_template(...)`. Новые тексты раздела настроек в `messages.toml`.
- **Документация**: `docs/database.md` (таблица + миграция), `docs/bot.md` (раздел настроек),
  `docs/features.md` (сценарий), `openspec/specs/client-message-templates/`, обновления спек
  `appointment-reminder` / `appointment-notify` / `subscription-payment-reminder` на этапе
  реализации, `README.md`.
