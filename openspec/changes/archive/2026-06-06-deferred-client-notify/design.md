## Context

Уведомление клиента о записи сейчас **stateless** и **немедленное**:
`src/bot/handlers/schedule.py` после операции показывает «Уведомить клиента?» с предпросмотром
(`_ask_notify` / `_ask_series_notify`), а по «Да» (`notify` / `notify_series`) разрешает
шаблон, рендерит текст и сразу шлёт `bot.send_message` (`_send_notify`). Всё нужное для
отправки (событие, `client_id`, время записи или правило серии) едет в `callback_data`
(`sched:ntf:…`, `sched:ntfs:…`); в БД ничего не сохраняется.

Минутный шедулер уже есть: `_scheduler_loop` в `src/__main__.py` раз в минуту вызывает
`run_reminder_pass` и `run_digest_pass` из `src/bot/scheduler.py` (каждый проход в
`try/except`). Паттерн «настенное время из настроек → проход по due» отработан на дайджесте
(`morning_notify_time`, `is_digest_due`). Конвертация времени и парсинг — в
`src/domain/schedule.py` (`utc_to_wall`, `today_in_tz`, `parse_hhmm`, `format_ru_*`). Карточка
клиента собирается в `ClientsHandlers._card_view` (`src/bot/handlers/clients.py`).

## Goals / Non-Goals

**Goals:**

- На «Да» предлагать момент отправки: Сейчас / пресет `deferred_notify_time` / своё время.
- «Сейчас» — без изменений к текущему немедленному поведению.
- Пресет и своё время → отложенная доставка, переживающая рестарт процесса.
- «Своё время» / пресет = ближайшее наступление `HH:MM` в tz (сегодня иначе завтра).
- Повторное откладывание по той же записи вытесняет прежнее, с явным сообщением; разные
  записи сосуществуют.
- Просмотр и отмена отложенных — на карточке клиента.
- Сбой отложенной доставки → уведомление специалиста + лог, без зацикливания.
- Настройка времени пресета (дефолт 20:00), 100% покрытие.

**Non-Goals:**

- Запись отправок в аудит-журнал — отдельный change `add-audit-log` (он уже специфицирует
  запись исходящих клиентам и охватит обе точки отправки). Здесь — обычные логи.
- Отдельный экран «исходящие» — управление живёт на карточке клиента.
- Произвольная дата (не только время), редактирование отложенного текста, повтор/ретраи
  доставки после сбоя, выбор момента для немедленных каналов (напоминания/дайджест) — YAGNI.
- Гарантированная доставка ровно в минуту (после простоя — догон позже).

## Decisions

### 1. Отложенная отправка требует персистентной очереди (новая таблица)

«Отправить в 20:00» должно пережить рестарт процесса (специалист правит ночью), поэтому в
in-memory `asyncio`-таске это держать нельзя — нужна строка в БД. Новая таблица
`scheduled_client_messages`:

| колонка         | тип              | назначение                                            |
| --------------- | ---------------- | ----------------------------------------------------- |
| `id`            | int PK           |                                                       |
| `specialist_id` | int FK           | владелец (изоляция + кому слать о сбое)               |
| `client_id`     | int FK           | получатель (карточка, вытеснение)                     |
| `chat_id`       | int              | снимок чата клиента на момент постановки              |
| `text`          | Text             | снимок одобренного текста (см. решение 2)             |
| `target_key`    | String           | устойчивый ключ цели для вытеснения (см. решение 3)   |
| `due_at`        | datetime (UTC)   | когда отправить                                       |
| `status`        | String/enum      | `queued` / `sent` / `failed` / `cancelled`           |
| `created_at`    | datetime (UTC)   |                                                       |
| `sent_at`       | datetime (UTC)?  | момент фактической отправки/попытки                  |

Индексы: `(status, due_at)` — для прохода доставки; `(specialist_id, client_id, status)` — для
блока на карточке; `(specialist_id, target_key, status)` — для вытеснения. Модель домена
(`ScheduledClientMessage`) + протокол репозитория в `domain`, ORM/импл в `infrastructure` —
как у `reminder`.

- **Альтернатива — `asyncio.sleep`/таск на сообщение.** Отвергнута: теряется при рестарте, а
  отсрочка — часы.
- **Альтернатива — переиспользовать `appointment_reminders`.** Отвергнута: у напоминаний своя
  семантика (подтверждение клиентом, натуральный ключ occurrence); это разовые произвольные
  исходящие.

### 2. Снимок текста, не пере-рендер при отправке

В очередь кладём **готовый текст** — тот же, что показан в предпросмотре. Это: (а) совпадает с
тем, что специалист одобрил; (б) не требует на момент отправки повторно разрешать шаблон и
поднимать состояние записи (которой при отмене уже нет). Проблема устаревания (запись
изменили после постановки) решается **вытеснением** (решение 3), а не пере-рендером: повторное
уведомление по той же записи заменяет прежний снимок свежим.

- **Альтернатива — хранить событие+ссылку и рендерить при отправке.** Отвергнута: сложнее,
  плюс крайние случаи (запись удалена/перенесена) и рассинхрон с одобренным предпросмотром.

### 3. Ключ вытеснения — устойчивый идентификатор записи, не время

«Та же запись» нельзя ключевать по времени: перенос меняет `starts_at`, и два переноса дали бы
два уведомления. Ключ по `client_id` — наоборот, слишком широкий (разные записи клиента должны
сосуществовать). Поэтому `target_key` — устойчивый идентификатор цели:

- разовая запись: `appt:<appointment_id>` (стабилен при переносе);
- серия целиком: `series:<series_id>`;
- одна дата серии: `series:<series_id>:<origin_date>`.

При постановке в очередь: если есть строка с тем же `(specialist_id, target_key)` в статусе
`queued` — перевести её в `cancelled` (вытеснение) и вставить новую; специалисту показать
«прежняя отправка на HH:MM заменена». Идентификатор цели **протягивается из обработчиков
операций** в шаг уведомления: для создания/переноса `appointment_id` доступен; для отмены
разовой — фиксируется **до** удаления (как уже делается для текста уведомления). Для серий
идентичность серии уже есть в colback'е (`series_id`, при одной дате — `origin_date`).

### 4. Тайминг: ближайшее наступление, `due_at` в UTC, «Сейчас» — без строки

Пресет и своё время трактуются одинаково чистой функцией: взять настенное `HH:MM` сегодня в
tz специалиста; если соответствующий UTC-момент `<= now` — прибавить сутки. Результат (UTC) —
`due_at`. «Сейчас» строку в очереди **не создаёт** — идёт прежним немедленным путём
(`_send_notify`), сохраняя текущую обратную связь в чате.

### 5. Доставка — третий проход в существующем минутном шедулере

В `src/bot/scheduler.py` — `run_outbox_pass(bot, session_factory, messages, now)`,
подключается в `_scheduler_loop` рядом с reminder/digest, в своём `try/except`. Проход
**запрашивает строки напрямую** (`status='queued' AND due_at<=now`, по возрастанию `due_at`) —
без обхода по специалистам (в отличие от дайджеста, где итерируем кандидатов). Для каждой:
повторно проверить привязку клиента (мог отвязаться), отправить `bot.send_message`, пометить
`sent`; при `TelegramForbiddenError`/`TelegramBadRequest` — `failed` + лог + сообщение
специалисту (решение 6). Догон после простоя — автоматически (любая просроченная `queued`
строка отправится на ближайшем тике); антидубль — переход статуса в `sent`/`failed`, повторный
тик их не подхватывает.

- **Альтернатива — отдельный таск/воркер.** Отвергнута: один процесс бота, минутной
  гранулярности достаточно (YAGNI).

### 6. Сбой отложенной доставки → сообщение специалисту

В отличие от немедленного пути (специалист видит «не доставлено» в чате), при отложенной
отправке его рядом нет. Поэтому при сбое — `failed` + лог `appointment.notify_failed` +
`bot.send_message` специалисту «не удалось доставить отложенное уведомление клиенту X».
Это сообщение само может не дойти (специалист заблокировал бота) — ловим и глотаем.

### 7. Управление — на карточке клиента, без отдельного экрана

`_card_view` дополняется блоком отложенных уведомлений клиента (выборка
`(specialist_id, client_id, status='queued')`) — строки с временем отправки в tz и кнопкой
`Отменить` (`clients:dnotify:cancel:<id>`). Отмена — owner-scoped (по `specialist_id`), статус
→ `cancelled`, перерисовать карточку. Пустой список — блок не показывается.

### 8. Транспорт выбора момента: callback для сейчас/пресет, FSM для своего времени

После «Да» бот рисует под-клавиатуру моментов. «Сейчас» и «пресет» — простые callback'и;
«Своё время» переводит в новый FSM-стейт (`Schedule.notify_custom_time`) и принимает текст
`ЧЧ:ММ` через `parse_hhmm`. Контекст уведомления (событие, `client_id`, occurrence/правило,
`target_key`, снимок текста) переносится с шага «Да»: для устойчивости к переполнению
`callback_data` (особенно с `target_key`) контекст складывается в **FSM-данные** при нажатии
«Да». Если к моменту выбора момента состояние утеряно (специалист начал другую операцию —
обработчики делают `state.clear()`), под-кнопки отвечают мягко («сессия устарела, повторите»),
ничего не отправляя.

- **Альтернатива — всё в `callback_data`.** Отвергнута: богатый случай (серия + дата +
  `target_key` + событие) рискует упереться в лимит 64 байта.

### 9. Настройка `deferred_notify_time` — по образцу `digest_time`, но проще

Только строка `HH:MM` (дефолт `20:00`); ни тумблера, ни `last_run` (откладывание доступно
всегда, антидубль не нужен — каждая строка явная). Поле в домене `Specialist` + константа,
колонка с server-default, пункт в меню настроек и ввод через существующий механизм
`SettingField`/`update_setting` с `parse_hhmm`.

### 10. Связь с `add-audit-log`

`add-audit-log` (in-flight) уже специфицирует запись исходящих клиентам (`kind=message`,
`status` sent/failed). Этот change добавляет **новую точку отправки** (проход доставки) помимо
немедленной. Запись в аудит обеих точек — ответственность `add-audit-log`; согласуем при
реализации (порядок мержа). Здесь — только `logger` и сообщение специалисту о сбое.

### 11. Миграция `0012`

Одна миграция: создать `scheduled_client_messages` + добавить колонку
`specialists.deferred_notify_time` (server-default `'20:00'`, существующие специалисты
получают дефолт). Downgrade — drop колонки и таблицы.

## Risks / Trade-offs

- **Потеря FSM-контекста между «Да» и выбором момента** (другая операция вызвала
  `state.clear()`) → мягкий фолбэк без отправки; специалист повторяет шаг. Покрывается тестом.
- **Устаревший снимок**, если запись изменили **без** нового шага уведомления → приемлемо:
  уведомления специалист инициирует явно; кейс «уведомил снова» закрыт вытеснением.
- **Смена tz между постановкой и отправкой** → `due_at` уже зафиксирован в UTC; приемлемо.
- **Сообщение специалисту о сбое тоже не доходит** → ловим `TelegramForbiddenError`/
  `TelegramBadRequest`, глотаем.
- **Много готовых строк за тик** → последовательная обработка; объёмы малы (один кабинет);
  при росте — батч/лимит, сейчас YAGNI.
- **Архивирование клиента с очередью** → клиенты не удаляются (только архив), `chat_id`
  остаётся; отложенные не трогаем особым образом (отправятся/видны на карточке). Приемлемо.
- **100% покрытие** → вся логика в чистых функциях (ближайшее наступление, фильтр due) и
  сервисах; проход и FSM — тонкие, тестируются на один прогон с замоканными зависимостями.

## Migration Plan

1. Alembic `0012`: таблица `scheduled_client_messages` + колонка
   `specialists.deferred_notify_time` (server-default `'20:00'`).
2. Код снизу вверх: домен (сущность + протокол + чистые `next_occurrence_utc`/`is_due`;
   поле и дефолт в `Specialist`) → репозиторий (enqueue с вытеснением, list due, mark
   sent/failed, list by client, cancel) → сервисы → проход в `scheduler.py` + подключение в
   `__main__` → хендлеры `schedule.py` (под-шаг момента + FSM) и `clients.py` (карточка) →
   настройки → тексты → доки.
3. Откат: revert кода + downgrade миграции (drop колонки и таблицы). Данных, требующих
   сохранения, миграция не создаёт.
