## Context

Навигация по дням расписания (`adjacent_shown_day` → `_nearest_appt_day`,
`src/services/appointments.py`) ищет ближайший «показываемый» день, комбинируя
чистый скан ближайшего рабочего дня (`nearest_working_day`) и один запрос к
**реальным** записям (`list_future_for_specialist` / `list_past_for_specialist`).
Виртуальные occurrence активных серий в этом расчёте не участвуют — функция не
принимает `SeriesContext`.

Все соседние пути уже series-aware: `list_specialist_day`, `list_specialist_week`,
`schedule_landing_day`, `series_taken_times`, `next_occurrence`, напоминания. Хендлер
`_day_view` (`src/bot/handlers/schedule.py`) уже загружает `SeriesContext` и
прокидывает его в `list_specialist_day` и `schedule_landing_day`, но в два вызова
`adjacent_shown_day` (prev/next) — нет. Поэтому день приземления виртуальной
occurrence на невыходной без реальных записей недостижим кнопками навигации.

Ограничения: модель «прошлое материализовано, будущее вычисляется» (recurring
design.md, decision 1) не меняем; перенос остаётся виртуальным исключением. Чистый
слой `domain/schedule.py` не должен зависеть от сервисов.

## Goals / Non-Goals

**Goals:**
- День приземления будущей occurrence активной серии (с учётом пропусков/переносов)
  считается показываемым при навигации в обе стороны.
- Симметрия с `schedule_landing_day`: одинаковое понятие «показываемого» дня во всех
  путях расписания.
- Минимальный диф; покрыть и перенос на невыходной, и обычную occurrence серии вне
  `working_days`.

**Non-Goals:**
- Не материализуем перенос в реальную строку (отвергнутый вариант B).
- Не меняем механизм переноса/пропуска и модель materialized/computed.
- Не трогаем недельный вид, напоминания, занятые слоты — они уже корректны.
- Не вводим бесконечный поиск: для навигации достаточно ближайшей occurrence в окне.

## Decisions

### Решение 1: `adjacent_shown_day` становится series-aware через `SeriesContext`

Добавляем `adjacent_shown_day` необязательный параметр `series: SeriesContext | None`
(по образцу `schedule_landing_day` и `list_specialist_day`). Кандидаты ближайшего дня
расширяются третьим источником — днём приземления ближайшей в нужном направлении
виртуальной occurrence любой активной серии; итог по-прежнему `min` (вперёд) / `max`
(назад) среди непустых кандидатов.

- **Альтернатива (отвергнута):** считать occurrence внутри `_nearest_appt_day`,
  смешав с запросом реальных строк. Хуже: размывает ответственность функции «реальные
  записи» и усложняет её сигнатуру.
- **Альтернатива (отвергнута):** материализовать перенос в реальную строку (вариант
  B). Чинит только перенос, оставляет occurrence-на-невыходной, идёт против decision 1
  (см. proposal).

### Решение 2: поиск ближайшего дня серии — переиспользуем `occurrences_landing_in`

Для направления «вперёд» ищем минимальную по дате occurrence строго после `day`;
«назад» — максимальную строго до `day` (только будущие, `≥ today`: прошлое уже в
реальных строках). Окно поиска вперёд ограничиваем тем же горизонтом, что и
`next_occurrence` (`_NEXT_OCCURRENCE_HORIZON`), чтобы бесконечная серия не давала
неограниченный скан. Назад occurrence серий не существует раньше `today` (всё
материализовано), поэтому обратный источник серий пуст — реальные строки покрывают
прошлое.

- **Альтернатива (отвергнута):** новая отдельная функция расчёта. `occurrences_landing_in`
  уже даёт нужную семантику приземления с учётом пропусков/переносов — дублировать
  логику дат серии нельзя.

### Решение 3: хендлер прокидывает уже загруженный `SeriesContext`

`_day_view` передаёт имеющийся `series` в оба вызова `adjacent_shown_day` — без
дополнительных запросов к БД (контекст уже загружен для `list_specialist_day`).

## Risks / Trade-offs

- [Скан горизонта вперёд для каждой серии при каждом переключении дня может быть
  затратным] → Окно ограничено горизонтом; серий на специалиста немного; расчёт —
  чистый, без БД. При необходимости — ранний выход на первой найденной occurrence.
- [Регрессия: пустые невыходные дни без серий должны по-прежнему пропускаться] →
  Покрывается тестом; источник серий пуст ⇒ поведение не меняется.
- [Несогласованность, если забыть прокинуть `series` в одном из двух вызовов] →
  Параметр необязательный с дефолтом `None`; оба вызова правятся в одном дифе, тест на
  next и prev.

## Migration Plan

Изменение чисто логическое: код + спека + тесты, без миграций БД и данных. Откат —
обычный revert коммита; на сохранённые данные не влияет.
