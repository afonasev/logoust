## Context

Регулярные записи сейчас реализованы как «серия» = одна строка `recurring_appointments`
с единственными `weekday` + `time_hhmm` + `start_date` + `materialized_through`.
Прошлые occurrence материализуются в реальные строки `appointments`
(`series_id`, `origin_date`, unique `(series_id, origin_date)`); будущие вычисляются на
лету (`occurrences_landing_in`) и мерджатся в чтение расписания и напоминания
(`SeriesContext`, `list_specialist_day`). Пропуск/перенос даты — строка
`recurring_exceptions` с `new_starts_at` (NULL = пропуск, значение = перенос).

Бизнес-домен логопеда требует несколько занятий в неделю с разным временем как одно
расписание клиента. Продакшена ещё нет — данные можно вайпнуть, миграция не обязана
сохранять старые регулярные строки.

## Goals / Non-Goals

**Goals:**
- Регулярное расписание клиента как мета-сущность с произвольным числом пар
  «день недели + время», включая повтор дня недели (два занятия в один день).
- Настройка расписания целиком: смена времени/дня слота, добавление/удаление слота,
  остановка всего расписания.
- Карточка расписания со списком ближайших occurrence на 14 дней; поштучные
  перенос/отмена/комментарий по конкретной occurrence.
- Комментарий уровня occurrence с наследованием из расписания, видимый везде, где виден
  комментарий встречи (день, история, напоминание).
- Переиспользовать существующую occurrence-математику (DST по каждой дате,
  материализация прошлого, исключения), а не переписывать её.

**Non-Goals:**
- Произвольные правила повтора (раз в 2 недели, месячные, по числам) — только
  еженедельные слоты.
- Разные комментарии на разные слоты (комментарий один на расписание + override на
  конкретную occurrence).
- Сохранение старых регулярных данных при миграции (прод пуст).
- Перенос/отмена occurrence за пределами 14-дневного окна из карточки расписания —
  такие даты достижимы тапом в дневном расписании.

## Decisions

### Decision 1: «Слот» = нынешняя «серия», расписание — тонкий родитель

Вся хитрая occurrence-математика завязана на тройку `weekday + time_hhmm +
start_date/materialized_through`. Это становится **слотом** (`RecurringSlot`). Сверху
добавляется **расписание** (`RecurringSchedule`), владеющее слотами и несущее
`specialist_id`, `client_id`, `comment`, `active`. Функции
`occurrences_landing_in`, `series_taken_times`, `settle`/`_settle_series`,
`_effective_starts_at` переносятся почти дословно с заменой `series`→`slot`.

**Почему:** минимизирует риск в самой сложной (и работающей) части; «несколько слотов» —
это аддитивная агрегация (`min`/конкатенация по слотам), а не новая математика.

*Альтернатива:* одна строка-серия со списком слотов внутри (JSON/массив) — ломает
per-date материализацию и unique-индекс, отвергнуто.

### Decision 2: Identity occurrence — по слоту, а не по расписанию

Расписание может иметь два слота в один день недели (Пн 12:00 и Пн 16:00) ⇒ `origin_date`
у двух occurrence совпадает. Поэтому материализованная строка `appointments` ссылается на
**слот**: `appointments.slot_id` (FK → `recurring_slots`), unique `(slot_id, origin_date)`.
Расписание про `appointments` ничего не знает.

**Почему:** сохраняет идемпотентность `settle` (insert-or-ignore по `(slot_id,
origin_date)`) без изменений; расписание-уровневый ключ был бы неуникальным.

### Decision 3: Исключение → общий «override записи»

Чтобы добавить независимый комментарий уровня occurrence, перегруженное поле
`new_starts_at` (NULL=пропуск, значение=перенос) разъединяется. `RecurringException`
становится `RecurringSlotOverride` с тремя независимыми осями:

| Поле | Смысл |
|---|---|
| `skipped: bool` | occurrence отменена (раньше — `new_starts_at IS NULL`) |
| `moved_to: datetime \| None` | перенос; `None` = время по сетке слота |
| `comment: str \| None` | `None` = наследует `schedule.comment` |

Логика occurrence: `skipped` → встречи нет; иначе время = `moved_to ?? wall_to_utc(сетка)`,
коммент = `comment ?? schedule.comment`. Любая комбинация (перенос+коммент, только коммент,
пропуск) — одна строка, key `(slot_id, original_date)`.

**Почему:** одна override-строка на «особую» дату; три ортогональных действия не путаются.

*Альтернатива:* отдельная таблица комментариев occurrence — лишняя сущность, отвергнуто.

### Decision 4: Эффективный комментарий

- Виртуальная (будущая) occurrence: `_occurrence` берёт `override.comment ??
  schedule.comment` (у слота своего комментария нет).
- Реальная (прошлая) строка: уже несёт собственный `comment` — не трогаем.
- `settle` пишет эффективный комментарий в `appointments.comment`, поэтому история и
  любые читатели видят его без спец-логики.
- Правка комментария прошлой occurrence = правка строки `appointments` (как у разовой).

### Decision 5: Двухуровневая карточка, окно 14 дней (скользящее)

- **Экран 1 «Расписание»**: клиент, общий комментарий, правило (список слотов), затем
  ближайшие occurrence в окне **скользящих 14 дней** `[today, today+14)` отдельными
  кнопками (по всем активным слотам, отсортированы по времени, с учётом
  переносов/пропусков через `occurrences_landing_in`). Внизу «Настроить» / «Отменить всё».
  Пустое окно → строка «На ближайшие 2 недели встреч нет».
- **Экран 2 «Одна встреча»**: конкретная `(slot, origin_date)`; «Перенести» / «Отменить» /
  «Комментарий»; «К расписанию».
- Тап на регулярную occurrence в **дневном расписании** открывает Экран 2 напрямую
  (так достижимы и даты за пределами 14 дней).

**Почему:** явный список убирает любую логику «угадать ближайшую»; действия всегда бьют по
конкретной `(slot, origin_date)`. Скользящие 14 дней (а не календарные недели) — по выбору
разработчика: всегда ровно две недели вперёд, предсказуемая высота карточки.

### Decision 6: «Настроить» — последовательно по слотам

«Настроить» → список слотов; по каждому `сменить время / сменить день / удалить`, плюс
«добавить день». Смена дня слота — как нынешний `edit_series`: пересчёт `start_date` на
ближайший новый день ≥ сегодня, `materialized_through`→сегодня (прошлое прежней сетки
остаётся фактом). Смена времени сетку не двигает. Удаление слота = `slot.active=false`
(история слота цела). Удаление **последнего** активного слота гасит расписание
(`schedule.active=false`).

### Decision 7: Честные имена + чистая миграция

Домен и БД получают честные имена: `recurring_schedules`, `recurring_slots`,
`recurring_slot_overrides`, `appointments.slot_id`. Миграция дропает старые
`recurring_appointments`/`recurring_exceptions`, пересоздаёт схему, перепривязывает
`appointments` (старый `series_id`→`slot_id`). Регулярные данные не переносятся (прод пуст);
разовые записи (`series_id IS NULL`) сохраняются.

## Risks / Trade-offs

- **[Потеря данных при миграции]** Старые регулярные расписания и их материализованные
  строки вайпаются → *Mitigation:* подтверждено отсутствие прода; разовые записи
  (`series_id IS NULL`) сохраняются; миграция чистит только регулярную часть.
- **[Сложность чтения с N слотов]** `SeriesContext`/`load_series_context` и
  `list_specialist_day` теперь агрегируют по слотам нескольких расписаний → *Mitigation:*
  структура `SeriesContext` хранит слоты со ссылкой на их расписание (для comment/active),
  чтение остаётся одним проходом; математика per-slot не меняется.
- **[Два слота в один день — занятость]** `series_taken_times`/`taken_slot_times` должны
  суммировать слоты, попадающие на один день → *Mitigation:* уже работает через
  объединение occurrence по дню; добавляется только перебор нескольких слотов.
- **[Рост числа callback-префиксов]** Перенос карточки на schedule/slot/occurrence ломает
  старые `recur:*` → *Mitigation:* прод пуст, обратная совместимость callback не нужна;
  префиксы переразложить целостно (`recur:sched:*`, `recur:slot:*`, `recur:occ:*`).
- **[Комментарий override остаётся после материализации]** override-строка с `comment`
  на прошедшую дату становится избыточной → *Mitigation:* безвредна (фильтр `< today`
  в чтении), очистку не делаем (YAGNI).

## Migration Plan

1. Alembic-ревизия: `drop_table('recurring_exceptions')`, `drop_table(
   'recurring_appointments')`; для `appointments` — удалить строки с `series_id IS NOT
   NULL` (материализованные регулярные), затем заменить колонку/FK `series_id` → `slot_id`
   и индекс `uq_appointments_series_origin` → `uq_appointments_slot_origin`.
2. Создать `recurring_schedules`, `recurring_slots` (FK → schedules),
   `recurring_slot_overrides` (FK → slots, unique `(slot_id, original_date)`), индексы под
   чтение (`recurring_slots` по `(schedule_id, active)`, schedules по
   `(specialist_id, active)`).
3. Downgrade: обратный дроп новых таблиц и восстановление прежней схемы (без данных).
