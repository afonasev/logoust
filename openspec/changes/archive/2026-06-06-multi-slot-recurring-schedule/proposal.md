## Why

Сейчас регулярная запись = ровно один день недели + одно время. Реальная регулярка
клиента у логопеда — это часто несколько занятий в неделю с разным временем
(например «каждый Пн в 12:00 и каждый Вт в 14:00»). Сегодня это приходится заводить
как несколько независимых «серий», что не отражает домен (у специалиста в голове одно
расписание клиента) и не даёт настроить/остановить его как единое целое.

## What Changes

- **BREAKING**: «серия» (`recurring_appointments`) перестаёт быть единицей с одним
  днём/временем. Вводится мета-сущность **регулярное расписание клиента**
  (`recurring_schedules`), владеющая произвольным числом **слотов**
  (`recurring_slots`) — пар «день недели + время». Слоты могут повторять день недели
  (например два занятия в один понедельник).
- Создание регулярного расписания через визард: цикл «выбрал день → выбрал время →
  добавить ещё?» с одним общим комментарием на расписание.
- Карточка расписания показывает правило (список слотов) и **ближайшие occurrence на
  14 дней вперёд** отдельными кнопками; тап на встречу открывает карточку одной
  встречи с действиями «Перенести / Отменить / Комментарий».
- Перенос/отмена/комментарий действуют строго на конкретную occurrence
  `(слот, дата)` — никакой логики «угадать ближайшую».
- «Настроить» — последовательная настройка расписания: по каждому слоту
  `сменить время / сменить день / удалить`, плюс «добавить день». Удаление последнего
  слота гасит расписание.
- **Комментарий уровня occurrence**: по умолчанию наследуется из расписания, но может
  быть переопределён для конкретной встречи и виден везде, где виден комментарий
  встречи (день, история, напоминание).
- Исключение даты обобщается из «пропуск/перенос» в **override записи**: независимые
  оси `skipped` / `moved_to` / `comment` (раньше пропуск и перенос кодировались одним
  полем `new_starts_at`).
- **BREAKING**: схема БД меняется (новые таблицы, `appointments.slot_id` вместо
  `series_id`); продакшена нет — старые регулярные данные вайпаются миграцией.

### Modified Capabilities
- `recurring-appointments`: единица регулярности становится мета-расписанием с
  несколькими слотами; меняются требования к созданию, карточке (список ближайших на
  14 дней), настройке (по слотам: смена времени/дня, добавление/удаление), переносу и
  пропуску (по конкретной occurrence), добавляется требование к комментарию уровня
  occurrence с наследованием из расписания.

## Impact

- **domain**: `src/domain/recurring.py` — новые `RecurringSchedule`, `RecurringSlot`,
  `RecurringSlotOverride` (бывш. `RecurringException`); протоколы репозиториев.
- **services**: `src/services/recurring.py` — `create_schedule`/`add_slot`/`edit_slot`/
  `remove_slot`/`stop_schedule`, override-операции (skip/move/comment), рефактор
  occurrence-математики `slot`-центрично, эффективный комментарий.
- **infrastructure**: новая миграция (дроп старых recurring-таблиц, создание
  `recurring_schedules`/`recurring_slots`/`recurring_slot_overrides`,
  `appointments.slot_id` + unique `(slot_id, origin_date)`); репозитории.
- **bot**: `src/bot/handlers/schedule.py` — визард создания, двухуровневые карточки
  (расписание / одна встреча), режим «Настроить», перенос/отмена/комментарий;
  пересборка callback-префиксов `recur:*`. `src/bot/messages.toml` секция `[recurring]`.
- **reads/напоминания**: `src/services/appointments.py`, `src/services/reminder.py` —
  чтение occurrence и эффективного комментария через обновлённый `SeriesContext`.
- **docs/specs**: `openspec/specs/recurring-appointments/spec.md`, `docs/database.md`,
  `docs/bot.md`, `docs/features.md`, `README.md`.
