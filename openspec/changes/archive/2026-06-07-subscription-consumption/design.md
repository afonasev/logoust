## Context

Абонемент (`subscriptions`) сегодня хранит только счётчики `purchased`/`remaining` и
меняется единственным ручным действием «вычесть встречу». Истории списаний нет — нельзя
сказать, за какую встречу и когда ушёл остаток. Параллельно уже есть инфраструктура
фоновых дневных проходов (утренняя сводка `digest`, напоминания `reminder`, payment-reminder),
каждый из которых: берёт кандидатов-специалистов, гейтится «пора ли» через настройку
времени + дневной флаг `*_last_run_on`, переживает рестарт и догоняется после простоя,
и держит aiogram вне сервисного слоя (доставка — в `bot/scheduler.py`).

Эта фича добавляет ещё один такой проход — **вечерний авто-учёт расхода** — и новую
сущность-журнал. Главная сложность не в проходе, а в **идемпотентности списания**: несколько
вызовов (повторный минутный тик, рестарт в момент прохода, ручная кнопка «Отправить сейчас»
поверх боевого прохода) не должны списать за одну встречу дважды. В одном asyncio-loop
корутины переключаются на `await`-точках, поэтому проверка «строки ещё нет» в Python —
гоночная (TOCTOU); замок обязан быть на уровне БД.

## Goals / Non-Goals

**Goals:**
- Остаток абонемента уменьшается сам по факту прошедших сегодня встреч; специалист правит
  только исключения.
- Неизменяемая по фактам история списаний с раздельными комментариями (к записи / после встречи).
- Гарантия «ровно одно списание на встречу» при любых повторных/параллельных проходах.
- Отчёт специалисту: что списано (кнопки на абонементы) и что прошло мимо (❗).

**Non-Goals:**
- Показ закрывающего комментария клиенту — отдельная будущая фича (поэтому сейчас не трогаем
  `client_audit`/`AuditEvent`).
- Догон вчерашних встреч после суточного простоя бота (берём только сегодняшние — см. решение 2).
- Учёт фактической явки/подтверждения: списываем по умолчанию, отказ от списания — ручная отмена.
- Перенос ретро-списаний за встречи до внедрения фичи (только сегодняшние → ретро не затрагивается).

## Decisions

### Решение 1. Идемпотентность списания — замок на уровне БД, не проверка в коде

`subscription_deductions` получает **уникальный частичный индекс** на `appointment_id`
(`WHERE appointment_id IS NOT NULL` — ручные списания с `NULL` не конфликтуют между собой).
Списание одной встречи выполняется **в одной транзакции, вставка-замок раньше декремента**:

```
BEGIN
  INSERT deduction(appointment_id = X)        -- замок; IntegrityError → ROLLBACK, пропустить
  UPDATE subscriptions
     SET remaining = remaining - 1
     WHERE id = :sub AND remaining > 0          -- атомарно, без read-modify-write
     -- 0 строк (остаток уже 0) → ROLLBACK, встреча идёт в отчёт как ❗ «закончился»
COMMIT
```

- Замок-вставка делает списание идемпотентным по встрече: повторный тик, рестарт, «Отправить
  сейчас» поверх боевого — все упираются в уникальный индекс и пропускают встречу.
- Отменённая строка **остаётся** в БД (`cancelled_at` проставлен) и продолжает держать замок —
  отменённая встреча повторно не спишется.
- `WHERE remaining > 0` вместо `remaining = прочитанное − 1` убирает потерю обновления, когда у
  клиента **две встречи за день** по одному абонементу.

_Альтернатива (отвергнута):_ проверка `if not exists(...)` в сервисе — гоночная на `await`-точках,
не даёт гарантии при «Отправить сейчас» поверх прохода.

### Решение 2. Окно прохода — только сегодняшние встречи

Проход берёт встречи с `starts_at ≤ now` за **сегодняшний** день в таймзоне специалиста.

- Защищает первый деплой: иначе сканирование «всех прошлых встреч без списания» при первом
  запуске списало бы каскадом всю историю каждого клиента с активным абонементом.
- Плата: если бот лежал весь день и пропустил 20:00, вчерашние встречи не догонятся (YAGNI —
  специалист спишет вручную; остаток виден на карточке).

_Альтернатива (отвергнута):_ «все прошлые встречи без строки журнала» — мощнее для догона, но
опасно на первом запуске и требует нижней границы по дате внедрения.

### Решение 3. Материализация повтора серии должна быть идемпотентной

Списание ссылается на `appointment_id`, поэтому виртуальный повтор сначала материализуется
(получает реальный id). Если две гонки создадут две строки встречи на одну `(slot_id,
origin_date)` — будет два `appointment_id` и замок по `appointment_id` их не свяжет.
Поэтому материализация выполняется через `INSERT … ON CONFLICT DO NOTHING` поверх
**уникального индекса `appointments(slot_id, origin_date)`** с повторным чтением. Наличие этого
индекса проверяется; если его нет — добавляется этой же миграцией.

### Решение 4. Отмена списания — мягкая (`cancelled_at`), не удаление

Отмена проставляет `cancelled_at` и возвращает `remaining + 1` (в той же транзакции). Строка
остаётся: скрыта из истории, но держит замок идемпотентности — отменённая встреча не спишется
снова следующим проходом. Жёсткое удаление освободило бы замок и встреча списалась бы повторно.
Возврат остатка делаем атомарным `UPDATE … SET remaining = remaining + 1`. Отмена доступна
только на **активном** абонементе; на закрытом журнал read-only.

### Решение 5. Снимок данных встречи в строке журнала

В строке журнала храним снимки `appointment_starts_at` и `appointment_comment` на момент
списания, а `appointment_id` — с `ON DELETE SET NULL`. Это делает историю самодостаточной и
честно неизменяемой: правка комментария записи или жёсткое удаление встречи не меняют прошлые
строки журнала. Редактируемым остаётся только `closing_comment` (заметка «после встречи») —
это не нарушает «историю фактов нельзя менять».

### Решение 6. Новый проход по образцу существующих, отчёт — в bot-слое

Сервис `services/consumption.py::run_consumption_if_due(specialist, now, …, report)` повторяет
контракт payment-reminder: гейт `is_consumption_due` (настройка времени + дневной флаг
`consumption_last_run_on`), сбор сегодняшних прошедших встреч (с учётом серий через
`load_series_context`/`list_specialist_day`), списание каждой по решению 1, накопление результата
(списанные абонементы + ❗-встречи) и вызов инъектированного `report(...)`. Сборка текста,
кнопок и `bot.send_message` — в `bot/scheduler.py`. Отчёт уходит **только если есть что показать**
(≥1 списание или ≥1 ❗); пустой вечер — молчит, но день помечается обработанным.

«Отправить сейчас» из настроек вызывает тот же сбор/списание **мимо** гейта и **не** трогает
дневной флаг (реальный проход в 20:00 потом отработает, но из-за замка ничего не задвоит и
останется молчаливым). Это реальное списание, не dry-run.

### Решение 7. Ручное «вычесть» становится записью в журнал

`decrement_meeting` переводится на атомарный декремент и в той же транзакции пишет строку
журнала с `appointment_id = NULL` (ручное списание; снимки даты/комментария записи пустые).
Комментарий добавляется потом на экране списания — единообразно с авто-списаниями.

## Risks / Trade-offs

- **Каскадное списание на первом деплое** → ограничиваемся сегодняшними встречами (решение 2);
  ретро не затрагивается.
- **Двойное списание при гонке проходов** → уникальный индекс + транзакция вставка-замок →
  декремент (решение 1); отдельный сценарий в спеке на «два прохода — одно списание».
- **Двойная материализация повтора → два списания** → уникальный индекс `(slot_id, origin_date)`
  + `ON CONFLICT DO NOTHING` (решение 3); риск только если индекса нет — проверяется миграцией.
- **Суточный простой бота → пропуск вечернего прохода** → встречи дня не спишутся
  автоматически; принимаем (YAGNI), специалист видит остаток и спишет вручную.
- **Изменение поведения ручной кнопки** (теперь оставляет след в журнале) → BREAKING для спеки
  `subscriptions` и тестов; правится в этом же change.

## Migration Plan

1. Alembic-миграция: таблица `subscription_deductions` + уникальный частичный индекс на
   `appointment_id`; колонки `specialists.consumption_enabled` (дефолт true),
   `consumption_time` (дефолт `20:00`), `consumption_last_run_on` (nullable); уникальный индекс
   `appointments(slot_id, origin_date)`, если его ещё нет.
2. Существующим специалистам авто-списание проставляется включённым с дефолтным временем.
3. Откат: down-миграция удаляет таблицу/колонки/индексы; журнал списаний теряется (история —
   только в этой таблице, на счётчики `remaining` откат не влияет).
