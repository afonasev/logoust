## Context

Три предшествующих change задают всё, на чём стоит эта фича:

- `add-subscriptions` — сущность `Subscription` (один активный на клиента; `remaining`
  меняется только вручную: «вычесть» блокируется на 0 и оставляет абонемент `active`;
  «продлить» увеличивает `purchased` и `remaining`).
- `daily-digest` — первый фоновый asyncio-таск рядом с long-polling: минутный тик, чистая
  функция `is_digest_due`, антидубль через `morning_notify_last_run_on`, отправка из слоя
  `bot/`, весь проход в `try/except` (исключение не роняет polling).
- `appointment-notify` — отправка текста клиенту по `clients.telegram_chat_id`, обработка
  `TelegramForbiddenError`/`TelegramBadRequest`, кнопка-опция вместо обязательного диалога.

Сборка записей на день — `list_specialist_day(repo, specialist_id, day, tz, series)` (реальные
записи + виртуальные повторы серий). Конвертация времени и форматирование — в
`src/domain/schedule.py` (`utc_to_wall`, `today_in_tz`, `format_ru_date`, `parse_hhmm`).

## Goals / Non-Goals

**Goals:**

- За сутки до записи (прогон в настенное время, дефолт 12:00) напомнить специалисту о клиенте,
  у которого активный абонемент с остатком 0 и запись завтра.
- По одному тапу отправить привязанному клиенту фиксированный текст с просьбой продлить.
- Один алерт на клиента в день; один алерт на «пустой цикл» абонемента (пока остаток 0).
- Устойчивость к минутному тику и рестарту; падение прогона не роняет polling.
- Вся логика «кому/когда/нужно ли» — чистые функции, тестируются без таймера.

**Non-Goals:**

- Редактирование текста напоминания специалистом (фиксированный шаблон; YAGNI).
- Автоматическая отправка клиенту без действия специалиста.
- Списание/продление абонемента из самого напоминания (только просьба продлить текстом).
- Напоминания за произвольное число дней / повторные «добивающие» рассылки.
- Учёт оплаты, ссылки на оплату, эквайринг.

## Decisions

### 1. Общий фоновый цикл вместо отдельного таска на фичу

`daily-digest` вводит `run_digest_loop`. Здесь он обобщается в один цикл-«планировщик»: на
каждом минутном тике последовательно прогоняются зарегистрированные джобы — утренняя сводка и
напоминание об оплате. Каждая джоба сама решает, кому она сегодня due. Один тик, один проход по
специалистам, несколько проверок.

- **Альтернатива — второй независимый asyncio-таск.** Отвергнута: два почти одинаковых цикла
  со `sleep` и `try/except`, дублирование обвязки; при росте числа джоб не масштабируется.
- **Цена решения**: эта change трогает код, введённый `daily-digest` (отсюда явный depends-on и
  порядок реализации). Рефактор минимальный — вынести «один тик → список джоб».

### 2. Триггер: записи на завтра, группировка по клиенту

В прогоне берём `tomorrow = today_in_tz(now, tz) + 1 день` и собираем срез
`list_specialist_day(..., tomorrow, series=ctx)` — те же записи (вкл. виртуальные повторы
серий), что и дневной экран. Группируем по `client_id`: один клиент = один кандидат, даже если
у него несколько записей завтра (берём ближайшее время для текста).

- **Альтернатива — «ближайшая будущая запись клиента == завтра».** Отвергнута пользователем:
  достаточно «есть запись завтра», совпадает с экраном дня и проще.

### 3. Два слоя антидубля

**Слой расписания (per specialist).** На `specialists` — `payment_reminder_last_run_on:
date | None`. Специалист due, когда `payment_reminder_enabled` И
`utc_to_wall(now, tz) >= payment_reminder_time` (как `HH:MM`) И `payment_reminder_last_run_on
!= today_in_tz(now, tz)`. Первый подходящий тик за день делает проход и в любом случае ставит
`last_run_on = today` (как у digest: фиксируем «решение за день», даёт догон после простоя и
отсутствие «поздних» прогонов).

**Слой абонемента (per subscription).** На `subscriptions` — `payment_reminded_at: datetime |
None`. Кандидату шлём алерт, только если `status = active` И `remaining = 0` И
`payment_reminded_at IS NULL`. После показа алерта ставим `payment_reminded_at = now`.

Зачем оба: слой расписания гарантирует один прогон в день и время прогона; слой абонемента
гарантирует «один раз, пока остаток 0» — следующий день (или следующая запись) при том же
пустом абонементе повторного алерта не даёт.

- **Важно**: флаг ставится по факту **показа алерта специалисту**, а не отправки клиенту —
  поэтому даже для непривязанного клиента (алерт без кнопки) напоминание не повторяется.

### 4. Сброс флага — в продлении абонемента

`payment_reminded_at` сбрасывается в `None` в use-case продления (`add-subscriptions`),
поскольку только продление переводит `remaining` из 0 в > 0. Так при следующем обнулении
остатка напоминание сработает заново. «Вычесть» до 0 флаг не трогает (он и так `NULL`).

- **Альтернатива — сбрасывать в любом коде, где remaining становится > 0.** На текущей модели
  это исключительно продление, поэтому достаточно одной точки.

### 5. Сообщение специалисту — интерактивное, отправка клиенту по callback

Впервые бот **сам** инициирует **интерактивное** сообщение (digest — без кнопок;
appointment-notify — кнопки, но инициирует человек). Сообщение специалиста содержит имя
клиента, время завтрашней записи и превью текста для клиента. Кнопка несёт
`pay:send:<client_id>` (укладывается в лимит 64 байта). По нажатию: загрузить клиента,
повторно проверить `telegram_chat_id` (рейс отвязки), отправить **фиксированный** шаблон,
поймать `TelegramForbiddenError`/`TelegramBadRequest` → «не доставлено», успех → «отправлено».
Если на момент прогона `telegram_chat_id IS NULL` — кнопка не добавляется.

- Отправка (`bot.send_message`) живёт в слое `bot/`, сервисы остаются без aiogram.
- Идемпотентности отправки клиенту нет (повторный тап = повторная отправка) — как в
  appointment-notify; дубль маловероятен и некритичен.

### 6. Решение «нужно ли» — чистые функции, проход — сервис

- `is_payment_reminder_due(specialist, now) -> bool` (домен, без БД/таймера).
- `subscription_needs_payment_reminder(sub) -> bool`: `active && remaining == 0 &&
  payment_reminded_at is None` (домен).
- Сервис `run_payment_reminders_if_due(specialist, appts_repo, subs_repo, clients_repo, now,
  alert)`: due → срез завтра → группировка по клиенту → для каждого активный абонемент →
  предикат → собрать данные алерта (есть/нет привязки) → `alert(...)` →
  `mark_payment_reminded(sub_id, now)`; в конце `mark_payment_reminder_run(today)`.
- Цикл в `__main__` — тонкая оболочка: `alert = bot.send_message(+keyboard)`; ловит ошибки
  Telegram; весь проход в `try/except` с логом.

### 7. Дефолты и миграция

Новые колонки на `specialists`:

| колонка                          | тип       | server-default | домен `Specialist`              |
| -------------------------------- | --------- | -------------- | ------------------------------- |
| `payment_reminder_enabled`       | bool      | `true`         | `payment_reminder_enabled=True` |
| `payment_reminder_time`          | str       | `'12:00'`      | `= "12:00"`                     |
| `payment_reminder_last_run_on`   | date NULL | NULL           | `= None`                        |

Новая колонка на `subscriptions`:

| колонка               | тип            | server-default | домен `Subscription`     |
| --------------------- | -------------- | -------------- | ------------------------ |
| `payment_reminded_at` | datetime NULL  | NULL           | `payment_reminded_at=None` |

Хранение времени строкой `HH:MM` — единообразно с `day_start`/`morning_notify_time`.

## Risks / Trade-offs

- **Долгий проход блокирует тик** → один кабинет, специалистов мало; при росте — батч (YAGNI).
- **Клиент заблокировал бота** → ловим `TelegramForbiddenError`/`TelegramBadRequest`,
  «не доставлено», операция не повторяется; алерт уже помечен (не зацикливаемся).
- **Смена времени/tz в течение дня** → решение на текущем значении в момент тика; день уже
  помечен → повторно не сработает. Приемлемо: одно напоминание на день.
- **Кнопка «Отправить» живёт в сообщении сколько угодно** → поздний тап отправит фиксированный
  текст тому клиенту; ожидаемо.
- **Тестируемость 100%** → вся логика в чистых функциях/сервисе; цикл со `sleep` тонкий,
  тестируется на один проход с замоканными зависимостями.

## Migration Plan

1. Alembic-миграция: три колонки на `specialists` (server-default `true`/`'12:00'`/NULL) и
   одна на `subscriptions` (`payment_reminded_at` NULL). Существующие специалисты — opt-in
   включён сразу; существующие абонементы — `payment_reminded_at = NULL`.
2. Код: домен (поля + `is_payment_reminder_due` + предикат), репозитории
   (`list_payment_reminder_candidates`, `mark_payment_reminder_run`, чтение абонемента клиента,
   `mark_payment_reminded`, сброс в продлении), сервис, рефактор общего цикла, сообщение +
   колбэк отправки, настройки, тексты.
3. Откат: обратный revert кода + downgrade миграции (drop колонок). Данных, требующих
   сохранения, миграция не создаёт.
