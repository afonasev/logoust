# deployment

## Purpose

Воспроизводимая выкатка бота на VPS и операционные процедуры вокруг неё: деплой кода
с миграциями и серверными конфигами, регулярный бэкап прод-базы с ретеншеном, скачивание
бэкапов и снимок локальной базы, создание приглашений специалистов против прод-базы и
сбор логов в systemd journal. Цель — безопасные, идемпотентные операции эксплуатации,
не затрагивающие прод-`.env` и базу данных.

## Requirements

### Requirement: Выкатка приложения на VPS

`make deploy` **SHALL** воспроизводимо выкатывать текущий код на VPS: синхронизировать
исходники, установить рантайм-зависимости, применить миграции БД, раскатать серверные
конфиги из репозитория и перезапустить сервис, не затрагивая прод-`.env` и базу данных.

#### Scenario: Успешная выкатка

- **WHEN** специалист запускает `make deploy` с настроенным SSH-алиасом `logoust`
- **THEN** код синхронизируется в каталог приложения на VPS (исключая `.git`, `.venv`,
  `*.db`, `.env`, кэши, `backups`, `logs`)
- **AND** на сервере выполняются `uv sync --no-dev` и `alembic upgrade head`
- **AND** systemd-юнит и cron-файл раскатываются из `deploy/` с `daemon-reload`
- **AND** сервис перезапускается, и после паузы команда подтверждает `systemctl is-active`

#### Scenario: Прод-данные не затрагиваются деплоем

- **WHEN** на сервере уже существуют прод-`.env` и файл базы данных вне каталога кода
- **THEN** синхронизация кода не удаляет и не перезаписывает ни `.env`, ни файл БД

#### Scenario: Сбой выкатки прерывает деплой

- **WHEN** любой шаг (синхронизация, `uv sync`, миграция, рестарт) завершается с ошибкой
- **THEN** скрипт останавливается с ненулевым кодом возврата и не сообщает об успехе

### Requirement: Регулярный бэкап БД на VPS

Сервис **SHALL** ежедневно создавать консистентную онлайн-копию SQLite-базы на VPS по
расписанию и хранить снимки с ретеншеном daily и weekly, не мешая работе приложения.

#### Scenario: Ежедневный снимок

- **WHEN** наступает запланированное время бэкапа (cron, 03:45 UTC)
- **THEN** создаётся консистентная копия базы через `sqlite3 .backup` и сжимается gzip
- **AND** снимок кладётся в каталог `daily/`

#### Scenario: Недельное продвижение и ретеншен

- **WHEN** ежедневный бэкап выполняется в воскресенье
- **THEN** копия дневного снимка дополнительно сохраняется в `weekly/`
- **AND** дневные снимки старше 7 дней и недельные старше ~месяца удаляются

### Requirement: Скачивание бэкапов на локальную машину

`make backup-prod` **SHALL** скачивать серверные бэкапы базы данных в локальный каталог
`backups/vps/` без изменения копий на сервере.

#### Scenario: Скачивание бэкапов

- **WHEN** специалист запускает `make backup-prod` с настроенным SSH-алиасом `logoust`
- **THEN** содержимое серверного каталога бэкапов синхронизируется в `backups/vps/`
- **AND** локальный список скачанных файлов выводится в stdout

### Requirement: Локальный снимок базы данных

`make backup` **SHALL** создавать датированную копию локальной базы `logoust.db`
в каталоге `backups/`.

#### Scenario: Снимок локальной базы

- **WHEN** специалист запускает `make backup`
- **THEN** файл `logoust.db` копируется в `backups/logoust_YYYY-MM-DD.db`

### Requirement: Создание приглашения специалиста на проде

`make create-invite-prod` **SHALL** создавать приглашение специалиста против
прод-базы на VPS и печатать deep-link с username прод-бота, не требуя ручного
входа на сервер.

#### Scenario: Создание приглашения на проде

- **WHEN** специалист запускает `make create-invite-prod` с настроенным SSH-алиасом `logoust`
- **THEN** CLI создания приглашения выполняется на VPS под пользователем `logoust`
  против прод-`.env` и прод-базы данных
- **AND** в stdout печатается только deep-link вида `https://t.me/<bot>?start=<token>`
  с username прод-бота

### Requirement: Логи бота в systemd journal

В продакшене бот **SHALL** писать структурированные логи (JSON) в stdout, чтобы они
попадали в persistent systemd journal; отдельные лог-файлы не используются.

#### Scenario: Логи доступны через journalctl

- **WHEN** сервис запущен на VPS с `LOG_FORMAT=json` и `LOG_FILE_ENABLED=false`
- **THEN** логи приложения доступны через `journalctl -u logoust-assistant`
- **AND** приложение не создаёт собственных файлов логов на диске
