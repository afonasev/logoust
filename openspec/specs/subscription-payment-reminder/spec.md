# subscription-payment-reminder

## Purpose

Накануне визита бот сам напоминает специалисту о клиенте, у которого закончился
абонемент (остаток 0) и на завтра есть запись, и по одному тапу отправляет
привязанному клиенту просьбу продлить — снимая рутину и риск занятия без оплаты.

## Requirements

### Requirement: Расписание прогона напоминания об оплате

Бот **SHALL** раз в день в настенное время `payment_reminder_time` (в таймзоне специалиста,
дефолт `12:00`) выполнять прогон напоминаний об оплате для специалиста, у которого
`payment_reminder_enabled` истинно. Прогон за день **SHALL** выполняться не более одного раза:
специалист считается «пора» (due), когда напоминание включено И настенное время `now` достигло
`payment_reminder_time` И `payment_reminder_last_run_on` не равно сегодняшней дате в его tz.
Первый подходящий тик за день **SHALL** выполнить прогон и в любом случае проставить
`payment_reminder_last_run_on = сегодня`. Прогон **MUST NOT** ронять основной long-polling при
исключении.

#### Scenario: Прогон в назначенное время

- **WHEN** наступает настенное время `>= payment_reminder_time` у включённого специалиста, для которого сегодня прогон ещё не делался
- **THEN** бот выполняет прогон напоминаний
- **AND** проставляет `payment_reminder_last_run_on` в сегодняшнюю дату

#### Scenario: Повторный тик в тот же день

- **WHEN** в тот же день после выполненного прогона наступает следующий минутный тик
- **THEN** повторный прогон не выполняется (`payment_reminder_last_run_on` уже равно сегодня)

#### Scenario: Догон после простоя

- **WHEN** бот не работал в момент `payment_reminder_time` и запустился позже в тот же день, прогон ещё не делался
- **THEN** прогон выполняется при первом тике (порог времени `>=` уже пройден)

#### Scenario: Напоминание выключено

- **WHEN** у специалиста `payment_reminder_enabled` ложно
- **THEN** прогон для него не выполняется

### Requirement: Триггер напоминания — запись завтра и пустой абонемент

В прогоне бот **SHALL** собрать записи специалиста на **завтра** (календарный день в его tz),
включая виртуальные повторы активных серий — тем же составом, что показывает дневной экран
расписания. Записи **SHALL** группироваться по клиенту: на один прогон **SHALL** приходиться не
более одного напоминания на клиента, даже если у клиента несколько записей завтра. Напоминание
**SHALL** формироваться только для клиента, у которого есть активный абонемент с остатком
`remaining = 0` и который ещё не был отмечен напомненным в текущем пустом цикле
(`payment_reminded_at IS NULL`).

#### Scenario: Пустой абонемент и запись завтра

- **WHEN** у клиента есть запись на завтра и активный абонемент с `remaining = 0`, `payment_reminded_at IS NULL`
- **THEN** бот формирует для специалиста напоминание об оплате по этому клиенту

#### Scenario: Остаток абонемента больше нуля

- **WHEN** у клиента есть запись на завтра, но активный абонемент имеет `remaining > 0`
- **THEN** напоминание по этому клиенту не формируется

#### Scenario: Нет записи на завтра

- **WHEN** у клиента активный абонемент с `remaining = 0`, но на завтра записи нет
- **THEN** напоминание по этому клиенту не формируется

#### Scenario: Несколько записей одного клиента завтра

- **WHEN** у клиента завтра две записи и активный абонемент с `remaining = 0`
- **THEN** формируется ровно одно напоминание по этому клиенту

#### Scenario: Нет активного абонемента

- **WHEN** у клиента есть запись на завтра, но активного абонемента нет
- **THEN** напоминание по этому клиенту не формируется

### Requirement: Антидубль по абонементу — один раз пока остаток ноль

После формирования напоминания об оплате по абонементу бот **SHALL** проставить на этом
абонементе `payment_reminded_at`. Пока абонемент остаётся с остатком `0` и
`payment_reminded_at` не сброшен, повторные напоминания по нему **MUST NOT** формироваться — ни
в последующие дни, ни перед последующими записями. Флаг **SHALL** ставиться по факту
формирования напоминания специалисту, независимо от того, было ли в итоге отправлено сообщение
клиенту.

#### Scenario: Повтор на следующий день без изменений

- **WHEN** по абонементу напоминание уже было сформировано (`payment_reminded_at` установлен), остаток по-прежнему `0`, и у клиента есть запись на следующий день
- **THEN** новое напоминание по этому абонементу не формируется

#### Scenario: Непривязанный клиент тоже помечается

- **WHEN** по абонементу сформировано напоминание для клиента без `telegram_chat_id`
- **THEN** `payment_reminded_at` всё равно проставляется
- **AND** повторное напоминание по этому абонементу не формируется

### Requirement: Сообщение специалисту с опцией отправки клиенту

По сформированному напоминанию бот **SHALL** отправить специалисту сообщение с именем клиента,
временем ближайшей завтрашней записи (настенное время в tz специалиста) и превью текста, который
будет отправлен клиенту. Если у клиента заполнен `telegram_chat_id`, сообщение **SHALL**
содержать кнопку отправки напоминания клиенту. Если `telegram_chat_id` не заполнен, кнопка
**SHALL NOT** показываться.

#### Scenario: Клиент привязан

- **WHEN** формируется напоминание для клиента с заполненным `telegram_chat_id`
- **THEN** специалисту приходит сообщение с именем клиента, временем записи и кнопкой отправки

#### Scenario: Клиент не привязан

- **WHEN** формируется напоминание для клиента без `telegram_chat_id`
- **THEN** специалисту приходит сообщение без кнопки отправки

### Requirement: Отправка фиксированного напоминания клиенту

По нажатию специалистом кнопки отправки бот **SHALL** отправить клиенту фиксированный текст с
просьбой продлить абонемент в его `telegram_chat_id`. Перед отправкой бот **SHALL** повторно
проверить наличие `telegram_chat_id`. Сбой доставки (`TelegramForbiddenError` /
`TelegramBadRequest`) **MUST NOT** ронять бота: специалисту **SHALL** сообщаться о недоставке.
Текст напоминания клиенту специалист не редактирует.

#### Scenario: Успешная отправка

- **WHEN** специалист нажимает кнопку отправки, у клиента есть `telegram_chat_id`
- **THEN** клиенту отправляется фиксированный текст с просьбой продлить абонемент
- **AND** специалисту показывается статус «отправлено»

#### Scenario: Клиент заблокировал бота

- **WHEN** при отправке возникает `TelegramForbiddenError` или `TelegramBadRequest`
- **THEN** бот не падает
- **AND** специалисту показывается статус «не доставлено»

#### Scenario: Клиент отвязался между показом и нажатием

- **WHEN** к моменту нажатия `telegram_chat_id` клиента стал пустым
- **THEN** сообщение клиенту не отправляется
- **AND** специалисту показывается статус «клиент не привязан»
