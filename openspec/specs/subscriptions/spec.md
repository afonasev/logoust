# subscriptions

## Purpose

Абонементы клиентов специалиста: создание абонемента с числом купленных встреч, ручное
списание и продление встреч, закрытие абонемента и просмотр активных и закрытых абонементов.
Абонемент принадлежит специалисту через клиента — каждый видит и изменяет только абонементы
своих клиентов.

## Requirements

### Requirement: Создание абонемента клиенту

Бот **SHALL** позволять специалисту создать абонемент для своего клиента — запись с числом
купленных встреч `purchased`, остатком `remaining = purchased`, статусом `active`,
проставленным `created_at`, `closed_at IS NULL`, привязанную к клиенту (`client_id`) и
специалисту (`specialist_id`). У клиента **MUST NOT** быть более одного абонемента со статусом
`active` одновременно. Число встреч при создании специалист выбирает кнопкой из вариантов,
заданных в настройках специалиста, либо вводит своё.

#### Scenario: Создание абонемента выбором варианта

- **WHEN** специалист на карточке активного клиента без активного абонемента создаёт абонемент, выбрав кнопкой один из вариантов числа встреч
- **THEN** в таблице `subscriptions` появляется запись со `status = active`, `purchased` и `remaining`, равными выбранному числу, `client_id` и `specialist_id` текущего клиента, заполненным `created_at` и `closed_at IS NULL`
- **AND** бот показывает карточку созданного абонемента

#### Scenario: Создание абонемента с введённым числом

- **WHEN** специалист при создании вводит своё число встреч (например, 4)
- **THEN** создаётся абонемент с `purchased = 4` и `remaining = 4`

#### Scenario: У клиента уже есть активный абонемент

- **WHEN** у клиента уже есть абонемент со `status = active`
- **THEN** второй активный абонемент не создаётся
- **AND** на карточке клиента вместо кнопки создания показана кнопка перехода на существующий абонемент

### Requirement: Карточка абонемента

Бот **SHALL** показывать специалисту карточку абонемента с датой создания, числом купленных
встреч `purchased`, остатком `remaining` и действиями: вычесть встречу, продлить, закрыть.

#### Scenario: Просмотр карточки активного абонемента

- **WHEN** специалист открывает карточку активного абонемента
- **THEN** бот показывает дату создания, `purchased`, `remaining` и кнопки «вычесть встречу», «продлить», «закрыть» и возврат на карточку клиента

### Requirement: Ручное списание встречи

Бот **SHALL** по действию «вычесть встречу» уменьшать `remaining` активного абонемента на 1.
Если `remaining` уже равен 0, списание **MUST NOT** выполняться (в минус остаток не уходит),
абонемент остаётся `active`. Списание не связано со встречами или расписанием — остаток
меняется только этим действием.

#### Scenario: Успешное списание

- **WHEN** специалист на карточке абонемента с `remaining = 5` нажимает «вычесть встречу»
- **THEN** `remaining` становится 4, статус остаётся `active`
- **AND** карточка обновляется с новым остатком

#### Scenario: Списание при нулевом остатке

- **WHEN** специалист нажимает «вычесть встречу» на абонементе с `remaining = 0`
- **THEN** `remaining` не меняется и остаётся 0
- **AND** бот сообщает, что списывать нечего

### Requirement: Продление абонемента

Бот **SHALL** по действию «продлить» увеличивать у активного абонемента и `purchased`, и
`remaining` на выбранное число встреч `N` (кумулятивно). Число встреч специалист выбирает
кнопкой из вариантов, заданных в настройках специалиста, либо вводит своё.

#### Scenario: Продление выбором варианта

- **WHEN** специалист на абонементе с `purchased = 8`, `remaining = 3` продлевает, выбрав вариант (например, 8)
- **THEN** `purchased` становится 16, `remaining` становится 11, статус остаётся `active`

### Requirement: Закрытие абонемента с подтверждением

Бот **SHALL** позволять специалисту закрыть активный абонемент даже при `remaining > 0`.
Закрытие **SHALL** требовать подтверждения. При подтверждении абонементу проставляется
`status = closed` и `closed_at`; запись **MUST NOT** удаляться. После закрытия у клиента
снова можно создать новый абонемент.

#### Scenario: Закрытие при ненулевом остатке

- **WHEN** специалист закрывает абонемент с `remaining = 2` и подтверждает действие
- **THEN** у абонемента `status = closed`, `closed_at` проставлен, запись сохраняется в БД
- **AND** на карточке клиента снова показана кнопка создания абонемента

#### Scenario: Отмена подтверждения закрытия

- **WHEN** специалист инициирует закрытие, но отменяет подтверждение
- **THEN** абонемент остаётся `active`, `closed_at IS NULL`

#### Scenario: Создание нового абонемента после закрытия

- **WHEN** у клиента есть только закрытый абонемент (нет активного)
- **THEN** специалист может создать новый активный абонемент штатно

### Requirement: Валидация числа встреч

Система **SHALL** при создании и продлении принимать как число встреч только положительное
целое в разумных пределах. Некорректный ввод (ноль, отрицательное, нечисловое, чрезмерно
большое) **MUST NOT** приводить к изменению абонемента.

#### Scenario: Некорректный ввод числа встреч

- **WHEN** специалист при создании или продлении вводит не положительное целое (например, `0`, `-3` или `abc`)
- **THEN** абонемент не создаётся и не изменяется
- **AND** бот просит ввести корректное число встреч

### Requirement: Раздел абонементов в главном меню

Бот **SHALL** предоставлять в главном меню специалиста пункт «Абонементы», открывающий
постраничный список **активных** абонементов всех его клиентов (в строке — имя ребёнка и
остаток встреч; переход на карточку абонемента). Из этого раздела **SHALL** быть доступен
постраничный список **закрытых** абонементов (история): имя ребёнка и дата закрытия, переход
на карточку. Карточка закрытого абонемента **SHALL** быть только для чтения — без действий
списания/продления/закрытия. Списки **MUST** показывать только абонементы клиентов текущего
специалиста.

#### Scenario: Список активных абонементов

- **WHEN** специалист открывает раздел «Абонементы»
- **THEN** бот показывает список активных абонементов его клиентов с остатком встреч в строке; тап по строке открывает карточку абонемента

#### Scenario: История закрытых абонементов

- **WHEN** специалист переходит к закрытым абонементам
- **THEN** бот показывает список закрытых абонементов с датой закрытия; карточка открытого из истории абонемента показывается без действий (только возврат)

#### Scenario: Пустой список

- **WHEN** у специалиста нет активных (или закрытых) абонементов
- **THEN** бот показывает соответствующее сообщение о пустом списке

### Requirement: Изоляция абонементов по владельцу

Абонемент принадлежит специалисту через клиента. Специалист **MUST NOT** видеть или изменять
абонементы чужих клиентов.

#### Scenario: Попытка открыть чужой абонемент

- **WHEN** специалист пытается открыть карточку абонемента, принадлежащего клиенту другого специалиста
- **THEN** бот не показывает абонемент и не выполняет над ним действий
